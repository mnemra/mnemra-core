## 1. Purpose / context

After `0.14.0`'s one-shot corpus migration, content keeps arriving — markdown notes, binary documents (PDF/PPTX/XLSX/DOCX/CSV/JSON), later clipped web content and feed items — and nothing ingests it: new content enters the system only if a human notices it and re-runs something by hand, and every derived view of it is re-computed per-session, the exact re-derivation cost the product exists to kill. The job (intake JTBD): content arriving in the corpus enters durably and automatically, so derived context is (re)computed from what was persisted rather than re-ingested.

- **What this designs.** The ingest subsystem inside the mnemra-core host: a source registry (registration as the admission-control boundary), reconciliation-first arrival drivers, the shared loader seam the `0.14.0` migrator and the `1.2.0` watcher both drive, durable raw staging, a transform harness on a minimal `JobQueue` port (materializing the architecture-overview's ADR-16 slot), egress-denied binary-extraction transforms (#324), recency instrumentation (the D1 axis), and the quarantine/failure path. In-scope surfaces: core + file-watcher + binary extraction.
- **What this does not design.** The `0.14.0` batch migration itself (Non-goal 1 — the seam is shared, nothing else); derived-layer content (Non-goal 2 — retrieval-cluster territory; this bundle owns the harness *conventions*, concrete transforms are consumers); generative summarization (Non-goal 3); the review/triage queue (Non-goal 4, parked); clipper/webhooks/polls (Non-goal 5, parked — designed later *against* this locked contract); RAS asset management beyond extraction-and-retention (Non-goal 6, parked); the classify-at-write stage itself (Non-goal 7 — only its insertion point, per D3); live operational data (Non-goal 8, per D1); internal writes (Non-goal 9 — not ELT).
- **Intake.** [`docs/intent/ingestion-pipeline.md`](ingestion-pipeline.md) — locked 2026-07-16, high stakes, `spec_type: code`, Stage 1c security review round 1 zero blocker/high, all findings folded.
- **Design substrate.** The maintainer's ELT-over-ETL ruling (the strangler-migration decision walk, 2026-07-06, carried verbatim in intake Hard constraint 2); the loader-seam ruling (2026-07-16, intake Hard constraint 1); the differentiation-trio decision record (ratified 2026-07-06 — D1 recency axis, D3 classify-at-write insertion point); the pre-ADR architecture-overview forward design — a **workspace-private forward-design companion** (the label [P-0017](../src/adrs/P-0017-storage-cluster-model.md)/[P-0018](../src/adrs/P-0018-core-entity-manifest.md) use for it), distinct from the in-repo [`overview.md`](../src/architecture/overview.md); its consumed verdicts (ingest ELT 5-step, the subsystem-table `JobQueue` and `IngestPipeline` rows, the load-shedding contract, the retention archetypes, the two-adapter test) are restated inline for self-containment and currency-verified in §8, not inherited; the intake's Stage 1c security review threat-model handoff (§5); the locked retrieval cluster ([`docs/specs/2026-07-02-retrieval-cluster.md`](../specs/2026-07-02-retrieval-cluster.md), [P-0014](../src/adrs/P-0014-retrieval-architecture.md) RA-1/RA-2) for the §8 reverse-dependency reconciliation. ## Proposed system boundary

mnemra-core gains an **ingest subsystem**: source registry → arrival drivers → shared loader seam → durable raw staging → transform harness → canonical entities with back-pointers. External-content only (internal writes are not ELT). In-scope surfaces: file-watcher driver + binary-extraction transforms. The `0.14.0` migrator is the seam's second driver — admitted by design, not built here.

## Proposed component map (architectural altitude)

1. **Source registry** — register/list/retire sources (corpus roots, source kinds). Registration is the admission-control boundary (intake Risk profile): who may register, and what fs-write on a watched root does and does not confer.
2. **Arrival drivers** — (a) file-watcher + startup reconciliation; (b) `0.14.0` one-shot walker (out of scope; seam must admit it). **Proposed stance: reconciliation-first** — the periodic/startup scan is the primary arrival mechanism (crash-safe by construction; SC11 falls out for free), fs events are a latency accelerator on top. The alternative (event-primary + catch-up special case) treats the fragile path as primary.
3. **Loader seam** (the shared machinery, per the locked ruling) — parse/normalize/frontmatter-map/provenance-stamp/write-through. Contract shape: bytes + source-metadata in → staged raw artifact with PE-7-mapped provenance out. Driver differences (kill-and-resume manifest, byte-equivalence, dedup, recency) stay in drivers.
4. **Durable raw staging** — per-source-kind staging schema; retention archetypes (long-lived original vs transient original); content-hash dedup identity. **Conflict to resolve:** the overview lists staging under `UNLOGGED` candidates, but the ELT ruling makes raw = truth → staging must be durable/logged. Propose: staging is durable; `UNLOGGED` stays cache-only.
5. **Transform harness** (ADR-16 materialized) — enqueue-per-staging-row; job workers apply transforms → canonical entities with source back-pointers; re-derivation as first-class re-run; new-rows-vs-update-with-audit policy; per-source-kind retention. **Dependency: rides the JobQueue subsystem (overview row 5, `FOR UPDATE SKIP LOCKED` sketch) — which is NOT built and has no ADR.** Decision axis below.
6. **Extraction transforms** (#324, in scope) — binary (PDF/PPTX/XLSX/DOCX/CSV/JSON) → content, running **egress-denied** (locked fold M3). Mechanism (WASM component reusing the plugin substrate vs sandboxed subprocess vs vetted in-process crates) is a Frame decision with a feasibility consult — heavy parsers in WASM is the open feasibility question.
7. **Recency instrumentation** — arrival→queryable metric per surface (D1 axis); bound set at spec.
8. **Quarantine / failure path** — designed errors, fail-closed, no partial canonical writes; all ops working-state → working-state.

## 4. System boundary + component map

The ingest subsystem is a **host subsystem** of mnemra-core — host code above the plugin boundary, beside the retrieval and coordination subsystems, sharing the substrate the host already owns. External-content only: internal writes (plugin storage via host-fns, MCP verbs, plugin events) are not ELT and never touch this path (intake Hard constraint 3).

- **Source registry.** Registered sources (corpus roots at V0.1; source kinds closed-enum) with per-source `trust_class` and `retention_policy`, registered/retired through admin-gated CLI control-plane operations, audited. Registration is the admission-control boundary: nothing outside a registered source is ever staged (`unregistered-source` is a designed rejection). One host-owned table.
- **Arrival drivers.** (a) The `1.2.0` watcher: an in-process supervised async task running the **reconciliation scan** (primary) with fs events as a latency accelerator (2a decision 3); startup reconciliation is the same scan (SC11 by construction). (b) The `0.14.0` one-shot walker (built in its own register entry, Non-goal 1): admitted by the seam contract — kill-and-resume manifest, byte-equivalence checking, and source-read-only posture live in that driver, not in the seam.
- **Loader seam** (the shared machinery, intake Hard constraint 1). A typed stage chain: `parse/normalize → frontmatter-map → provenance-stamp → admission chain → write-through (transactional staging insert; enqueue post-commit)`. Contract: bytes + source-metadata in → staged raw artifact with PE-7-mapped provenance out, transform job enqueued (best-effort; the reconciliation scan re-enqueues orphans — IP-2). The admission chain is the D3 classify-at-write insertion point (§6 IP-2).
- **Durable raw staging.** One superset staging table (logged, never `UNLOGGED`), closed `source_kind` vocabulary, exactly-one-of raw text/raw bytes payload, BLAKE3 content-hash identity, per-source-kind retention. Raw = truth; the staging row is the re-derivation source-of-record (§6 IP-4).
- **Transform harness** (the ADR-16 slot, materialized). Enqueue-per-staging-row on the `JobQueue` port; workers apply registered transforms producing canonical content rows with `derived_from` lineage ([P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT); re-derivation is a first-class re-run; re-derivation policy is update-with-audit (§6 IP-6).
- **`JobQueue` port + Postgres adapter** (2a decision 1). Minimal port — enqueue / claim / complete / fail-with-retry / dead-letter / scheduled delivery — at-least-once, single-claimant semantics; Postgres adapter on `FOR UPDATE SKIP LOCKED`; the `{{P-job-queue}}` ADR slot opens in this bundle (§6 IP-5).
- **Extraction transforms** (#324). PDF/PPTX/XLSX/DOCX/CSV/JSON → extracted content, running **egress-denied** in the per-format sandbox lanes §6 IP-7 locks (WASM-component default; sandboxed-subprocess lane where the crate cannot run in WASM); original binary retained as the raw staging artifact.
- **Recency instrumentation** (D1). Arrival→staged and staged→canonical-committed spans per source kind, plus per-run reconciliation records; the composite arrival→indexed span is computed by join with the retrieval cluster's index-build records (§6 IP-12).
- **Quarantine / failure path.** Closed rejection-class vocabulary, fail-closed, no partial canonical writes; quarantined rows are inert, recorded, operator-listable; every operation moves working-state → working-state (§6 IP-14).
- **Storage.** All operational tables (registry, staging, jobs, run records) under the engine-agnostic `Storage` trait (`libs/mnemra-host/storage.rs`), workspace-scoped, outside the content taxonomy; canonical outputs are ordinary content rows.

**Agent-visible surface: none.** Zero new MCP tools (the verb budget's last slot is reserved — [P-0022](../src/adrs/P-0022-coordination-cluster.md) D-VERBS); ingested content is reached through the existing retrieval verbs once indexed. The operator surface is CLI (registration, quarantine listing) plus the observability stream.

## 5. Trust boundaries + risk profile (resolved)

The intake declares the trust boundary touched: external, partially-untrusted content crosses into the durable corpus, and the pipeline introduces a standing-service posture the system does not yet have. The Stage 1c security review handed this Frame a six-item threat-modeling set with a stated attribute priority — **Integrity** (provenance/origin trust, complete mediation of writes) > **Availability** (resource-exhaustion bounds, load-shedding) > **Confidentiality** (zero-egress preservation, PII via the D3 hook). Each item gets a resolution or a named, fireable deferral. The security reviewer's security-mode pass runs on this Frame (the intake's required Stage-2 constraint) — the resolutions below are its review surface.

**TB-1 — Extraction sandbox: egress-deny, not merely fault isolation (resolved; mechanism consult-gated).** The locked posture (intake fold M3): binary extraction runs egress-denied, because OOXML is ZIP-of-XML and an XXE-triggered outbound fetch would break the zero-egress promise (HC6) — the sandbox is load-bearing for the sovereignty configuration, not just for crash containment. Resolution is layered (P-SecurityLayered): (a) parser configuration — XML parsing bans DTD/external-entity resolution; (b) **structural egress-deny** — the sandbox grants no network capability at all, so even a parser regression cannot fetch (P-GuaranteeByMechanism: the guarantee is a capability property, not a crate-behavior convention); (c) resource bounds (TB-2). The *mechanism* — WASM component on the existing Wasmtime substrate vs sandboxed subprocess — is decided in §6 IP-7 with the backend feasibility consult; the egress-deny property is invariant across **every lane that runs**, because IP-7's lane selection is fail-closed: a lane whose egress-denying enclosure is not verifiably present on the host is not available, and a format clearing no available lane quarantines (`unextractable-here`) rather than running unsandboxed. The rejected third option (vetted in-process crates, no sandbox) fails this boundary by construction: it would make zero-egress rest on the absence of network calls in every parser dependency's transitive closure, unverifiable at update time.

**TB-2 — Resource exhaustion at intake (resolved).** Bounds exist at every stage, all config-tunable with spec-set defaults: per-file size cap enforced before read-in (oversized-input rejection); decompression/expansion budget on archive-shaped formats (ZIP-of-XML) enforced as **explicit counters in the extraction wrapper itself** — decompressed-total-bytes, entry-count, nesting-depth — independent of any lane's memory ceiling (a memory limit catches memory expansion; it does not catch a bomb decompressing to Lane B's scratch/disk surface, which the spec SHALL additionally bound — `RLIMIT_FSIZE` or a bounded scratch write); per-transform execution budget — [P-0007](../src/adrs/P-0007-plugin-resource-limits.md) mechanisms (fuel + epoch + memory, kill-and-replace) on the WASM lane with **extraction-calibrated numeric bounds set at spec, not P-0007's plugin-calibrated V0 values**, and Lane B's named bound set per IP-7 (the two lanes are *not* bound-equivalent — recorded there, not papered); scan batching bounds the arrival rate structurally (reconciliation-first coalesces event floods by construction — a thousand fs events against one file resolve to one scan observation). The load-shedding contract (§6 IP-13) absorbs transform backlog without blocking staging. Numbers are spec work; the bounds' existence is locked here, and their placement is locked **except two named at-spec placements the spec SHALL make** — the quarantine aggregate bound (IP-14) and the Lane-B disk expansion budget — named obligations, not open questions.

**TB-3 — Watched-root filesystem ↔ durable corpus (admission control; resolved).** The Stage 1c M4 question: does fs-write on a watched root equal write-through into the durable corpus? Resolution: **yes within an explicit, admin-attested trust scope — never silently.** Source registration is an admin-gated, audited control-plane operation ([P-0009](../src/adrs/P-0009-rls-admin-token.md) admin family; the matrix rows land via `{{P-0009-A1-ingest-control-plane}}`, §9); registering a root is the authorization act, and the registration row carries the `trust_class` the admin attests (§6 IP-10). The effective boundary therefore does not *silently* reduce to fs-write permission: it reduces to fs-write **on a root an admin explicitly registered with a declared trust class**, and everything staged from it carries that class in provenance. Path containment is structural **at the open, not check-then-open** (a canonicalize-then-open sequence is TOCTOU-vulnerable — a symlink flipped between check and open resolves outside the root at read time): the spec SHALL render enforcement at the file-open itself — Linux `openat2` with `RESOLVE_BENEATH`/`RESOLVE_NO_SYMLINKS`; `O_NOFOLLOW` on the final component plus post-open same-file verification where `openat2` is unavailable — with canonicalization and traversal-name rejection as the pre-filter (path-traversal rejection class), and the scan admits **regular files only** (`S_ISREG`: in-root symlinks, FIFOs, devices, and sockets are skipped — reading a FIFO blocks forever and a device file is unbounded, so special files are a DoS surface, not content). Archive extraction inside the sandbox writes no filesystem at all (bytes-in/content-out).

**TB-4 — Provenance / origin trust for the new source kinds (resolved; PE-7 dispositioned in full, PE-11 on its ingest half).** Adversarial corpus text must not flip trust ([P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-7). This bundle is register `1.2.0` — the named firing event for PE-7's `declared`-weight re-open (N10) and the ingest half of PE-11's polarity revisit. Disposition (§6 IP-10): per-source **trust class at registration** — `first-party` roots (the V0.1 dogfood corpus, admin-attested) keep `declared` frontmatter authority, the first-party-corpus assumption made explicit rather than assumed; `external`-class sources (the parked clipper/webhook/feed follow-ons) are demoted **where trust is read**: their frontmatter relations still record as honest `declared`-origin edges via the ES-6 sanctioned path, but PE-7's trust predicates — amended by `{{P-0015-A1}}` — do not admit a `declared` trust-affecting edge whose declaring artifact is external-class (trust-inert; promotion only via the PE-6 gate), lifecycle fields cannot fire `authoritative`, and `model_egress: deny` is stamped explicitly. No external-class source ships at V0.1; the rule is locked now so the parked follow-ons inherit it, and the P-0015 amendment carrying it is named (§9); PE-11's adopter-deployment half stays live on its own instrument (§10). The residual: a first-party attestation is only as good as the admin's control of **what writes into** that directory — a sync-target, shared drive, or externally-contributed repo under a first-party root admits content the operator did not author under a first-party attestation, even with full ownership of the path — accepted at single-operator dogfood scope, re-opened by the same amendment when the first external-class source lands.

**TB-5 — Standing-service authority (resolved).** The watcher runs unattended with write-through authority. Containment is by privilege scope (2a decision 2): read-only on registered roots; writes only through the loader seam; the seam writes under a distinguished **ingest system principal** bound to the registered source's workspace — the PE-3 system-principal pattern, a subsystem-naming principal variant, not a workspace role and not a synthetic admin token (§6 IP-9). The principal cannot register sources, mutate policy fields, or touch the PE-6 admin-gated paths — least authority for the role's job. Failure containment: per-file errors quarantine (never halt the pipeline); a watcher crash is a supervised-task restart whose recovery path *is* the reconciliation scan (SC11) — the crash-recovery mechanism and the normal arrival mechanism are the same code, so the recovery path is exercised on every startup, not only in incidents (P-TrustworthySignal: the recovery signal cannot silently rot).

**TB-6 — Content-identity / dedup as an integrity boundary (resolved).** Identity is `(workspace_id, source_id, source_path, content_hash)` with BLAKE3 as the hash (§6 IP-11). Dedup poisoning is bounded two ways: **hash collision resistance** — BLAKE3 is a cryptographic hash function whose 256-bit collision resistance is its own documented design property, so an attacker cannot forge a hash match; the [P-0008](../src/adrs/P-0008-admin-token-shape.md) citation in IP-11 is the in-corpus *algorithm-choice* precedent only (P-0008's "fast non-cryptographic" phrasing described its token-hashing usage posture there, not a limit of the algorithm — this boundary's security claim stands on BLAKE3's collision resistance directly, not on that precedent) — and **path-scoped identity** — equal content at a different path is a distinct staging row, so a planted duplicate in one location can never suppress a legitimate arrival elsewhere. A dedup no-op is a recorded arrival event (operator-observable), so suppression is visible, never silent.

**RLS/tenant boundary on the ingest write path (Stage 1c L3; resolved — claim stated precisely).** The ingest system principal must be able to write *any registered source's* `workspace_id`, so it is cross-workspace-write-capable by design and workspace-scoped RLS **cannot** be its constraint. Tenant isolation on the ingest write path is therefore **structural-by-code-site, not structural-by-RLS**: every written row carries `workspace_id` derived from the source registration row through a single context-construction site (the [P-0006](../src/adrs/P-0006-v0-tenant-enforcement.md) pattern — one construction point, one audit point; that site is the audited chokepoint), WHERE-clause discipline on every read, no bypass path — and the spec SHALL render the negative test: the ingest principal cannot commit a row whose `workspace_id` differs from its source's registered workspace. Latent at V0 single-workspace; the invariant is what carries multi-tenant correctness at V0.1+. At V0.1+ RLS activation the same writes run under the P-0010 preconditions unchanged (no `BYPASSRLS`, `SET LOCAL` per transaction) — RLS then constrains tenant principals; the system principal's constraint remains the construction-site invariant.

**PII surface (Stage 1c L5; carried) — and its injection sibling.** The D3 admission point (§6 IP-2) is the designed hook for the eventual PII/data-sensitivity classification stage — that is *why* the insertion point exists; the stage itself stays Tier-2-gated (D3, ratified 2026-07-06). Nothing else in this Frame holds PII machinery. **Adversarial-content-as-context (named, deferred):** ingested content is served to downstream agent consumers as context, so a document that parses cleanly but carries injection payloads aimed at those consumers is a defining threat class for this product category — the PII surface's sibling, sharing the same designed home: the D3 admission hook is the insertion point for injection-scanning exactly as for sensitivity classification, and the admission point's rationale carries both. Deferred with D3 (Tier-2-gated); nothing at V0.1 scans for it — the deferral is recorded here rather than silent.

**Residual risks, stated:** (1) first-party trust rests on the admin's registration attestation, and that attestation covers the admin's control of **what writes into** the registered directory, not merely ownership of the path (a sync-target or shared root admits third-party content under the attestation) — accepted at dogfood scope, re-opens with the P-0015 amendment's external-class arrival; (2) the subprocess lane's sandbox strength varies by OS, **and on a host with no real enclosure its structural egress-deny is absent outright** (kernel-enforced on Linux; on macOS only a container boundary provides it — no supported host-level primitive exists) — dispositioned by IP-7's fail-closed terminal: Lane B is *legal* only where its enclosure precondition verifiably holds, and a format clearing no available lane quarantines (`unextractable-here`) rather than running unsandboxed, so the residual exposure is availability (formats unavailable on some hosts), never egress; (3) the WASM lane's XLSX/PDF viability is dependency-graph-inferred pending the implementation-start build spike (IP-7's criterion; a failing spike moves the format to the subprocess lane, never to an unsandboxed path — and a host clearing neither lane quarantines the format, the same terminal); (4) no rate limiting on first-party arrival volume beyond scan batching, size caps, and the IP-3 event-debounce/arrival-event rate bounds — nuisance-class at single-operator scope, visible in run records, re-examined at the first external-class source (named non-control, not an omission); (5) quarantined raw content is retained in staging (inert but present) — deliberate, because fail-closed must not destroy evidence; retention is bounded (the IP-14 aggregate quarantine bound) and listing is admin-visible, metadata-only; (6) an in-place, size-preserving edit with a reset mtime evades the scan's size/mtime fast-path until the periodic full-content rehash (IP-3) — a bounded, spec-named staleness window, not an integrity boundary claim; low exploitability at first-party single-operator scope, re-examined at the first external-class source. None is silent.

## 6. Locked architectural directions

Each direction is locked with its canon anchors at the decision-and-rationale line. Requirement-precision rendering (R-IDs, DDL, numeric defaults) is Stage-3 work inside this closed world.

### IP-1 — Residence: core host subsystem; zero new MCP tools; registration is CLI control-plane

The ingest subsystem — registry, drivers, loader seam, staging, harness, queue, instrumentation — is **host code inside the mnemra-core host process**, beside the retrieval and coordination subsystems. The [P-0002](../src/adrs/P-0002-core-plugin-partition.md) walk, recorded: *verb-on-content?* No — arrival detection, staging, and job execution are operational substrate, not CRUD on content rows (the subsystem *produces* content rows; its own machinery is not content). *Foundational-substrate dependency?* Yes — staging and the queue must exist before any transform consumer, and the standing service binds at host startup. Nothing exports through the plugin ABI; the [P-0013](../src/adrs/P-0013-plugin-invocation-model.md) deferred-domain-verb trip-wire does not fire; [P-0005](../src/adrs/P-0005-v0-signing-chain.md)'s plugin-load verification surface is unchanged (vacuously-by-residence, stated). **Zero agent-visible MCP tools ship**: the verb budget stands at 8 of ≤9 with the last slot reserved for authoring paths ([P-0022](../src/adrs/P-0022-coordination-cluster.md) D-VERBS — "this cluster SHALL NOT consume the reserved slot" binds this bundle identically). Source register/list/retire and quarantine listing are **admin-scoped CLI control-plane operations** in the [P-0009](../src/adrs/P-0009-rls-admin-token.md) admin family (the workspace-lifecycle/migrate/backup siblings), audited — with the honesty note the fold adds: **the P-0009 matrix does not yet enumerate these operations**, so the `{{P-0009-A1-ingest-control-plane}}` amendment (§9) adds their rows (`source register/retire`, `quarantine list/inspect`: admin Allowed, read-observer Denied, audited like their siblings) rather than claiming coverage the matrix text does not carry. The registration row is itself control-plane *input*: the spec SHALL validate its fields against allow-lists, not mere well-formedness — `trust_class` ∈ enum, `retention_policy` ∈ enum, root path canonicalized + must-be-directory + within an allowed prefix (an admin registering `/` or `/etc` as a first-party root is refused, because that would ingest system files under `declared` authority). *(Anchors: P-0002 discriminator; host-placement precedents — retrieval (P-0014 RA-4), coordination (P-0022 D-RESIDENCE); the verb-budget constraint, maintainer-ratified 2026-07-06; P-0009 admin family + `{{P-0009-A1-ingest-control-plane}}`; P-LeastAuthority; P-ShiftLeft D2.)*

### IP-2 — The loader seam contract: typed stage chain; driver differences stay in drivers; the admission chain is the D3 insertion point

The seam (intake Hard constraint 1, LOCKED share) is a typed stage chain with one contract: **bytes + source-metadata in → staged raw artifact with PE-7-mapped provenance out, transform job enqueued.** Stages, in order: `parse/normalize` (format detection, text normalization; binary kinds pass through raw) → `frontmatter-map` (structured frontmatter extraction for markdown kinds) → `provenance-stamp` ([P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) envelope; trust-class mapping per IP-10; PE-2 not-set defaults applied on this sanctioned write path) → **`admission chain`** (ordered, registered admission stages; V0.1 content: the fail-closed validation set — size, format, path, registration checks) → `write-through` (transactional staging insert; the transform-job enqueue follows **post-commit, best-effort** — the durable staged row is the guarantee, and the reconciliation scan re-enqueues any staged-but-untransformed orphan (IP-3/IP-14), so the `JobQueue` port carries no SQL-transactional coupling and IP-5's two-adapter test holds; the Postgres adapter MAY enqueue inside the staging transaction as an adapter-local optimization, never as a port guarantee; orphan latency is bounded by the scan cadence). 

**The admission chain is the D3 classify-at-write insertion point** (intake Hard constraint 4; SC8): a classification stage inserts by registering one more admission stage — no schema change, no path reshape. The stage contract is typed: `(candidate artifact + provenance) → admit | quarantine(reason) | annotate(policy fields)` — annotation writes only the PE-2 policy columns that already exist, so a future classifier stamping `model_egress: deny` lands in existing schema. The stage itself stays undesigned (Non-goal 7; Tier-2-gated per D3, ratified 2026-07-06).

Driver differences stay in drivers, per the locked ruling: the `0.14.0` migrator owns kill-and-resume manifest, byte-equivalence verification, and source-read-only posture; the `1.2.0` watcher owns arrival detection, dedup consultation, and recency stamping. The seam admits both (SC1); building the migrator driver is out of scope (Non-goal 1). *(Anchors: intake HC1 + HC4; D3, differentiation-trio record ratified 2026-07-06; P-LockContract — the seam is the intrinsic contract; P-MinBlastRadius — one chain, one slot per concern; P-0015 PE-2.)*

### IP-3 — Arrival: reconciliation-first; events are an accelerator; startup catch-up is the same mechanism

Locked as ratified (2a decision 3): the **periodic/startup reconciliation scan is the primary arrival mechanism**; filesystem events accelerate latency only. The scan walks registered roots and joins observed state (path, size/mtime, content hash on candidates) against staging identity — reconciliation on data, not on event labels — so a missed event costs latency, never correctness, and SC11 (missed-arrival catch-up) is the same code path as normal operation, exercised at every startup. The scan reconciles a **second** join alongside filesystem→staging — **staging→jobs**: a staging row whose post-commit best-effort enqueue never landed a transform job is an orphan (IP-2/IP-14), and the scan re-enqueues it, so orphan recovery rides the same reconciliation mechanism as missed-arrival recovery — one scan, two joins; the exact orphan predicate renders at spec (`{{P-ingest-pipeline-shape}}`) (fold — FIV-1). **The size/mtime candidate pre-filter is a fast-path heuristic, not an integrity boundary** (fold): it is spoofable — an in-place, size-preserving edit with a reset mtime is not selected, not re-hashed, and stale canonical content keeps serving — so the spec SHALL bound the heuristic with a **periodic full-content-rehash cadence** (every file re-hashed within a named window regardless of the pre-filter). The missed-event claim above is thereby scoped honestly: event loss is covered by the scan; heuristic evasion is bounded to the rehash window's staleness (residual (6), §5). Event subscription (a cross-platform fs-notification dependency, Green-tier, named at spec) triggers targeted early scans of changed paths; event floods coalesce structurally into scan batches (TB-2), with the **event→scan debounce/coalescing window a spec-named parameter**, and identical-re-arrival event emission rate-bounded at spec (a touch-flood writes bounded arrival-event rows — availability nuance, not correctness). The scan interval is config-tunable with the spec setting the default against the recency bound. *(Anchors: 2a decision 3, LOCKED; intake SC4/SC11; P-GuaranteeByMechanism — correctness is carried by the scan mechanism, not by event delivery guarantees the OS does not make; P-TrustworthySignal — the recovery path cannot silently rot because it runs constantly.)*

### IP-4 — Durable raw staging: logged, one superset table, in-substrate blob home, per-source-kind retention

- **Durability:** staging tables are **ordinary logged tables — never `UNLOGGED`**. Raw = truth (intake Hard constraint 2): a raw store that Postgres truncates on crash recovery cannot be the ELT source-of-record. `UNLOGGED` remains cache-only in this system. This re-derives the pre-ADR overview's UNLOGGED-staging note (§8 — falsified by the later ELT ruling; re-derive-on-reshape per the when-to-lock edge).
- **Shape:** **one superset staging table**, not per-kind table proliferation: `(workspace_id NOT NULL, source_id FK, source_kind [closed enum], source_path, content_hash, raw_text | raw_bytes [exactly one non-null, CHECK-enforced], frontmatter jsonb [nullable], provenance/policy stamp columns, status [closed enum: staged | transformed | quarantined], rejection_class [nullable, closed enum], arrival + processing timestamps)`. Per-kind variance is data (registry fields + kind column + payload polarity), not schema forks — the [P-0016](../src/adrs/P-0016-edge-schema.md) ES-1 one-superset-table precedent. The spec renders exact DDL; kind-specific *staging schema* in ADR-16's sense is this superset shape plus the per-kind registry policy, dispositioned by name in `{{P-ingest-pipeline-shape}}`.
- **Blob home:** retained binaries live **in-substrate** (`raw_bytes`, TOAST-backed) at V0.1, bounded by the TB-2 size cap. Chosen over a filesystem blob store because the staging row and its payload commit in one transaction — no fs/db two-phase residue, which is what makes SC9's working-state→working-state and SC10's no-partial-write cheap to guarantee — and the single self-hosted binary posture ([P-0010](../src/adrs/P-0010-storage-substrate-engine.md) V0-engine) gets backup/PITR over blobs for free. Externalization is deferred with a mechanical trip-wire: staging-storage bytes or backup duration crossing a spec-named threshold in the run records. Wiring retained binaries to the [P-0018](../src/adrs/P-0018-core-entity-manifest.md) `attachments` core entity is deferred, self-announcing: the first plugin that needs to FK an ingested binary asks through the ABI. P-0018's note that blob handling "is owned by the storage/ingest layer" is thereby given its V0.1 answer: the staging raw store is the blob substrate.
- **Retention:** per-source-kind, a closed enum `{long-lived, transient}` on the registry (the overview's two archetypes, verified current in §8). **Every V0.1 source kind is `long-lived`** (watched markdown and retained binaries alike — SC3 requires re-derivation with the arrival source absent, so the fs copy is never the system's raw). The `transient` value exists in the vocabulary now (validator-first) but no in-scope kind uses it; the first transient-original source kind (e.g. inline agent edits — an internal-write path, out of ELT scope anyway) arrives with its own intake. *(Anchors: intake HC2 (ELT, LOCKED) + SC2/SC3/SC5; P-0017 D-SoT reconstructability; P-0016 ES-1; Simplicity; P-Defer/DF1 on externalization + attachments wiring.)*

### IP-5 — The `JobQueue` port: minimal, at-least-once, single-claimant; Postgres SKIP LOCKED adapter; `{{P-job-queue}}` opens

Locked as ratified (2a decision 1): this bundle specs a **minimal `JobQueue` port + Postgres adapter**; the harness is its first consumer. The port's contract, expressed in delivery semantics, never in engine idioms:

- Operations: `enqueue(kind, payload, scheduled_for?)` · `claim(kinds, lease)` · `complete(job)` · `fail(job, retryable?)` (bounded attempts, backoff policy) · dead-letter disposition for exhausted retries · `schedule` via `scheduled_for`.
- Semantics: **at-least-once delivery, single live claimant per job** (claim carries a lease/visibility window; an expired claim re-delivers). Consequence bound into the harness contract: **handlers are idempotent** (IP-6) — at-least-once is only safe over idempotent transforms, which the ELT shape already requires (re-run is first-class).
- Job identity: UUID v7; payload JSONB; job kinds are code-registered closed vocabulary (a kind without a registered handler does not enqueue — validator-first).
- **Two-adapter test applied at drafting** (the overview's Postgres-for-now class): the port's verbs and semantics are satisfiable by an SQS-class or Temporal-class adapter — no `LISTEN/NOTIFY` in the contract (a low-latency wake hint is an optional adapter capability; polling is the portable baseline), no SQL-transactional coupling in the port surface. The Postgres adapter implements claim with `FOR UPDATE SKIP LOCKED` per the overview's sketch.
- Queue tables are host-owned operational tables outside the content taxonomy ([P-0022](../src/adrs/P-0022-coordination-cluster.md) D-STORAGE precedent). The queue is a distinct primitive from P-0022's *leases* (actor-held intent across time); a job claim is worker work-distribution — the two share the substrate-atomicity posture and nothing else, stated to prevent a future conflation.
- The `{{P-job-queue}}` ADR slot opens in this bundle (§9); the port is shaped so other subsystems (event-bus consumers, projection refresh) can adopt it later without amendment — but no second consumer is designed now (P-PerRepoFirst posture at subsystem scope).

*(Anchors: 2a decision 1, LOCKED; overview subsystem row 5 (verified current, §8); the two-adapter test as design-time check (P-0010 D5 pattern); P-LockContract; Simplicity — minimal verb set, no priority lanes/cron DSL/workflow engine at V0.1.)*

### IP-6 — Transform-harness conventions (ADR-16 materialized): transform-as-job, update-with-audit re-derivation, lineage, declared determinism

The conventions a transform consumer writes against, without reading pipeline internals (intake SC6):

- **Transform-as-job:** the loader enqueues one transform job per staged row (enqueue is off the critical path — IP-13); workers claim jobs and apply the registered transform for the staging row's source kind. A transform is a registered, versioned unit: `(source_kind, transform_id, version, declared_deterministic: bool) + (staged raw in → canonical candidate rows out)`. Transforms hold no capabilities beyond bytes-in/content-out (extraction transforms additionally run sandboxed, IP-7).
- **Canonical writes:** transform outputs land as ordinary content rows ([P-0001](../src/adrs/P-0001-storage-layout.md) C1) carrying a **`derived_from` soft ref to the staging row** — the [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT lineage clause, verbatim. Source and derived are distinct rows with a known relationship. The staging row is the source-of-record for *re-derivation*; the canonical row is D-SoT's source of truth for the artifact's served data — two truths at two layers, linked by lineage, no contradiction.
- **Re-derivation policy (overview open question 8, dispositioned): update-with-audit.** A re-run (new transform version, or re-arrival superseding content at the same identity) **updates the existing canonical row** — same artifact identity — with the [P-0001](../src/adrs/P-0001-storage-layout.md) system-versioned history pattern as the audit trail. New-rows-append is rejected: it would mint a new artifact identity per derive-run, breaking [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-10 citation stability and pushing supersession bookkeeping into every consumer. Derived is disposable projection (HC2); its history is audit, not identity.
- **Canonical artifact identity scheme (pinned — fold r1):** "same artifact identity" is carried by a **stable, lineage-addressed canonical id** — deterministically derived from the staging lineage (workspace, source, source path — the IP-11 identity minus its content hash) plus transform identity — stable across content-changing re-derivations (update-with-audit's requirement) and reconstructable from staging alone after delete-derived (SC3's requirement, including for a `declared_deterministic: false` transform). Content-hash addressing is rejected for canonical identity: it would mint a new identity on every content change, contradicting update-with-audit and breaking citation stability. The spec SHALL pin the scheme's exact rendering and reconcile [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-10's "content-addressed, per the V0 substrate" citation wording against it — confirming at the gate that PE-10 citation stability is preserved by the stable-id-with-history model, not by content-hash addressing (the two anchors IP-6 leans on — P-0001 system-versioned history and PE-10 — must be read to point the same way, and this is the reading).
- **Re-run is first-class:** delete-derived → re-run → repopulate from staging alone, no read of the arrival source (SC2/SC3 observables). Deterministic byte-identity of outputs is **not** a harness requirement (intake fold M2); the harness requires **idempotent convergence** — re-running an unchanged transform over unchanged raw converges to a stable canonical state. Per-transform determinism is a **declared property** (`declared_deterministic`), testable where declared (seeded re-run byte-compare); the V0.1 extraction transforms declare `true` (parsers), and a future model-backed transform declares `false`. This answers the Frame-carry determinism question: no global pin; a declared, validated per-transform property.
- **Failure:** a transform failure quarantines the staging row with the designed error (IP-14) after retry exhaustion; it never writes a partial canonical set (the canonical write is transactional per staging row).

### IP-7 — Extraction sandbox: egress-denied by construction; mechanism decided per-format on the landed feasibility consult

The egress-deny posture is LOCKED (intake fold M3; TB-1). The **mechanism is a per-format hybrid of two sandboxed lanes**, decided with the backend feasibility consult (landed 2026-07-16 — §13):

- **Lane A — WASM component on the existing substrate (the default lane).** Extraction transforms compile as `wasm32-wasip2` components ([P-0019](../src/adrs/P-0019-plugin-contract.md) D4) executed on the host's Wasmtime engine ([P-0012](../src/adrs/P-0012-plugin-runtime-and-mcp-sdk.md) Decision A; P-0019 D3 floor). Egress-deny is **capability-based**: the component world imports no network interface and no filesystem — bytes in, extracted content out — so egress is structurally impossible, not filtered. Resource bounds ride [P-0007](../src/adrs/P-0007-plugin-resource-limits.md) **mechanisms** (fuel + epoch + memory, kill-and-replace) with extraction-calibrated numeric bounds set at spec — P-0007's V0 values (64 MiB / 10B fuel / 5 s) are plugin-workload-calibrated and are not inherited; large-document extraction sets its own numbers. Extraction components are **host-owned transform units, not plugins**: no manifest, no content types, no [P-0023](../src/adrs/P-0023-plugin-distribution.md) distribution artifact — they ship inside the host binary's trust boundary.
- **Lane B — sandboxed subprocess (the native lane, for formats whose viable crate cannot run in WASM).** The host runs a bundled worker in a sandboxed child. Shape at architectural altitude (consult Q4): Linux — landlock filesystem confinement + a no-interface network namespace for kernel-enforced egress denial, with seccomp-bpf `socket()`/`connect()` denial as the second-choice layer where netns capability is absent; macOS — the subprocess runs inside a container boundary rather than host-level sandboxing (`sandbox-exec` is deprecated with no replacement; reusing an already-audited container isolation beats inventing a weaker host mechanism). The lane's sandbox strength varies by OS where Lane A's is uniform — a recorded cost, which is why A is the default lane. **Lane-B resource bounds (spec-pinned):** the spec SHALL name Lane B's concrete bound set — `RLIMIT_AS` + `RLIMIT_CPU` + `RLIMIT_FSIZE` + wall-clock `SIGKILL` + a bounded asynchronous output read (the host reads child output async under a hard timeout, so a wedged child filling its stdout pipe can never deadlock the host reader) — and assert the no-host-hang property Lane A gets from P-0007 as an explicit Lane-B observable. The two lanes are **not** bound-equivalent (no fuel determinism, no pool-slot poison-and-replace in Lane B) — the inequivalence is recorded here, not papered with "equivalents."
- **Lane-B legality precondition (structural, per host — fold r1).** Lane B is a *legal* lane on a host **iff a structural egress-denying enclosure is verifiably present**: Linux — a no-interface network namespace (or seccomp-bpf `socket()`/`connect()` denial where netns capability is absent); macOS — an actual container boundary enclosing the subprocess. The precondition is a **mechanism requirement verified by a startup/registration lane-availability check**, never an assumed deployment property — a stated-but-unverified egress invariant is exactly the convention-carried guarantee P-GuaranteeByMechanism forbids, and worse than a narrow one. Consequence, stated for the V0.1 dogfood host: on a native (non-containerized) macOS run the precondition is unmet — PPTX (and DOCX, if its spike fails) quarantine `unextractable-here` there until the host provides the enclosure; whether to accept that unavailability or stand up the container boundary for dogfood is the maintainer's operational call at the gate (§15 item 4).
- **Per-format assignment (initial; the criterion below governs reassignment without amendment):** CSV + JSON → Lane A (confirmed viable); XLSX (`calamine`) + PDF (`pdf-extract`/`lopdf`) → Lane A, **subject to the build-verification spike** the consult flags (their WASM-cleanliness is dependency-graph-inferred, not build-verified); DOCX → Lane A contingent (`docx-rust` pulls a tokio/async-zip chain with no clean WASI story — the spike decides; failing it, DOCX joins Lane B, with `docx-rs`'s read path as the alternate to verify); **PPTX → Lane B** (no mature WASM-clean crate exists today: the actively-maintained `pptx-to-md` carries a non-optional `rayon` dependency that cannot run single-threaded WASI, and the only WASM-clean candidate is maintenance-disqualified — the consult's named gap, resolved by the native lane where rayon is unproblematic).
- **The criterion, binding at implementation:** a format runs Lane A iff its crate **compiles to `wasm32-wasip2` and performs within the P-0007-class budgets on representative documents**, verified by an actual build+perf spike at implementation start (the consult's flagged next step — its two Lane-A claims are inferred, and per P-VerifyInheritedState the spike is the re-verification, owed before the assignment hardens). Transform registration carries the execution lane per format, so reassignment is a registration change, not a design change. **The lane-selection terminal is fail-closed (fold r1):** a failing spike moves a format to the subprocess lane, never to an unsandboxed path — and a format that clears **no** lane whose egress-denying precondition holds on the running host does not run at all: it quarantines with the `unextractable-here` rejection class (IP-14). There is no unsandboxed fallback anywhere in the selection function; "never to an unsandboxed path" is thereby carried by a terminal rule, not by an assumption that Lane B is always available.
- **Invariant across both lanes:** the XML parser families in every viable crate (`quick-xml`, `xmlparser`/`hard-xml`, `roxmltree`) implement **no entity resolution at all** — internal or external — so the XXE substitution machinery is absent by construction (consult Q3, source-verified), *and* the sandbox denies egress structurally regardless — defense in depth; the zero-egress promise never rests on parser behavior alone. Crate picks are Green-tier + RUSTSEC-screened per intake HC10; the consult's Red-tier exclusions are adopted outright (AGPL `mupdf`; GPL-linked `poppler`; `pdfium-render` additionally disqualified as native-FFI).

*(Anchors: intake fold M3 + HC6 + HC10; the backend feasibility consult, landed 2026-07-16 (§13); P-GuaranteeByMechanism — capability/OS-mechanism over convention; P-VerifyInheritedState — inferred compile claims re-verified by spike before hardening; P-0012/P-0019/P-0007 — reuse of locked substrate; P-StackDiscipline; Security | Simplicity default-to-Security — the in-process no-sandbox option is rejected (TB-1).)*

### IP-9 — Ingest writes run under a system principal bound to the source's workspace

### IP-11 — Content identity + dedup: BLAKE3, path-scoped, observable no-ops

Staging identity is **`(workspace_id, source_id, source_path, content_hash)`** with `content_hash = BLAKE3(raw payload)` (the in-corpus hash precedent: [P-0008](../src/adrs/P-0008-admin-token-shape.md)'s BLAKE3 token hashing). ### IP-12 — Recency instrumentation (D1): two ingest-owned spans; composite by join; bound set at spec

Ingestion recency is a designed, measured axis (intake HC5/SC7; D1 ratified 2026-07-06 — distinct from live-data awareness, which is out of identity). The instrumentation shape:

- **Ingest-owned spans, emitted per source kind:** `arrival→staged` (arrival detection to durable staging commit) and `staged→canonical-committed` (staging commit to canonical transform commit). Recorded on run/job records (plain workspace-scoped timestamped tables — the RA-7 family; P-0010 D8), emitted as telemetry per the observability baseline.
- **Emission redaction (fold r1):** ingest telemetry and run-record emission carry **derived IDs only** — source paths are hashed in emitted lines, and frontmatter content and titles never appear in emitted records. The observability baseline's emit-not-store discipline excludes artifact *bodies*; this extends the exclusion explicitly to source-path/title *metadata* (the source-corpus threat row's own mitigation, carried into ingest rather than assumed). The stored run-record rows (workspace-scoped, admin-read) may hold real paths; the redaction discipline binds what is *emitted*.
- **The composite `arrival→indexed` span is computed by join**, not by coupling: ingest run records join the retrieval cluster's index-build records (spec R-0034-d — the run-records family is verified against P-0014 RA-7; the `-d` sub-id is confirmed against the retrieval spec at spec render) on artifact id + timestamps — both sides backfillable from stored rows (IB1). "Queryable" in the intake's sense is thereby honestly decomposed: canonical-committed content is reachable by exact-key/citation surfaces immediately; dense/FTS retrieval reachability lands at the next index run, whose cadence the retrieval subsystem owns.
- **The spec states the bound** for the ingest-owned spans (the number is spec work, per the intake). The composite bound depends on index-run cadence — deferred, named: if the measured composite breaches the spec's stated recency expectation, an **event-triggered index-refresh hook** (ingest completion signaling the index scheduler) is the pre-identified mechanism; the recency measure itself is the mechanical firing instrument (DF1). No coupling is built now — the subsystems stay independent (§8's no-retrofit reconciliation depends on exactly this restraint).

### IP-13 — Load-shedding contract adopted: staging is the critical path; transforms absorb backlog

### IP-14 — Failure path: closed rejection vocabulary, fail-closed, working-state → working-state

- **Rejection classes are a closed enum** (validator-first; extension is a spec amendment): `malformed-file · unparseable-binary · oversized-input · decompression-bomb · path-traversal-name · unregistered-source · transform-failure · unextractable-here` (the intake SC10 set + transform exhaustion + the IP-7 fail-closed lane terminal — a format clearing no egress-denying lane on the running host quarantines under this class rather than running unsandboxed; fold r1). Each rejection is a distinct designed error with a per-class error-path test; each quarantines (staging row `status = quarantined` + `rejection_class` + diagnostic) or refuses pre-staging (unregistered-source, path-traversal — nothing to stage), never partially writes.
- **No partial canonical write:** the canonical transform write is transactional per staging row; failure rolls back whole. The staging write is transactional; the enqueue follows **post-commit, best-effort** (fold r1 — a cross-resource staging+enqueue transaction would put SQL-transactional coupling in the `JobQueue` port surface, falsifying IP-5's two-adapter portability): a staged-but-unenqueued orphan is re-enqueued by the reconciliation scan (IP-3), so the no-partial-write property lives on the staging and canonical writes, and staged-durability-plus-reconciliation is the delivery guarantee.
- **Working-state → working-state** (intake SC9, the maintainer's determinacy ruling applied ingest-wide): watcher crash → supervised restart + reconciliation scan (IP-3/IP-8); kill mid-load → the transaction never half-commits, and the next scan re-observes the source; transform kill/failure → job lease expires, at-least-once re-delivery, idempotent handler converges (IP-5/IP-6); residue of any interrupt is inert (an unclaimed job, an un-transformed staged row — both re-processable, neither corrupting). Kill-mid-operation tests per pipeline stage are the spec's observable.
- **Quarantine is inert, visible, and bounded (fold r1):** quarantined rows never reach transforms or canonical; they are listable via the admin CLI and counted in run records; release-from-quarantine (after an operator fix) is a re-scan-driven re-arrival, not a mutation path. Two bounds the prior draft left unstated: (a) **the quarantine store carries an aggregate bound** — the per-file size cap bounds one file, not a malformed-file flood, so the spec SHALL place an aggregate quarantine size/count bound with an eviction/rotation policy (a designed bound, not an attacker-controlled unbounded write); (b) **"inert" holds only if nothing re-parses it** — quarantine listing and inspection are metadata-only (path, `rejection_class`, diagnostic; never a payload render), and payload retrieval, if ever offered, is an explicit operator action that re-enters the sandboxed extraction path, never an implicit display.

*(Anchors: intake SC9/SC10 + the W2-1 determinacy ruling as scoped by the intake; P-SecurityLayered fail-closed; P-ShiftLeft D2 — closed enums carry validators; P-TrustworthySignal.)*
