## 1. Purpose / context

Multiple concurrent instances of the orchestrator operate in one workspace today and cannot coordinate safely. Resource claims are advisory files with no atomic acquisition and no liveness binding; handoffs carry no delivery or consumption state; cross-session continuity rides hand-rolled carrier files whose every claim a zero-trust consumer re-verifies from scratch. The intake's evidence section records the failure classes live: a handoff consumed with no trace, an addressed item no instance could safely archive, a live instance's claim file adopted by another session after a context reset, and a documented archive convention that decayed silently despite exact commands — file conventions have no mechanism by which they hold (the constraint the workspace canon names generally: a guarantee carried by convention breaks silently under agent authorship; P-GuaranteeByMechanism).

- **What this designs.** The coordination cluster inside the mnemra-core host: durable actor identity over the P-0018 `actors` core entity, a lease subsystem for mutual-exclusion claims on the duration line, a message subsystem with addressed delivery and disposition state, two agent-visible MCP tools (`claim`, `message`) on the existing stdio server, an operational + audit observability shape, and the client-facing availability contract. This is the system's first live workload by design intent: a coordination plane over **zero migrated content**, defining what "the workspace starts using mnemra" means — migration step 1 of the strangler program (intake Hard constraint 3).
- **What this does not design.** No content migration, no interim file-convention tooling, no workflow primitives in core (P-0018 D-BOUNDARY untouched), no push transport, no offline/queued coordination, no composer/optimization tooling, and no stash-successor as a designed surface — the session-carrier is a consumer pattern of messaging (intake Non-goals 1–7, all carried).
- **Intake.** [`docs/intent/coordination-wedge.md`](coordination-wedge.md) — locked 2026-07-06, high stakes, `spec_type: code` with `--designed-tier` as the sanctioned pre-implementation verify mode, Stage 1c security review round 1 zero blocker/high.
- **Design substrate.** The merger-role decision record (2026-07-05 — the merge-request particulars that become the founding message type, and the instance-vs-agent tripwire this Frame dispositions); the strangler-migration decision walk (2026-07-06 — one-spine sequencing, the availability matrix, the cutover ritual, the flip rules); the memory-taste decision record (2026-07-05 — wedge-core pre-confirmation); the permissions-model research brief (2026-07-05, locked — actor-as-principal spine over four enforcement planes; consumed here as a posture note only, per ratified direction 9).

## 4. System boundary + component map

The coordination cluster is a **host subsystem** of mnemra-core — host code above the plugin boundary, beside (not inside) the retrieval subsystem, sharing the substrate the host already owns:

- **Actor identity (consumed, not built).** The P-0018 `actors` core entity, populated through the identity builtins (`P-builtin-users` / `P-builtin-agents`, P-0002). The wedge adds *usage semantics* — role-instance grain, session attachment, liveness — not a new identity table.
- **Lease subsystem.** A first-class `leases` entity (holder actor FK, structured resource identifier, acquisition time, TTL/expiry, state) plus the acquisition/renew/release/takeover operations with substrate-enforced mutual exclusion. Host code + one coordination-owned table.
- **Message subsystem.** A `messages` entity (sender actor FK, addressee actor FK, message-type + schema-version, schema-validated payload, delivery/ack/disposition state, timestamps) plus a host-owned message-type registry (C6). Host code + coordination-owned tables.
- **Verb surface.** Two agent-visible MCP tools — `claim` and `message` — registered on the existing stdio MCP server as host-served verbs, exactly as the retrieval verbs are (retrieval Frame R1 precedent). Actions ride as a closed enum parameter per tool.
- **Session plane.** Actor resolution and attachment ride the session builtin (`P-builtin-sessions`): the session presents its role-instance, the host resolves-or-creates the actor row through the identity-builtin path and holds the attachment as session state. No per-write actor parameter exists on the tool schemas.
- **Observability emission.** Operational log entries at every coordination operation through the P-0011 facade; audit emission for the privileged subset to the P-0018 D-SURFACE core emit-target.
- **Storage.** Coordination tables under the engine-agnostic `Storage` trait (P-0010), workspace-scoped (P-0006), hard-FK to core entities only (P-0018/P-0017).

### Zero content-machinery dependency as a boundary property

Intake Hard constraint 1 and Success criterion 6, stated as a property of the boundary rather than a wish: coordination tables carry **no reference of any kind — FK, soft ref, join, or read path — into content tables**; the two tools touch no retrieval, provenance-envelope, edge, or index machinery; message payloads are schema-validated JSON owned by the coordination store, not content artifacts; and coordination rows are not served by `get_context_for` at V0 (coordination is not context; the re-open is a named tripwire, §10). A fresh deployment with only core-entity and coordination rows supports full fleet coordination. *Binary-observable (SC6): a schema and code audit finds no dependency from the coordination cluster on content tables or corpus population.*

## 5. Trust boundaries + risk profile (resolved)

The intake flags the wedge as touching a trust boundary and hands this Frame a five-item threat-modeling set with a stated attribute priority: **integrity** (exactly-one-holder, no traceless consumption) and **availability** (fail-closed, no split-brain) are load-bearing; confidentiality is secondary in a single workspace, with the instance-isolation default as its one edge. Each item gets a resolution or a named, fireable deferral.

**TB-1 — Instance-grain isolation default (resolved; ratified direction pressure-tested).** The ratified default: reads default-visible within the workspace; writes strictly authority-scoped; default-deny re-opens at the multi-team trust boundary. Pressure test against the attribute priority: the confidentiality cost of visible reads is coordination metadata (who holds what, who has mail pending) **and message payload content** — a `merge-request`'s particulars, a `handoff` body are readable by any resolved actor under the default; the cost is recorded at its accurate extent (security-review r1 correction), not understated as metadata-only. That is still strictly less exposure than the current convention, where every instance reads every claim file and the whole inbox — bodies included — by design, among actors already inside one trust boundary sharing one filesystem and one credential today. The integrity risk of visible reads is nil: visibility grants no write authority anywhere (the write-authority matrix below is unconditional on visibility). The counterfactual — default-deny with per-actor grants — would add a grant model the auth bundle owns later (ratified direction 9), would break the live "do not mutate a claimed resource" workflow that *requires* reading others' claims, and per the Security value's own usable-security clause, a control whose friction drives bypass has failed as security. Verdict: the default stands; the residual (below) is stated, not waved. *(Anchors: intake attribute priority; Security value usable-security clause; Security | Simplicity edge — the axis is named, the concession is ratified and tripwired.)*

**TB-2 — Takeover authority (resolved).** Takeover is a distinct, privileged, always-audited action — never a flag on acquire. At V0: any workspace actor may take over a lease **whose TTL has expired**, through the explicit `takeover` action, which records prior holder, new holder, and the expiry evidence to the audit surface. A live (unexpired) lease cannot be taken over through the tool surface at all — there is no force-break verb at V0; a genuinely wedged live lease waits out its TTL (bounded by lease-duration policy), and an emergency break is a maintainer operation at the store, audited. Deferral, named: a force-break verb (with its own authority scoping) designs when an operator first hits a wedged live lease in practice — self-announcing. One further bound is deliberately absent at V0 and named so it is not a silent omission: renew is holder-only and **uncapped**, so a live-and-renewing holder can hold a resource indefinitely; total-hold bounding (a max-renewal or absolute-hold cap, operator visibility of long-held leases) is nuisance-class at trusted single-team scope — visible in the op log — and re-examines alongside the force-break deferral and the C8 isolation tripwire. Takeover races are backstopped by the availability contract: a deposed holder's next coordination write fails closed and stops the session (C12), so two actors cannot both proceed as holder even across a takeover misfire. *(Anchors: P-LeastAuthority — takeover authority scoped to the recovery job, not standing break authority; intake SC3; P-Defer/DF1.)*

**TB-3 — Registration authenticity (resolved with stated residual).** Who may register as which role-instance: at V0, any session authenticated at the workspace boundary (the admin-token era) may declare a role-instance and attach. Actor identity is therefore **declared, not proven**, inside the workspace trust boundary — the wedge's version of claim-adoption is possible for a hostile or confused co-tenant session. Structural mitigations that hold now: (a) the acting principal is host-derived session state, never a caller-supplied per-write parameter, so a write cannot silently assert a different actor than its session attached as (P-0019 context-derivation precedent generalized); (b) **at most one live session attaches per actor row** — attachment liveness is held under the same stale/TTL discipline as leases, so attaching to an actively-attached actor is refused mechanically, and succeeding a *dead* session is the designed recovery path (SC1) rather than adoption; (c) every attach/resolve event is audited (C10). The residual — a co-tenant session attaching to an idle role-instance it should not speak for — is accepted at single-team scope and closes when the auth bundle delivers per-actor credentials (an identity-provider subject claim mapping to `actor_id`; the actor-as-principal spine). The multi-team tripwire re-opens the whole default before any second team exists. *(Anchors: intake risk profile — "impersonation is the wedge's version of claim-adoption"; permissions-model research headline, consumed as posture; P-LockContract — the one-live-attachment invariant is intrinsic and locks now.)*

**TB-4 — Addressing authorization (resolved).** Within the workspace: any resolved actor may send to any role-instance (open addressing matches the live inbox semantics and keeps the mechanism smallest); only the addressee acks or dispositions its messages; send always executes under the sender's host-resolved actor context, so every message is attributed — anonymous or spoofed-sender messages are structurally impossible. No rate or volume limiting exists on `send`/`acquire` at V0 — named deliberately, not omitted silently: mis-addressing, queue flooding, and lease churn are nuisance-class at workspace scale, visible in the op log, and re-examine at the C8 isolation tripwire alongside the visibility default (carried to the spec as a stated non-control, not designed here). *(Anchors: intake SC4; P-LeastAuthority on the disposition side; ratified direction 5.)*

**TB-5 — Permissions-substrate posture (name-and-defer, per ratified direction 9).** The wedge's authority rules (the write-authority matrix below) are host-enforced data-level rules at the V0 enforcement point (application-layer checks under `WorkspaceCtx`; the P-0008/P-0009 era). They are written to be *expressible as policy rows later*: when the auth bundle lands its four-plane composition, these rules become the coordination plane's policy content, with actors already the shared principal vocabulary. No policy-engine integration lands now; this paragraph is the Frame's entire dependency posture toward that bundle. *(Anchor: ratified direction 9; the sequencing constraint — wedge first.)*

**Write-authority matrix (the data-level rules TB-5 defers into policy later):**

| Operation | Authorized principal (V0) | Audited |
|---|---|---|
| resolve/attach (register) | any workspace-authenticated session, onto an **`actor_type = agent`** row with no live attachment (attach as `human`/`system` is refused — C3); succession over a stale/dead attachment is an **audited takeover recording the prior attachment holder + staleness evidence** (TB-2 parity) | yes (audit surface) |
| lease acquire / renew / release | any resolved actor / holder only / holder only | op log (acquire, renew, release); refusals logged |
| lease takeover | any resolved actor, expired leases only | yes (audit surface) |
| message send | any resolved actor | op log |
| message ack / disposition | addressee only | ack: op log; disposition: audit surface |
| message poll (own queue — the delivery + attach read) | the polling actor, self-scoped | op log |
| message/lease list (the workspace-visible read) | any resolved actor | op log |

**Residual risks, stated:** (1) declared-not-proven actor identity within the workspace boundary until per-actor credentials land (TB-3); (2) coordination metadata **and message payloads** visible to all workspace actors by design (TB-1); (3) TTL-expiry takeover can depose a slow-but-alive holder — bounded by renew discipline plus the fail-closed backstop, and the deposition is audited (TB-2); (4) open in-workspace addressing admits queue noise, unbounded by rate limits at V0 (TB-4); (5) **the exactly-one-holder guarantee is on the lease *row*** — for resources outside the system (git repos, config, filesystem) the lease is advisory, and a mid-action TTL expiry during a service outage (the C12 read-only-finish window, where renew itself fails closed) does not gate an in-flight external side effect already underway; the takeover path is also a coordination write and equally fails closed during the outage, so the exposure is the narrow expiry-during-outage-during-external-action window, bounded by lease-duration policy sized to the largest external unit of work — partly inherent to advisory leases over external resources. Each is accepted at single-team dogfood scope with its named re-open condition; none is silent.

## 6. Locked architectural directions

Each direction is locked with its canon anchor at the decision-and-rationale line (decide-and-lock posture). Directions render the ratified 2a items at Frame precision; none re-opens them. §7 carries the binary measures.

### C1 — Residence: core host subsystem (the P-0002 walk, recorded formally)

**Decision:** The coordination cluster is **core** — host code above the plugin boundary, per ratified direction 1, satisfying intake Hard constraint 7's required formal disposition. *(Anchors: P-0002 discriminator; P-0018 — actors are the core entity; precedents: retrieval as host subsystem (retrieval Frame R1 / P-0014), report definitions as host-core-owned (P-0021); wedge-core pre-confirmed at the memory-taste decision walk, 2026-07-05.)*

The P-0002 criterion, walked:

- **Verb-on-content?** No. Acquire/release/takeover/send/ack/disposition are state transitions on coordination substrate, not CRUD against content-artifact rows. Coordination rows carry no document identity (no frontmatter/body, no provenance envelope); they are operational state.
- **Foundational-substrate dependency?** Yes. Actors ARE the P-0018 core entity, populated by the identity builtins at bootstrap; claims and messages key off actor identity; and safe multi-instance operation is a precondition for plugin-era work, the same "must exist before and independent of plugin work" property that put workspaces, users, agents, sessions in the builtin tier.
- **Bootstrap ordering:** coordination state must be usable the moment a session binds — before any plugin loads. Plugin residence would invert the ordering.
- **Hard constraint 1 (standalone)** is satisfied trivially by core residence: no plugin machinery, no content dependency.

**Consequences, stated precisely (intake fork 6 — verb shape × residence, resolved together):** nothing exports through the plugin ABI, so **the P-0013 deferred-domain-verb tripwire does not fire** — its firing condition is "the first domain/non-CRUD verb that must dispatch at the V0 [plugin export] surface," and host-served MCP verbs are not plugin exports (the retrieval cluster's host-placement precedent: "host placement keeps that trip-wire unfired"). The WIT world and the fixed typed `content` interface are untouched by this cluster. The signing chain (intake Hard constraint 6) is likewise satisfied by residence: no new signed plugin artifact is minted; the subsystem ships inside the host binary's existing trust boundary and P-0005's plugin-load verification surface is unchanged.

### C2 — Agent-visible verb surface: two tools, actions as parameters; the budget quantified

**Decision:** Exactly two agent-visible MCP tools ship with this cluster — **`claim`** (actions: `acquire`, `renew`, `release`, `takeover`, `list`) and **`message`** (actions: `send`, `poll`, `list`, `ack`, `disposition`) — each action set a closed enum in the tool schema; the exact action spelling is pinned at spec, the two-noun surface and closed-enum shape are locked here. *(Anchors: ratified direction 2; the verb-budget constraint's falsifiable form — content-type/action as parameter, never a tool name; P-ShiftLeft D2 — closed enums get mechanical validators; P-0009 verb-category gating applies per action.)*

### C3 — Actor grain, registration, and attachment

**Decision:** Each role-instance (operator, design lane, merger, research lane, …) is one durable `actors` row, `actor_type = agent`, at the P-0018 grain — **no actor↔session intermediate entity** (intake fork 7, confirmed). A session **resolves-or-creates** its role's actor row through the identity-builtin path and **attaches**; the attachment is host-held session state, and the acting principal on every coordination write is derived from it — **never a caller-supplied per-write parameter** (the P-0019 host-derived-context rule, generalized from `workspace-ctx` to the actor). **At most one live session is attached per actor row**; attachment liveness rides the same stale/TTL/takeover discipline as leases — a successor session after a dead one resolves the *same* actor row and re-attaches (SC1), while attaching to a *live*-attached actor is mechanically refused. Two invariants harden this surface (folded from security-review r1, F5/F7): **succession over a stale/dead attachment is an audited takeover at TB-2 parity** — the audit record carries the prior attachment holder (session) and the staleness/expiry evidence, because an actor-identity succession has *higher* integrity stakes than a resource-lease takeover (it governs the attribution of every subsequent write); and **registration/attachment is restricted to `actor_type = agent`** — attach validates the resolved row's type and refuses `human` or `system`, so a session cannot launder its writes as host-initiated (`system`) or human action (the P-0018 D-ACTOR three-facet alignment stays intact by mechanism, not by the builtin path's implication). The attachment TTL is **security-load-bearing, not only an availability knob**: it sets the TB-3 impersonation-after-death window on one side and successor-lockout on the other — its spec-tier calibration runs under the security lens on exactly that tension. The preferred realization implements attachment under the lease mechanism itself (an attachment is held intent on the role-instance — one mechanism, two uses); the spec may realize it as dedicated columns on the session/actor linkage instead, provided the invariants (one live attachment; agent-type-only; stale-recoverable; takeover-audited with prior-holder evidence) hold — the invariants are locked, the realization is not. *(Anchors: ratified item 6 / intake fork 7; intake SC1; P-0018 D-ACTOR; P-0019 context-derivation precedent; P-GuaranteeByMechanism — type restriction enforced, not implied; Simplicity — one liveness mechanism preferred; P-LockContract — invariants lock, realization varies.)*

Canonical bind moment: the session-start `poll` (C7) carries the declared role-instance; the host resolves, attaches, audits the attach event, and returns the queue plus the workspace's live leases — parity with today's session-start injection in one call. The mechanism by which a session presents its role-instance declaration (session metadata vs first-call parameter) is spec-tier.

### C4 — Claims: the duration line; leases as the first-class entity

**Decision:** The claims split follows **duration, not residence** (ratified items 3+4, rendering the intake fork-2 refinement): an **atomic write** to in-system state is a storage transaction — the engine's semantics absorb it, no claim entity involved (P-0010 `Storage` trait); **held intent** — an hour-long design-doc edit, a repo's main-merge lane, a config surface, any resource inside or outside the system held across time — is a **lease**: a first-class entity with holder (actor FK), structured resource identifier, TTL, and explicit state. Acquisition is **substrate-enforced atomic**: a uniqueness-constrained insert / compare-and-set inside a transaction on the coordination store — never application-level check-then-act, never advisory-file semantics. Exactly one live lease per resource can exist (SC2's mechanism, picked: substrate-level atomicity); the losing contender receives a structured, logged refusal. Stale semantics (SC3): the holder renews within TTL; an expired lease is takeover-eligible via the explicit audited `takeover` action (TB-2); no manual-wipe limbo exists — every dead-holder path ends in a mechanical recovery rule. TTL bounds, renew cadence, and the resource-identifier namespace scheme (a registered structured-string scheme, e.g. repo-lane / config-surface / document families) are spec-tier calibration behind the locked contract. Today's instance-claim files rehome here; the merger lane's main-merge lease is the first designed consumer. *(Anchors: ratified items 3+4; the strangler-migration decision walk Q1 extension 2 — the duration-line refinement is the maintainer-walked substrate; intake SC2/SC3; P-GuaranteeByMechanism — the invariant lives in the substrate, not a convention; P-LeastAuthority on takeover.)*

### C5 — Messages: rows with queue semantics; the disposition state machine

**Decision:** Messages are **rows in a coordination-owned table** (intake fork 1, resolved row-shape per ratified item 3): sender and addressee actor FKs, message-type + schema-version, schema-validated JSON payload, and a **closed-enum lifecycle** — delivered → acknowledged → dispositioned, with a closed terminal-disposition vocabulary (spec pins the members; the shape locks here). Every consumption leaves a record (actor + timestamp at ack and at disposition); every item has a disposition path. Two invariants make SC4's "structurally impossible" claim structural rather than an artifact of the current verb list (folded from security-review r1, F4): **a message's payload, sender, addressee, type, and schema-version are immutable after send** — no edit, recall, or retraction path exists at any layer, not merely no verb today; and **the lifecycle is monotonic with append-only consumption records** — delivered → acknowledged → dispositioned admits no reverse transition, and a disposition cannot be overwritten or reverted (the same append-only integrity shape P-0018 D-SURFACE applies to audit, applied to the message store). Deletion is not a message-lifecycle operation at V0 — deleting a pending message is retraction by another name, and deleting a dispositioned one erases the consumption trail; any future retention/archival policy is spec-tier (or event-bus-ADR-tier) and preserves consumption records under the same append-only shape. With those invariants locked, the intake's traceless-consumption and unarchivable-item failure modes are **structurally impossible**, not procedurally discouraged (SC4). Addressing is role-instance (the actor row), so a queue survives session death exactly as claims do (intake fork 4, confirmed). Messages are not content artifacts: no provenance envelope, no retrieval, not served by `get_context_for` at V0 (coordination is not context; named tripwire, §10). *(Anchors: ratified item 3; intake SC4; Hard constraint 1 — the artifact-shape alternative presumed content machinery and was rejected at ratification for exactly that dependency; P-0017 D-SoT — coordination rows are their own source of truth; P-GuaranteeByMechanism.)*

### C6 — Message-type registry governance (residual 1, discharged)

**Decision:** Message types are a **host-owned, code-registered closed set at V0**. A message type = a named, versioned schema shipped with the host; `send` validates the payload against the type's schema at write and **fails closed on violation** (an invalid coordination write is a refused write, C12). Defining or evolving a type is governed by the **same designed-tier pipeline as any core contract change** — spec/ADR + review + the merge gate — not by a runtime grant model: the governance mechanism is the one that already exists, which is the smallest mechanism that satisfies the requirement. Schema evolution is **additive-only within a version**; a breaking change mints a new schema version on the same type name, and consumers read the version field. The **founding registration is `merge-request`**, carrying the merger-role input contract's locked particulars — repo · branch + worktree ref · governing-artifact + marker refs · gate facts · ride-alongs · base-pin / expected conflicts · CI expectations (the merger-role decision record, 2026-07-05) — every particular a schema field, none falling back to freeform prose (SC5's schema-audit observable). A second founding type, a minimal generic **`handoff`** (subject + body + artifact refs), covers unstructured instance-to-instance handoffs so ordinary inbox traffic has a home without per-case type-minting; it is deliberately minimal to avoid designing the stash-successor surface (intake Non-goal 7; §12 strain 2). Deferral, named: a **data-driven runtime registry** (types defined without a host release) designs when the first out-of-host author — an adopter or a plugin — needs to mint a type; self-announcing. *(Anchors: Simplicity — smallest mechanism, governance rides existing gates; P-ShiftLeft D2 validator-before-field — a type without a mechanical schema validator does not register; P-0018 closed-enum precedent (D-ACTOR); P-Defer/DF1; intake SC5.)*

No push, no long-poll, no notification daemon at V0; push rides the streamable-HTTP tripwire (intake Non-goal 4) and its arrival re-arms the merger tripwire (C9). *(Anchors: ratified item 7; parity with the live session-start injection semantics — dogfooding as forcing function; P-Defer.)*

### C8 — Isolation defaults

**Decision:** As ratified (item 5) and pressure-tested (§5 TB-1): **reads default-visible within the workspace** — any resolved actor may list live leases and read queues; **writes strictly authority-scoped** per the §5 write-authority matrix; **default-deny re-opens on the multi-team tripwire** — the first deployment tier that crosses single-team trust re-opens the visibility default at its own intake, before any second team exists. ### C9 — Merger-shape tripwire disposition: keep instance; re-arm on push transport

### C10 — Observability shape (ratified item 11, as amended)

**Decision — firm requirements:** (a) **every coordination operation emits an operational log entry** — register/attach, acquire, renew, release, takeover, send, poll, ack, disposition — including **refusals** (the losing contender's mechanical refusal is evidence; P-TrustworthySignal) — with outcome and actor attribution; (b) the **privileged/integrity-bearing subset — takeover, registration/attachment events, disposition — additionally emits to the P-0018 D-SURFACE audit surface** (append-only, artifact-outliving, host-fn/core-mediated): the wedge emits to the surface the core already locks, it does not invent a store; (c) **every entry carries a timestamp.** The cluster is a production surface at first fleet use and ships instrumented in the same increment as the verbs — not after the first incident (P-InstrumentBefore; there is no pre-existing coordination baseline to backfill — the cluster starts empty by design, so day-one forward capture *is* the baseline). Emission sites ride the P-0011 `tracing` facade (library macros; binary-owned subscriber).

**Topology (maintainer-ratified 2026-07-06, on the log-topology survey):** operational logging is **one unified structured tracing stream** — subsystems as **targets** (per-target filtering), **spans** carrying causality, **filtered subscriber layers** routing the one stream to its sinks — composed behind the P-0011 facade (emit sites unchanged; topology is subscriber composition in the binary). Per-subsystem log files rejected at the ratified call. **Audit rides existing canon, nothing new:** the privileged subset emits to the P-0018 D-SURFACE core emit-target exactly as D-SURFACE already defines it — this Frame invents no audit semantics; audit-store detail belongs to the event-bus ADR P-0018 already names. **One constraint:** no log-store infrastructure (TimescaleDB-class or otherwise) is stood up for log capture — the stream writes to file/stdout, and an external log-capture mechanism (system log directories, rotation, per the deployment) handles the rest.

**Emit-guarantee posture (the Frame's answer to "is a privileged action guaranteed audited" — security-review r1 F1; gate-evidence for the Frame-exit walk):** a privileged coordination write (takeover, registration/attachment events, disposition — the same subset requirement (b) enumerates) is **not observably complete until its audit record is captured — fail-closed on audit-emit failure, the same contract as on state-write failure (C12)**; an un-audited privileged transition is the repudiation the audit surface exists to prevent, so it is an integrity invariant of the operation, not a store question. The named resolution direction, strictly within the maintainer-locked options: the **outbox composition** — the audit row commits atomically with the state transition host-side, and external capture reads from there — which satisfies the guarantee while keeping the file/stdout landing and the no-log-store constraint intact; mechanism detail stays with the emission layer / event-bus ADR. The maintainer has additionally **pre-blessed audit-as-DB-table with reports generated from it as an acceptable landing** ("that works too," 2026-07-06); the event-bus ADR formalizes rather than re-opens that choice, and the client-visible contract stays the host-fn call either way. *(Anchors: the log-topology survey (maintainer-ratified 2026-07-06); P-0018 D-SURFACE; P-0011 — lock the contract (emit sites), vary the implementation (subscriber composition); P-0010 D8 — the same no-specialized-log-store posture; P-InstrumentBefore; P-SecurityLayered fail-closed; Observability | Simplicity edge resolved toward Observability.)*

### C12 — Availability contract (client semantics at Frame altitude)

**Decision:** The ratified availability matrix (intake Hard constraint 4; the strangler-migration decision walk Q3) renders as client-contract semantics:

| Condition | Contract |
|---|---|
| Session start + service down | **No work.** The session does not fall back to file conventions — there is no fallback substrate after cutover. |
| Mid-session + service down | The current unit may finish **read-only**; **all writes fail closed**; **no local write queue, ever** (reconciliation = split-brain by construction; intake Non-goal 5); the session then stops. |
| Coordination-write failure (any cause) | **Immediate, observable stop.** The failure renders as a structured error state the client surfaces as a stop — never empty-success, never silent proceed-unclaimed/unacked (SC7). Proceeding unclaimed defeats the wedge. |

Fail-closed is also the **integrity backstop**: a deposed lease holder (TB-2 takeover race) or a session whose attachment lapsed discovers it at its next coordination write, which fails closed and stops the session — two actors cannot both proceed as holder under any interleaving. Offline mode remains a planned future choice on the topology tripwire recorded program-side, never an automatic crash fallback. *(Anchors: intake HC4/SC7; ratified item 10's source walk; P-TrustworthySignal — structured states, the retrieval cluster's structured-unavailable precedent; P-SecurityLayered fail-closed.)*

## 7. Quality-attribute scenarios (ATAM)

The binary measures the spec renders into acceptance tests. Each traces to an intake success criterion. QA-1 through QA-6 are measurable stimulus/response scenarios; QA-7 is a forward pointer, not a peer scenario.

- **QA-1 — Exactly one holder under contention (SC2).** *Stimulus:* two sessions concurrently acquire the same resource. *Environment:* normal operation. *Response:* exactly one acquisition succeeds; the loser receives a structured refusal, and the refusal appears in the op log. *Measure:* a concurrent-acquisition test across repeated runs never observes two live holders; the refusal is machine-readable, not an error string.
- **QA-2 — Dead-holder recovery (SC3, SC1).** *Stimulus:* a lease holder's session is killed without release. *Environment:* holder's TTL elapses. *Response:* a successor session resolves the same actor row (no duplicate minted), re-attaches, and acquires the resource via takeover per the stated rule; the takeover is on the audit surface with prior-holder evidence. *Measure:* recovery requires no manual state wipe by the maintainer; the audit row exists.
- **QA-3 — No traceless consumption, no unarchivable item (SC4).** *Stimulus:* a message is polled, acked, and dispositioned by its addressee. *Response:* consumption records (actor + timestamp) exist at ack and disposition; a non-addressee's ack/disposition attempt is refused. *Measure:* every terminal message state carries its consumption trail; no reachable state lacks a disposition path; no path exists that mutates payload or addressing post-send or reverts a disposition (the C5 immutability + monotonicity invariants).
- **QA-4 — Founding-type schema audit (SC5).** *Stimulus:* a schema audit walks the locked merge-request particulars against the `merge-request` message type. *Measure:* every particular maps to a schema field; zero particulars fall back to freeform prose; an out-of-schema payload is refused at send.
- **QA-5 — Standalone cluster (SC6).** *Stimulus:* a fresh deployment with only core-entity and coordination rows. *Response:* full fleet coordination (attach, lease lifecycle, message lifecycle) functions. *Measure:* schema + code audit finds no dependency on content tables or corpus population.
- **QA-6 — Fail-closed write (SC7).** *Stimulus:* the service becomes unavailable mid-session; the session attempts a coordination write. *Response:* structured failure; the client contract surfaces a stop; no queueing, no silent proceed. *Measure:* the write is provably absent from the store; the session's continuation attempt after the stop is refused per the availability contract.
- **QA-7 — Flip dry-run (SC8) — forward pointer.** SC8's observable (the dry-run flip plan) is a spec-tier check, not renderable as a Frame-tier stimulus/measure scenario; the design property that makes it satisfiable is recorded in §8.

## 8. Cutover obligations (program constraints the design satisfies)

The flip rules and the per-function ritual are program obligations from the ratified strangler mechanics (ratified item 10; the strangler-migration decision walk Q4/Q7); this Frame records how the design satisfies them **by construction**:

1. **Small-batch flips — leases, messaging, stash-carrier as separate passes.** Satisfied structurally: leases and messages are separate tables and separate tools with no cross-dependency (C13, §3 P-MinBlastRadius row) — each is independently flippable. The stash-carrier pass is a consumer-pattern change over messaging (intake Non-goal 7): it rides the already-flipped message primitives and adds no coordination surface of its own.
2. **Drain-then-flip; in-flight coordination state is drained, never migrated.** Satisfied by omission, deliberately: **this cluster designs no import/migration tooling for coordination state.** The file inbox is processed to empty before the messaging flip; file claims are released or expire before the lease flip. The cluster starts empty — which is also its design intent (zero-corpus, SC6). Absence of a migration path here is a decision, not a gap.
3. **No dual-authority window (intake Hard constraint 5).** Exactly one claim system is authoritative at any instant: authority transfers at the flip statement, and the drain precondition guarantees no in-flight item straddles the boundary. Between statement and removal, the old substrate is frozen (intake Non-goal 2), not co-authoritative.
4. **The ritual — statement → migrate → backup → remove — instantiates for coordination as statement → drain → backup (archive snapshot of the file substrate) → remove** (files and their conventions retired). The migrate step is vacuous by rule 2.
5. **Scheduled, operator-only-live flips.** An operational precondition for the cutover statements (only the operator instance live at the statement); recorded here as an obligation on the flip plans, which belong to the program frame and the statements themselves — not designed further in this cluster.

SC8's dry-run observable is the spec-tier check; the design property that makes it satisfiable is (1)+(2): independently flippable surfaces with drain-not-migrate semantics leave nothing that could straddle a flip.

## 12. Intent self-report

**(a) I read the JTBD as:** make instance coordination a designed, substrate-enforced capability of the context layer — durable identity, atomic mutual exclusion, and addressed messaging whose delivery and consumption state are facts in a store rather than conventions in files — shipped as a standalone cluster over zero content, so that the fleet's first real use of the system is the coordination plane itself, and "the workspace starts using mnemra" has a concrete, observable meaning: the file conventions stop being the medium. The deeper intent I read underneath: the wedge is the canon's guarantee-by-mechanism discipline applied to the workspace's own coordination substrate, and its evidence section is four instances of exactly the convention-decay that principle names.

**(b) Decisions that strain an enumerated Non-goal or Success criterion, named:**

1. **Registration below the tool surface vs the intake's primitive walk (SC1 / Evidence use-case).** The intake's end-to-end walk lists `register` among the wedge primitives. This Frame keeps register a primitive but realizes it at the session plane (resolve-or-attach at bind, canonically via the session-start poll) rather than as an agent-visible verb — forced by the ratified two-tool surface and justified by the P-0019 host-derived-context precedent (a caller-supplied principal is an impersonation vector). The strain: SC1's observable says "register → session dies → successor resolves the same actor identity"; the register *event* remains individually observable (attach events are audited, C10), so the observable survives, but a reader expecting `register` as a callable tool will not find one.
2. **The generic `handoff` founding type vs Non-goal 7 (stash-successor as consumer pattern).** A second founding message type edges toward designing the carrier surface the intake explicitly excludes. It is kept deliberately minimal (subject + body + artifact refs, nothing more) so that unstructured handoffs have a schema-valid home without the cluster designing the stash-successor; the stash-successor's actual shape rides its own later cutover pass. If the gate reads even this as scope creep, the `handoff` type can drop to Stage-3 without touching any other direction.
3. **Attachment-as-lease as preferred realization (SC1, Simplicity).** Realizing session attachment under the lease mechanism is a unification the ratifications did not ask for. It is recorded as a preferred realization with the invariants locked and the realization explicitly left variable (C3), so it cannot silently harden into a constraint the spec must obey.

## 14. Routine decisions (batched)

1. `poll` doubles as the canonical attach moment (C3/C7) — one session-start call, parity with live semantics; an explicit separate attach step would add a verb for no observable gain.
2. Refusals are logged in the op log with machine-readable reason codes (P-TrustworthySignal); poll operations are logged (cheap, and absence-of-poll is itself diagnostic for a dead instance).
3. Lease `list` and message `list` are read actions on the existing two tools, not separate query verbs — visibility rides the ratified read default.
4. Message `ack` and `disposition` are distinct lifecycle steps (receipt vs terminal outcome), both addressee-only; collapsing them would erase the delivered-but-not-yet-resolved state today's inbox demonstrably needs (the stale-addressed-item evidence).
5. Acquire declares its requested duration, bounded by a policy maximum (spec calibrates both); takeover evidence (expiry timestamp vs takeover timestamp) is recorded in the audit row.
6. The coordination cluster registers no HTTP surface; health reporting rides the existing host `/health` (no new endpoint).
7. The register entry for this cluster lands at Frame-merge time per the register-promotion convention (intake open-item 4); the promotion rides the docs PR, orchestrator-side.
8. Schema-validation failure at `send` returns the same structured-refusal shape as lease contention — one refusal grammar across the surface (MB3 vocabulary consistency).
