## Stage 2a — Elicitation (input record, decomposer-ratified 2026-07-03)

The three load-bearing architectural directions, ratified before synthesis. They are DECIDED; this Frame elaborates them, it does not reopen them.

- **2a-1 Placement: host-core service, not a plugin.** Rationale ratified with the decision: report SQL spans plugin table families; plugins are leaves (P-0019) that cannot read sibling tables; host-core makes cross-plugin reads host-mediated by construction — same shape as retrieval's host-served MCP verbs (P-0014). This resolves the intake's coupled decision 6(a).
- **2a-2 Definition store: report definitions are content artifacts written through the EXISTING host-fn write machinery.** No new writer, no new schema; definitions inherit workspace scoping, P-0015 `visibility`, audit emission, and MCP authorability. Pulls in P-0017 cluster classification (the definition artifact's data shape is classified in this Frame) and a definition-validation step at write time. Resolves the intake's SC1↔Non-goal-2 tension and Frame decision 5.
- **2a-3 User-SQL guard: dedicated read-only Postgres execution context + RLS on the report path.** A dedicated read-only role AND read-only transaction (two independent write-blocks — the worked reference design's R85 defense-in-depth pair re-derived for Postgres), PLUS row-level-security policies activated specifically for the report execution role (a **scoped exception** to the general V0 RLS deferral — the workspace predicate becomes database-enforced on exactly the path where caller-authored SQL means the host cannot thread it), PLUS `statement_timeout` and a row cap for query-cost containment.

The intake-exit gate additionally deferred two coupled decisions to this Frame, to be resolved jointly (transcribed from the intake's gate record):

- **6(a) Host-core-vs-plugin ⟷ cross-plugin-read enforcement.** "Report SQL over tasks + dispatches + skill-runs spans plugin-namespaced table families. A plugin is a leaf (P-0019) and cannot read sibling plugins' tables; host-core makes cross-plugin reads host-mediated (P-0017). The placement decision and the enforcement shape are one decision." — Resolved jointly by 2a-1; elaborated in decision D1 below.
- **6(b) Phasing ⟷ P-0015-enforcement sequencing.** "The policy-envelope enforcement machinery (serving predicate, egress gate, decision port) lands with the V0.1 retrieval feature. A report read path that must honor P-0015 predicates cannot ship its enforcement before that machinery exists. The ratified post-`1.0.0` placement (item 1) respects this; the Frame carries it as a stated dependency." — Carried as a stated dependency; elaborated in decision D4 below.

And the intake's Frame decision 5 (report-definition persistence + authoring model) is resolved by 2a-2; elaborated in decision D2 below.

## System boundary and component map

The reporting engine is a **host-core service** inside the single mnemra-core process. Nothing about it crosses the plugin boundary: no plugin gains a new host-fn, no plugin sees a sibling's tables, no new WIT surface ships at this feature's tier. Plugins remain leaves (P-0019, recorded Frame Hard-constraint §Correction 1 + D2.5); the engine is a host-side *projection/read* surface over the substrate, which is exactly where P-0002 places cross-plugin aggregation ("a projection concern, not a substrate concern").

Components at architectural altitude, typed against the architecture-overview DFD convention (the same convention the retrieval cluster extended in [P-0014](../src/adrs/P-0014-retrieval-architecture.md)) so the pre-implementation security review gets per-element STRIDE coverage:

| Element | Type | Role |
|---|---|---|
| `P-report-registry` | Process (host-core) | One lookup keyed by report name over two report kinds: **built-in** (compiled host code, reserved names, registered at startup) and **user** (definition artifacts resolved from the definition store). Lookup prefers built-in; a definition can never shadow a built-in (D6). |
| `P-report-executor` | Process (host-core) | The **dedicated read-only execution context** (D3): a dedicated Postgres role, read-only transactions, RLS policies scoped to that role, session-setting threading of the caller's `WorkspaceCtx`, `statement_timeout`, row cap, bounded connection pool. Both report kinds execute here. |
| `P-definition-validator` | Process (host-core) | The write-time definition-validation step (D2): schema validity, name rules + reservation, single-read-statement check, placeholder↔parameter reconciliation. Runs inside the existing content-write path before the row lands. |
| `P-report-renderer` | Process (host-core) | Engine-owned uniform output (D6): one renderer for both kinds; structured (JSON-native) result canonical on MCP; table/CSV renderings for the admin CLI derived from the same structured result. |
| `report.list` / `report.get` / `report.invoke` (names illustrative; spec pins) | Process (verbs on `P-mcp-handler`) | Host-served MCP **read** verbs, the P-0014 RA-4 precedent. Definition **authoring** is a host-served MCP **write** surface riding the existing content-write internals (D2/D5). |
| Admin-CLI `report` subcommand | Process (on `P-cli-handler`) | Convenience frontend over the same registry/executor/renderer — never a second execution path (intake ratification 3: MCP primary, CLI secondary). |
| `DS-report-definitions` | Data store | The definitions' per-type content table under the C1 layout (P-0001/P-0017), workspace-scoped, carrying the P-0015 policy columns. The **one** standard per-type table this feature adds — sanctioned by 2a-2, see the intent self-report. |
| `DF-report-sql` | Data flow | Caller-authored SQL crossing from an authenticated principal into `P-report-executor` — the boundary the discovery-stage security review names TB1. |

**Where built-ins live relative to the registry:** canonical built-in reports are compiled host code that ride their V0 capability-family increments (intake ratification 1 — they exist before this engine does). When the engine lands, the registry registers them under reserved names; only their dispatch path (registry lookup) and rendering (engine-owned renderer) change — never what they compute (intake Non-goal 7; carried from the worked reference design's built-in-preservation requirements R90/R91).

**What the engine writes:** report *execution* writes nothing, ever, of any kind — for both kinds (intake Non-goals 1–2). The only writes anywhere in the feature are (a) definition authoring through the existing content-write machinery and (b) telemetry *emission* through the observability baseline (emitted, not stored in-app — [P-0010](../src/adrs/P-0010-storage-substrate-engine.md) D8/E1 posture).

## Cross-cutting decisions

The three 2a directions elaborated into their mechanism consequences. Every decision carries its canon anchor at the decision line; numeric calibrations (timeout value, row cap, pool size) are deliberately spec-tier.

### D1 — Host-core placement and the cross-plugin read shape (elaborates 2a-1)

**Decision:** The engine is a host-core service. Report SQL — both kinds — executes host-side over the plugin-namespaced table families; no plugin reads a sibling's tables, because no plugin reads anything new: the reads happen above the plugin boundary, host-mediated by construction. The engine is a projection/read surface in exactly the sense P-0002 reserves for cross-plugin aggregation. *Anchors: 2a-1 (ratified); P-0019 recorded Frame Hard-constraint (§Correction 1) + D2.5 (plugins are leaves, host-mediated IO, host-derived `WorkspaceCtx`); P-0002 (cross-plugin aggregation is a projection concern); P-0014 Considered-Option-3 precedent (retrieval rejected plugin-exported verbs on the same grounds); P-0013 (host placement keeps the deferred domain-verb machinery's tripwire unfired — report verbs are host-served, not plugin exports).*

Consequences:

- The discovery-stage security review's coupling finding (its P2-4: placement and cross-plugin-read enforcement are one decision) is **satisfied**: resolving placement to host-core resolves the enforcement shape in the same act. There is no separate cross-plugin enforcement mechanism to design — a leaf plugin still cannot read sibling tables, and the host always could.
- The engine adds **no WIT surface**. `wit/host.wit` is untouched at this tier; a future plugin-authored-report capability would be a new decision, not an implication of this one.
- Built-ins are host code in the host-core reporting module, not plugin exports — consistent with their pre-engine existence as capability-family workspace-fidelity content (intake ratification 1).

### D2 — The definition store: a host-core-owned content type through the existing write machinery (elaborates 2a-2; resolves intake Frame decision 5)

**Decision:** A report definition is a **content artifact**: one row in its own per-artifact-type table under the C1 layout (JSONB frontmatter + body + system fields per [P-0001](../src/adrs/P-0001-storage-layout.md)), owned by **host-core** (like the core entities of [P-0018](../src/adrs/P-0018-core-entity-manifest.md), it lives above the plugin boundary; unlike them it is *not* a hard-FK target and does not join P-0018's closed core-entity set). It is written through the existing content-write internals — validation, `WorkspaceCtx` threading, workspace scoping, P-0015 policy columns with their DDL defaults, audit emission — and is MCP-authorable through host-served write verbs riding that path. No new writer class, no bespoke store. *Anchors: 2a-2 (ratified); P-0001 C1 + P-0017 D-CM/D-SoT; P-0015 PE-2/PE-6; P-LockContract (the definition schema is the locked contract; storage mechanics vary behind the `Storage` seam).*

Mechanism consequences:

- **Cluster classification (P-0017): `reference`.** A definition is a read-mostly, long-lived, named lookup — authored rarely, invoked repeatedly, resolved by exact name — the same shape as the taxonomy's `templates` exemplar. It is not `state-bearing` (it carries no mutating lifecycle state at V0) and not `narrative` (frontmatter dominates; the body is one SQL statement). Classification is proposed here for confirmation at the Frame-exit gate, per P-0017's maintainer-call rule for non-self-evident shapes. Audit on authoring writes comes from the write path itself (admin-gated, attributable — the PE-6 pattern), not from the shape's per-shape policy.
- **Definition shape (Frame altitude; spec pins the schema):** frontmatter carries `name` (pattern `^[a-z0-9][a-z0-9-]*$`, carried from the worked reference design's R86), `description`, typed parameter declarations (closed V0 type set `{string, date, int}` with `required`/`default` semantics, carried from R93; extension is a spec amendment), and a default output format; the body is **exactly one read statement**. Uniqueness is `(workspace_id, name)` among definitions, plus the reserved-name exclusion (D6).
- **Write-time validation (the 2a-2 validation step):** the write path runs, before the row lands — (1) schema validity at declared types; (2) name rules + built-in-name reservation; (3) the single-read-statement check (the statement parses, is exactly one statement, and is a read — `SELECT`/`WITH…SELECT`, with every CTE itself read-only: a data-modifying CTE such as `WITH t AS (DELETE … RETURNING *) SELECT …` is `WITH…SELECT`-shaped yet writes, and is rejected at write time; the D3 role and read-only transaction already reject it at execution, so this bar is validation-completeness, not the security boundary — an invalid definition never registers, rather than registering and failing at every invocation); (4) placeholder↔parameter reconciliation (a `:param`-style placeholder with no declaration is a hard error; a declared-but-unused parameter is a warning, carried from R94); (5) parameter `default` values pass their own type validation. An invalid definition never registers — the validity-vs-not-a-manifest discrimination the worked reference design drew (R87) collapses here to plain schema validation, because the store contains only rows written through this gate; the "unrelated file in a shared directory" case has no Postgres analog and is **dropped** with the filesystem discovery model.
- **Definitions inherit the full P-0015 policy record** with unchanged semantics: `visibility` governs who can list/read/invoke a definition (its `sql` text can embed sensitive filter literals — the intake's stated motivation); `dont_use` is the curatorial kill switch (a killed definition resolves per PE-4's stub disposition — it announces "exists; do not use"); `visibility: owner-only` follows PE-4's V0 semantics (serves no one until identity lands); `tenant_share` is the structural constant. Policy-field writes ride the PE-6 admin-gated path with the attributable audit row.
- **Registry resolution reads the store as source of truth** (P-0017 D-SoT): the content row is authoritative; any in-memory registry cache is derived and rebuildable, never written as authoritative. Runtime extensibility (intake SC1) is therefore write-then-invoke with no restart: a definition that lands is resolvable on the next lookup.
- **Live schema is the contract at V0** (carried from R98): a definition referencing a table or column that does not exist — or that a later migration renamed — fails at statement preparation with a structured error naming the unknown identifier. No curated `report_*` view layer ships at V0; the deferral is **self-announcing** (DF1 outcome b): a broken definition surfaces its actual reason to its next invoker, and prepare-failure telemetry (OC-9) gives the operator the aggregate signal.

### D3 — The dedicated read-only execution context and the scoped RLS exception (elaborates 2a-3)

**Decision:** All report SQL — user definitions AND built-ins — executes in a dedicated execution context: a dedicated Postgres role, read-only transactions, row-level-security policies scoped to that role, per-execution session settings carrying the **caller's** identity, `statement_timeout`, a host-enforced row cap, and a bounded, separate connection pool. Running built-ins in the same context is deliberate: intake Non-goal 1 makes read-only the engine's identity "of any kind," so the execution posture is uniform — built-ins simply carry a *third* layer (their host-built SQL keeps the P-0006 host-threaded workspace predicate). *Anchors: 2a-3 (ratified); worked reference design (2026-05-31) R85 defense-in-depth pair, re-derived; P-SecurityLayered (each layer independently load-bearing); the `Security ⇄ Simplicity` conflict edge, resolved default-to-Security; P-0009's V0.1 RLS policy shape (session-setting keyed policies), adopted early on this one path.*

Mechanism consequences:

- **Two independent write-blocks, each alone sufficient (the R85 pair, re-derived for Postgres):** (1) the report role holds `SELECT`-only privileges — no `INSERT`/`UPDATE`/`DELETE`, no DDL, no sequence usage; (2) every report transaction is `READ ONLY` (`SET TRANSACTION READ ONLY` / `default_transaction_read_only`). Either alone blocks a write; both apply together. The spec's acceptance criteria prove each independently (the worked reference design's "belt" acceptance shape carries).
- **The grant set is an allow-list — the Postgres re-derivation of the worked reference design's authorizer allow-set.** The report role's `SELECT` grants enumerate the readable surface: the plugin content-table families (tasks, dispatches, skill-runs, activity, artifacts, and their kin) plus the product-measurement tables, all workspace-scoped. **Excluded by default-deny** (no grant = unreadable): the identity/auth/session/permission builtin tables (`admin_tokens` above all), and the P-0015 policy-audit stores (`policy_write_audit`, `policy_decision_audit`) — those are operator-side signals whose exposure through caller SQL would itself be an existence-disclosure channel. Extending the readable set is an amendment to the `{{P-ReportExecutionContext}}` ADR, not a config knob. The allow-list extends to the **function-EXECUTE surface**: Postgres grants `EXECUTE` to `PUBLIC` by default, so a table-only allow-list is default-*allow* on the function axis — `set_config` (the session-key threat, below), the `pg_stat_*`/server-info family, any in-schema `SECURITY DEFINER` function. The report role's callable-function set is therefore an enumerated allow-list too, locked alongside the table set under the same ADR and extended only by amendment; how the `PUBLIC`-default revocation is mechanized is spec-tier under that slot. Per OC-10, the grant set — tables and functions — and its RLS coverage ship with a reconciliation check (computed reality == declared sets; every granted table has RLS enabled with the workspace policy; no function outside the enumerated set is executable by the report role) that demonstrably fails on a seeded violation.
- **The scoped RLS exception.** RLS is **enabled on the tables in the grant set, with policies scoped `TO` the report role**, keyed on session settings the host sets per execution from the caller's validated `WorkspaceCtx` (`SET LOCAL` of the workspace id and role — the exact shape P-0009's V0.1 hardening sketch already records, adopted here into an adversarial caller-SQL context that sketch never faced: which is why the session-key integrity property below locks alongside it, and why the key's *source* may move under the ADR's mechanism pick). The host's primary role and every host-built path are untouched: the general V0 RLS deferral (P-0006/P-0009 accepted risk R-0001) **stands** — application-layer `WorkspaceCtx` discipline remains the enforcement on every host-constructed query, and the R-0001 trip-wire (first multi-workspace production deployment) is unchanged. The exception is scoped to exactly the path where the deferral's premise fails: the deferral assumed host-built WHERE clauses; a caller-written SELECT has none. When R-0001 later fires and general RLS activates, the report-path policies converge with — do not fork from — the general activation: candidate (a) of the session-key mechanism preserves the V0.1 keying verbatim; if the ADR picks candidate (b) it trades keying identity for a stronger key-isolation boundary and must state its convergence path in the same decision. One mechanic is **not** spec-tier detail but a **correctness precondition** of the deferral-stands claim itself: `ALTER TABLE … ENABLE ROW LEVEL SECURITY` — required to attach the report-role policy — activates RLS on that table for **every** role, so the host's primary role stays untouched only if it provably bypasses the newly-enabled RLS (table ownership without `FORCE ROW LEVEL SECURITY`, or `BYPASSRLS`). The ownership/`BYPASSRLS`/`FORCE` model is therefore locked scope under `{{P-ReportExecutionContext}}`, and the OC-10 reconciliation asserts it per granted table: RLS enabled AND the host path provably bypasses it AND the report role provably does not.
- **Session-key integrity — a locked property of the execution context:** the settings are written once per execution by the host from the validated token context, inside the same transaction (`SET LOCAL`), before the user statement is prepared. That write discipline alone is not tamper-resistance: Postgres does not treat GUC changes as writes, so a `READ ONLY` transaction does **not** block `set_config()`, and `EXECUTE` on `set_config` is a default `PUBLIC` grant — a caller's single prepared SELECT can lexically carry a re-key attempt (a `set_config()` call in an expression, or inside a `MATERIALIZED` CTE), and whether it lands at runtime turns on evaluation-order internals a tenant-isolation guarantee must not rest on. The Frame therefore locks the **property**: the RLS session keys MUST be unreachable from caller-authored SQL, proven by threat test, never assumed from transaction mode. The **mechanism** is `{{P-ReportExecutionContext}}`'s decision, from a fixed candidate space: (a) revoke the GUC-setter surface from the report role — `set_config(text,text,boolean)` and its `pg_catalog` kin — which the function-EXECUTE allow-list (grant-set bullet above) gives structural footing by construction, provided no permitted function can set a GUC directly or indirectly; or (b) key the report-path RLS policies on a value caller SQL cannot reach (e.g. per-execution `SET ROLE` to a workspace-bound role, so policies key on `current_user` rather than a settable GUC). The ADR picks and argues the property end-to-end; the property and the candidates are fixed here. The spec's threat tests MUST seed a `set_config`/materialized-CTE re-key attempt that demonstrably fails — this feeds SC2's negative-acceptance suite.
- **Single-statement, bind-never-interpolate execution:** the user statement executes via the extended query protocol as **one prepared statement** with typed parameter binds — single-statement by construction (the multi-statement injection case is structurally absent, not filtered), and parameter values are data, never SQL text (R95 carried verbatim). The write-time validation (D2) already rejected non-read statements; the role + transaction layers back-stop anything validation misses (defense in depth, not a single gate).
- **Query-cost containment:** `SET LOCAL statement_timeout` per execution (an availability primitive the plugin resource limits of [P-0007](../src/adrs/P-0007-plugin-resource-limits.md) do not provide — those bound plugin CPU, not Postgres query cost); a host-enforced row cap with **reported truncation** — the result envelope carries a truncation flag and the capped count, never a silent trim (P-TrustworthySignal); a bounded, separate connection pool so runaway report load cannot starve the host's primary pool. Numeric values (timeout, cap, pool size) are spec-tier. A cost-*estimate* ceiling (planner-cost rejection before execution) is **deferred**: decision content — whether to add an `EXPLAIN`-based pre-execution cost gate and its threshold; deferral anchor — P-Defer (timeout + cap bound the damage; a cost model is mechanism ahead of evidence) with the `P-LockContract ⇄ P-PreserveDecisionSpace` discriminator (the read-only identity is intrinsic and locks now; the cost-model refinement is separable); firing instrument — the OC-9 telemetry reporting timeout-kill events above a spec-named rate, i.e. the timeout layer measurably absorbing sustained abuse.
- **Built-ins in this context:** built-in SQL is host-built (P-0006 `WorkspaceCtx` predicate threaded as on any host path) *and* executes under the report role/read-only transaction/RLS. Three independent layers on the trusted kind costs nothing and keeps one execution posture; percentile computations use Postgres-native functions (D6).

### D4 — Policy-envelope enforcement on a caller-written query (elaborates 2a-3 × OC-5; carries gate item 6(b))

**Decision:** The P-0015 policy record binds on the report path as **row-exclusion predicates rendered into the report role's RLS policies**, generated from the same single fragment module that renders the retrieval serving predicate (PE-3's "exactly one module" rule extends to this second consumer), with the caller's role threaded via the D3 session settings so `visibility: admin-only` resolves per caller. The differential conformance test (PE-3's locked pick) **extends to the report role's rendering**: fragment and `decide()` must agree over the generated dimension × pole × undecidable resource set, on this path as on retrieval's, failing demonstrably on a seeded divergence. *Anchors: P-0015 PE-1..PE-4 (the policy side is a security layer, structural, fail-closed, single-boundary); P-MinBlastRadius (one fragment module, N consumers); intake SC2.*

Mechanism consequences:

- **Which dimensions bind, and how:** `dont_use` — excluded rows (the curatorial kill holds on report reads; a killed artifact's rows never enter a result). `visibility` — `workspace` rows serve both roles; `admin-only` rows serve callers whose session role is `Admin`; `owner-only` rows serve no one at V0 (PE-4 semantics unchanged). `tenant_share` — subsumed by the workspace predicate at V0 (structural constant). `model_egress` — **not implicated on this path**: the engine makes no model calls (report execution has no generative placement), and results returning to the invoking caller are *serving*, which `visibility` governs. If a later feature ever feeds report output to a model, that feature's egress gate consults the same port — the dimension's enforcement point does not move here.
- **Withholding is caller-silent with the operator-side audit** (PE-4): an excluded row is simply absent from the result — indistinguishable from non-matching data — with the policy-decision audit row emitted operator-side. A report result is therefore computed over the caller's entitled row set *before* aggregation: a count, sum, or percentile never includes a withheld row (the same before-fusion placement discipline the retrieval cluster locked for rank/budget accounting).
- **The trust axis is deliberately absent from report rows.** P-0015's two-axis contract separates policy (an enforcement input) from trust (a served provenance label on *context items*). Report rows are query results — aggregates and projections — not served context items; they carry no `{trust, freshness, citation}` envelope at V0. This is canon-consistent (the policy side is the authorization control and it binds structurally; the trust side is informational) but it has a consequence worth naming: a user report that SELECTs raw artifact content receives it **policy-clean but provenance-unlabeled**. The retrieval verbs remain the labeled context-serving surface; the report surface is the measurement/aggregation surface. Flagged in the intent self-report as an interpretation the gate should see. Decision content if it ever needs revisiting — whether content-body columns should be excluded from the report-readable column set or results above some content-payload threshold should carry envelope metadata; deferral anchor — P-Defer + Simplicity (per-row envelopes on arbitrary aggregates are mechanism without a consumer); firing instrument — self-announcing: the first consumer that treats report output as trust-bearing context (an agent-workflow ask, arriving as a feature request against this surface).
- **The stated dependency (gate item 6(b), carried):** the fragment module, decision port, conformance suite, and `policy_decision_audit` store land with the `1.1.0` retrieval feature per the retrieval-cluster spec ([`docs/specs/2026-07-02-retrieval-cluster.md`](../specs/2026-07-02-retrieval-cluster.md) R-0025/R-0026). This Frame designs against that machinery as a build-time dependency; the engine's `1.3.0`+ placement sequences after it by ratification. If the retrieval cluster's landing slips or reshapes those contracts, this Frame's D4 is re-derived against the new world before spec lock (the lock is scoped to its assumptions). The re-derive clause fires by machinery, not human recall, on both sides: **(spec-authoring time)** the reporting spec's BOM sidecar blob-pins the consumed contract surface — the retrieval-cluster spec's R-0025/R-0026 and the single-fragment-module contract R-0025-g locks — so a reshape fails the BOM hash check and mechanically fires the re-derivation; **(runtime side)** the extended differential conformance run over the report-role rendering (this decision's own proof obligation) fails demonstrably on a fragment↔`decide()` divergence.

### D5 — Role matrix, caller scoping, and enumeration discipline (elaborates 2a-3 × OC-4)

**Decision:** Invocation and listing are **read verbs**, available to `admin` and `read_observer` alike; definition authoring (create/update/delete, and any policy-field flip on a definition) is a **write verb**, `admin`-gated with `read_observer` denied pre-dispatch. A report's results scope to the **caller's** workspace and role — never the author's: the D3 session settings are constructed from the invoking principal's validated token context, so a saved report is a saved *read*, re-authorized on every invocation. *Anchors: P-0009 permission matrix (read/write verb categories); P-0015 PE-6 (admin-gated mutation as the write-authorization pattern); the discovery-stage security review (2026-07-03) TB3.*

Mechanism consequences:

- **No ambient authority in a definition:** a definition stores *what to compute*, never *as whom*. There is no author-privilege capture; an `admin`-authored report invoked by a `read_observer` returns the observer's entitled rows only (RLS + policy predicates keyed to the caller's session settings).
- **Enumeration / no-existence-oracle discipline:** `report.list` returns built-ins plus the definitions the caller is serve-entitled to under `visibility`; a withheld definition is absent, not marked. Invoking or fetching a visibility-withheld definition by name returns not-found, indistinguishable from a genuinely absent name (PE-4's visibility disposition); invoking a `dont_use`-killed definition returns the metadata-only stub with the policy reason (PE-4's dont_use disposition — curatorial policy announces itself). Authoring errors disclose nothing across principals: reserved-name collisions reference the public built-in set, and same-name conflicts within a workspace are visible only to `admin` callers, who are entitled to see every workspace definition anyway (`admin-only` is the strictest operative visibility at V0 given owner-only serves no one).
- **Audit:** every authoring write and every policy-field flip lands an attributable audit row (actor token identity NOT NULL — the PE-6 record shape); invocation is telemetry (OC-9), not content audit.

### D6 — Registry, verb surface, and uniform output (completes the component contracts)

**Decision:** One registry, two kinds, one renderer. The verb surface is **tool-shaped** (MCP tools/verbs, not MCP resources): report invocation is a parameterized computation with typed arguments — the tool shape — not an addressable static document; this also keeps the surface uniform with the existing verb-category role gating (OC-4) and the host-served-verb precedent (RA-4). Resolves the intake's tool-vs-resource Frame decision (ratification 3). *Anchors: P-0009 verb categories; P-0014 RA-4; P-AgentPrimarySource (typed, tool-checkable call contracts); Simplicity.*

Mechanism consequences, carrying the worked reference design's durable decisions adapted to substrate:

- **Registry:** one lookup keyed by name; built-in kind and user kind; built-in wins any lookup; the reserved built-in name set is enforced at definition **write time** (a create/update naming a reserved name fails with a structured error naming the collision) — strictly stronger than the worked reference design's discovery-time collision handling, because the store's write gate can reject atomically (R83's intent carried; its directory-precedence machinery **dropped** — no filesystem discovery exists to need it, and `(workspace_id, name)` uniqueness replaces cross-directory shadowing).
- **Output shapes and uniform rendering (SC3):** each report declares one of three output shapes — `scalar`, `matrix`, `row-list`; user reports are always `row-list` (columns = the statement's result columns); built-ins declare theirs. The canonical result on MCP is a structured, shape-tagged JSON object (columns, rows, row count, truncation flag, parameter echo); the admin CLI renders `table`/`csv` views from that same structured result through the one engine-owned renderer. A built-in and a user report of the same shape render identically for the same format (R88 carried, formats re-homed: JSON-native canonical on MCP, table/csv as CLI renderings).
- **Empty-result-is-success and NULL-renders (SC3):** zero rows is a success with the format's empty rendering (empty `rows: []`, header-only table), never an error; a NULL cell renders explicitly (JSON `null`, empty CSV field, a consistent table token) and a NULL-bearing row is never dropped (R89 carried verbatim).
- **Stats functions:** Postgres-native `percentile_cont` (linear interpolation — the same type-7 definition the worked reference design locked as engine-wide) and `percentile_disc` serve user SQL directly; the custom `median()`/`percentile()` registrations are **dropped** (substrate provides them; `median(x)` ≡ `percentile_cont(0.5)`). The one-definition parity rule carries: built-in percentile computations use the same linear-interpolation definition (by calling the same SQL functions or an implementation proven equal on fixed data), so a built-in p95 and a user `percentile_cont(0.95)` agree on identical input.
- **Parameters:** typed declarations (closed `{string, date, int}` V0 set), required/default/NULL-binding semantics, ISO-8601 date validation, all carried from R93 with Postgres-typed binds; richer types are a spec amendment.

### D7 — Instrumentation (OC-9 rendered as a deliverable)

This emission is the named firing instrument for intake Non-goal 3's materialization/caching tripwire (a per-report latency budget the spec sets, crossed in the telemetry, fires the revisit) and the evidence source for the D3 cost-ceiling deferral. *Anchors: P-InstrumentBefore (+ the P-Defer ⇄ P-InstrumentBefore edge, resolved to instrument — this is a production surface); P-0017 D-CP (the query-instrumentation surface); P-0010 D8/E1 (telemetry emitted, not stored).*
