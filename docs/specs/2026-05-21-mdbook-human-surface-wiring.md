# Spec: Wire mdBook to the human-translated tree + strip YAML frontmatter at build

## Goal

One bug — the published docs site does not display properly — with two root causes, fixed together:

1. **Unwired human surface.** `docs/book.toml` sets `src = "src"`, so mdBook builds the
   canonical agent-first `docs/src/` tree, not the EXPLAIN-translated `docs/_published/human/`
   tree. The human translations generated by the step-6 pipeline are not served. G-0029 §2
   specifies the human surface is built from `_published/human/`.
2. **Frontmatter rendered as page text.** Pages carry YAML frontmatter (`title` / `summary` /
   `primary-audience`). mdBook has no native frontmatter handling and no stripping
   preprocessor is configured, so the `---`-fenced block renders as visible body text (the
   trailing `---` turns the lines into a setext heading — the spurious "title" the maintainer
   observed). Affects all pages.

The agent surface is already correct and **out of scope**: `docs-llms.py` defaults
`--src docs/_published/agent`, so `llms.txt` / `llms-full.txt` build from the STRIP tree.

## Non-goals

- No change to `docs-llms.py`, the agent tree, or `llms.txt` / `llms-full.txt`.
- No change to the translation pipeline (`docs-translate.py`, `/docs-translate`).
- No content edits to `docs/src/` or `docs/_published/`.

## Decision: strip via an mdBook preprocessor

Frontmatter is removed at mdBook build time by a preprocessor, not at publish time. Rationale:
`_published/human/` pages are produced by two writers (the script's verbatim-copy path and the
slash-command's EXPLAIN Write path), and LLM output frontmatter is variable; a render-layer
preprocessor strips uniformly regardless of source and keeps the `_published/` trees uniform.

## Acceptance criteria

1. **Preprocessor exists** at `scripts/mdbook-strip-frontmatter.py`, Python standard-library
   only (no third-party deps), invoked via `uv run`.
2. **`supports` protocol.** Invoked as `<cmd> supports <renderer>` it exits 0 for any renderer
   (supports all).
3. **Strip behavior.** Invoked with no args, it reads the `[context, book]` JSON array from
   stdin and writes the modified `book` object as JSON to stdout. For every chapter — including
   nested `Chapter.sub_items`, recursively — it removes a **leading** YAML frontmatter block
   (content beginning with `---\n`, up to and including the next `---\n` line) from
   `Chapter.content`. Content without a leading frontmatter block is left byte-unchanged.
   Frontmatter `---` fences are stripped only at the start of content, never mid-document
   (a horizontal rule lower in the page is untouched).
4. **book.toml repointed + preprocessor registered.** `docs/book.toml`: `src = "_published/human"`;
   add `[preprocessor.strip-frontmatter]` with `command = "uv run scripts/mdbook-strip-frontmatter.py"`.
   The existing `[preprocessor.mermaid]` and `[preprocessor.d2]` entries are preserved.
5. **CI provisions uv.** `.github/workflows/docs.yml` installs `uv` (pinned `astral-sh/setup-uv`)
   before the `mdbook build docs/` step, so the preprocessor's `uv run` resolves in CI.
6. **End-to-end — wiring.** A clean `mdbook build docs/` succeeds. The rendered HTML of an
   agent-primary page that the EXPLAIN pass rewrote (e.g. `docs/book/intent/mnemra-core.html`)
   contains a phrase present in `docs/_published/human/intent/mnemra-core.md` but **absent**
   from `docs/src/intent/mnemra-core.md`, and does **not** contain a phrase unique to the
   agent-source page. (The red-phase author selects the concrete literals from the committed
   trees.) This proves the build reads `_published/human/`, not `docs/src/`.
7. **End-to-end — strip.** After a clean build, no rendered page under `docs/book/**/*.html` displays YAML frontmatter as body text — specifically, the keys `primary-audience` and `summary:` must not appear **outside** an inline-code (`<code>`) or code-block (`<pre>`) element. A frontmatter key quoted inside backticks/code fences is legitimate documentation prose, not a leak. This proves the frontmatter block no longer renders.
8. **No build artifacts committed.** Tests build to a temporary output dir (e.g.
   `mdbook build docs/ -d <tmp>`) or clean up `docs/book/`; the test run leaves no new tracked
   or untracked files in the repo.

## Test plan (red-phase first)

Black-box tests in `tests/` (project convention; pytest, `uv run`):

- **Preprocessor protocol (unit, via subprocess on stdin/stdout):**
  - `supports html` → exit 0.
  - A book JSON with a chapter whose `content` has a leading frontmatter block → output strips it.
  - A chapter with no frontmatter → `content` byte-unchanged.
  - A nested `sub_items` chapter with frontmatter → stripped (recursion).
  - A chapter whose body contains a mid-document `---` thematic break → that break is preserved.
- **End-to-end (build the real book):** clean `mdbook build docs/` to a temp dir; assert AC #6
  (wiring literals) and AC #7 (no `primary-audience` in any rendered HTML).

mdBook 0.5.2 + `mdbook-mermaid` + `mdbook-d2` are available locally and pinned in CI; the
end-to-end test requires them on PATH.

## Out of scope / follow-ups

- The `docs-llms.py` module docstring still says "from docs/src/" though the argparse default is
  `docs/_published/agent`; a cosmetic doc fix, tracked separately if desired.
