## Purpose

Tier-1 (865f4dc) bought CI determinism by serializing the embedded-Postgres / plugin test group and left the suite structurally slow: each member binary boots one embedded engine **per test** (~100+ serial boots per full group pass; full CI 12m24s–12m43s at HEAD), carries vestigial per-binary `STARTUP_LOCK` machinery, and runs under a lint gate blind to every test target (28 warnings invisible to CI). This spec collapses engine boots to **one shared engine per member binary** with a **fresh, uniquely-named DATABASE per test** — keeping the determinism guarantee (SIGABRT = 0, re-proven by measurement, never asserted) and the R-0006 isolation semantics exactly as they are — and retires the vestigial locks and the `--all-targets` blind spot. Speed AND trust, proven before/after; the measurement is the deliverable (P-TrustworthySignal).

As in tier-1, this spec states **constraints, not mechanisms**: the fixture's internal boot/teardown machinery, the lint's implementation shape, and check placement are implement-time HOW, bounded by the requirements below (tier-1 §Purpose precedent; P-LockContract — one locked seam, varying implementation).

RFC-2119 keywords are used throughout. `SHALL`/`MUST` = mandatory; `SHALL NOT`/`MUST NOT` = prohibited; `SHOULD` = preferred; `MAY` = optional.

### R-0026 — Shared-engine fixture: one boot per binary through a single entry point

**Decision.** All test-side embedded-engine acquisition SHALL go through a **single shared fixture entry point** in a new module under `libs/mnemra-host/tests/common/` (module name is implement-time). The fixture SHALL boot the binary's engine **lazily on first use** (get-or-init: the first caller boots, every later caller reuses), so each fixture-consuming member binary boots **exactly one** embedded engine per run. The fixture's once-semantics SHALL live in the fixture itself and SHALL NOT depend on the recipe-level `--test-threads 1` flag for correctness (code-traveling: any invocation path — the three `verify-*` recipes and any future fourth — inherits the collapse). *Anchors: Frame F-1 (R-0018-b — share the real engine, never mock or externalize; P-LockContract — one locked seam; Simplicity — smallest mechanism that collapses boots; R-0022 pattern — code-traveling uniformity).*

**Acceptance Criteria:**
- [ ] Per fixture-consuming member binary, a fresh run observes **exactly one** embedded-engine boot. Evidence method, pinned: one scoped `scripts/flake-runner.sh` invocation per member (single member, N = 1, `--test-threads 1` per R-0039; `artifact_list_paging_whitebox` exercised non-vacuously per R-0031/R-0039), with the boot count read by a **counting** observable, recorded with the evidence. The default evidence path is **one-per-boot engine startup events**: the fixture emits a boot-marker line and the harness already captures full per-run logs (`scratch/flake-runs/<bin>_run<i>_t<threads>.log`) — counting the marker needs zero script changes. Counting distinct fixture-spawned postmaster PIDs is acceptable only as a declared alternative requiring a **sampler extension**: the harness's sampler tracks a scalar peak, never a PID set, so this path adds a script-diff (accumulate a PID union) recorded like R-0039's. The harness's aggregate `PEAK_DELTA` (which, on a run scoped to a single member, is that member's peak-above-baseline) additionally reads ≤ 1 as a crosscheck; peak concurrency alone is NOT sufficient evidence — strictly serial per-test boots also peak at 1, and the instrument emits exactly one aggregate `PEAK`/`PEAK_DELTA` per invocation, not a per-binary breakdown.
- [ ] Group total in-process boots per full pass collapse from ~100+ (≈ per-test at HEAD) to **≤ 18** (16 fixture members × 1, plus `startup_run_full`'s production-path boots, 2 at HEAD — see R-0030), recorded before/after per R-0036.
- [ ] The fixture module's engine init is guarded in the fixture itself (once-style get-or-init), not by an assumption on the recipe flag — reviewable in the fixture source AND backstopped mechanically: a test performs two concurrent **in-process** acquisitions (e.g. joined futures within one test body — intra-process concurrency exists even under `--test-threads 1`) and asserts both observe the same engine with exactly one boot.
- [ ] The fixture is the only test-side construction path — enforced mechanically by R-0032's guard lint (cross-reference; the lint is the enforcement, this criterion is the property).

### R-0027 — Per-test database, application-role pool, and the privileged provisioning path

**Decision.** From the shared engine, each test SHALL obtain: (1) a **fresh, uniquely-named DATABASE** created for that test on the shared engine, schema-initialized including per-database `CREATE EXTENSION vector`; and (2) an **application-role pool bound to that database**. Database provisioning SHALL go through a **named privileged method on `EmbeddedEngine`** (A-14 discipline: `superuser_pool` is deliberately `pub(crate)` with privileged ops exposed as named methods, `libs/mnemra-host/storage/postgres/engine.rs:380-384`); the fixture SHALL NOT re-widen the pool field and SHALL NOT grant the application role new privileges. **Change surface, declared (review r1):** no per-test-database provisioning method exists at HEAD — the privileged named-method surface is `ensure_pgvector` / `ensure_extension` / `create_least_privilege_roles` (`engine.rs:584-612`) — so the implementation ADDS one: `libs/mnemra-host/storage/postgres/engine.rs` gains a `#[doc(hidden)] pub` named method (widened from `pub(crate)` by the 2026-07-02 post-lock amendment — see Review history: the fixture is `#[path]`-included into integration-test crates that link `mnemra-host` externally, across which `pub(crate)` is unreachable and does not compile; `#[doc(hidden)]` keeps the method out of the published API surface, preserving the A-14 minimal-surface intent). This is a production-path-file touch, in-pattern (the struct's A-14 doc already anticipates new named methods on this type rather than field re-widening) and P-MinBlastRadius-clean (no P-0007 containment surface); the Frame's "test harness + justfile" change-surface shorthand widens by exactly this one file. There SHALL be **no per-test `DROP DATABASE`**: test databases live for the binary's lifetime and die with the ephemeral engine (`.temporary(true)`, `libs/mnemra-host/storage/postgres/engine.rs:409`); resource exhaustion, if it ever occurs, announces itself in that binary's run (self-announcing tripwire). **Pool lifecycle (SO-1, locked here):** each test's app-role pool is **owned by the test and scope-dropped at test end**; the fixture SHALL NOT cache app-role pools across tests — the only fixture-held process-global state is the shared engine handle and its privileged surface. *Anchors: Frame F-1 sub-decisions (A-14; P-Defer/Simplicity — no DROP without evidence of need); R-0006 (clean DB per test preserved); SO-1 discharge (Simplicity — matches the per-test-engine model's connection lifetime; R-0006 — a per-test pool bound to a per-test database, nothing shared across tests).*

**Acceptance Criteria:**
- [ ] Two tests in the same member binary receive **distinct database names** and cannot observe each other's rows (Scenario S2 green).
- [ ] `CREATE EXTENSION vector` succeeds per test database: pgvector-dependent suites pass on fixture-provisioned databases.
- [ ] The role-shape test `app_role_is_not_superuser_and_not_bypassrls` (`libs/mnemra-host/tests/postgres_engine.rs`) passes with its **assertions unedited** — its acquisition line MAY migrate to the fixture per R-0030 (the same acquisition-only edit discipline R-0037 pins for the isolation members; `postgres_engine.rs` is necessarily edited by R-0029's lock retirement, so "unchanged" here is assertion-level, not byte-level).
- [ ] `superuser_pool` remains `pub(crate)`, and `grep -rn 'superuser_pool' libs/mnemra-host/tests/` returns nothing (no test-side field access; provisioning is via the named method).
- [ ] `grep -rni 'DROP DATABASE' libs/mnemra-host/tests/` returns nothing.
- [ ] The fixture module holds no static/global collection of app-role pools (per-test pool is returned by value). This property is **explicitly judgment-reviewed** in the fixture source — a mechanical "no statics in the fixture" grep is unsound, since the fixture legitimately holds the process-global engine handle. Its mechanical backstop is the full R-0035 measurement completing with zero connection-exhaustion errors (e.g. "too many clients") — connection accumulation would announce there — and the no-DROP posture's resource growth likewise self-announces via the same sweep's existing `DISK`/`SHMEM` classifiers (`scripts/flake-runner.sh` classifies "No space left on device" and shmget failures).

### R-0028 — Deterministic exit-time teardown; static-Drop forbidden

**Decision.** The shared engine SHALL have **explicit, deterministic teardown at binary exit**, firing on every normal process exit — pass or fail, including the all-tests-passed path. Reliance on `Drop` of a `static` is FORBIDDEN: Rust statics never drop, so a static-without-teardown guarantees one leaked postmaster per binary run (~17 per full CI pass — recreating exactly the zombie backlog that masked the abort-frame measurement; the discriminator dispatch found 24 pre-existing zombie postmasters masking 20/20 of its fresh runs via SHMEM exhaustion). The teardown *mechanism* (exit-time hook, guardian teardown, custom harness, …) is implement-time HOW, bounded by: **no non-Green dependency**, and it fires on normal process exit. R-0024's reaping harness remains the **measurement backstop**, not the primary mechanism. **Exit-window ordering (SO-2, stated here):** the teardown introduces a new exit-time operation in the same exit-adjacent window where the #1852 machports handler-thread abort fires. The stated interaction posture: the teardown runs as a **serial** exit-path operation (no concurrent teardown fan-out is introduced — R-0034 keeps `--test-threads 1`, so the concurrent-teardown windows the dose-response evidence tracked do not exist), the wasmtime lifecycle is untouched (Out of scope, F-4), and the residual ordering interaction is **verified, not assumed**, by R-0035's mandatory N ≥ 20 measurement — which fails acceptance outright on any ABORT. *Anchors: Frame F-1 lifecycle sub-decision (P-TrustworthySignal — a leaked-postmaster backlog corrupts the instrument); SO-2 discharge.*

**Acceptance Criteria:**
- [ ] After every member-binary run that exits normally (pass or fail), the live `bin/postgres` postmaster count returns to the pre-run baseline (`BASE_PM` as measured by `scripts/flake-runner.sh`) — demonstrated across the full R-0035 runs (this observable is also the statics-never-drop proof: a static engine would leak exactly one postmaster per run and fail it).
- [ ] Teardown fires on the all-tests-passed path specifically (Scenario S6: a fully-green member run leaves no fixture-spawned postmaster).
- [ ] No new non-Green dependency is introduced for the teardown mechanism (`dependencies_added` in the implementation report; `Cargo.lock` diff).

### R-0032 — Guard lint: no engine construction outside the fixture

**Decision.** A CI-gated check SHALL FAIL if engine-construction symbols (`EmbeddedEngine::start`, `start_embedded`, `postgresql_embedded`) appear **in code** in `libs/mnemra-host/tests/` **outside the fixture module**, with production-bootstrap callers excepted **by name** (today exactly one: `startup_run_full`; `smoke_e2e` contains no textual match at HEAD and needs no exception). **Match semantics — obligations on the check itself (review r2 precision fold):** the check SHALL match the banned symbols only in **executable-code tokens**. Three exclusions bind the implementation directly — stated as requirements, not as properties inherited from a precedent: **(i)** ordinary comment content SHALL NOT match (discharged by matching parsed source, never raw text); **(ii)** doc-comment content SHALL NOT match *in the form it survives parsing* — `//!`/`///` desugar to `#[doc = "..."]` string-literal **attributes**, which comment-stripping does NOT remove — so the check SHALL exclude attribute tokens (`#[doc = ...]` however produced); **(iii)** string/char-literal contents SHALL NOT match — the exclusion that also keeps the check from self-flagging if its own file is in scan scope (its symbol references live in literals; the implementation MAY additionally exclude its own file by name, as the fixture module already is). These obligations are load-bearing, not a nicety: at the refactored HEAD the symbols legitimately persist in prose — `common/mod.rs:14,20,26,32` (rustdoc example text; the file is **byte-locked** by R-0037/SO-3, so the "just delete the match" escape is closed) and `startup_run_ordering.rs:8,11` (doc comments in a NONPG binary on no exception list) — and this requirement's "passes at the refactored HEAD" criterion is jointly satisfiable with R-0037 AC1 **only** if (i)–(ii) hold. The obligations are verified from both directions by this requirement's own ACs: the seeded proof (real construction code FAILs) and passes-at-refactored-HEAD (the prose survivors above do not). The exception list SHALL be the **same named list** R-0033's enumerator unions in — one list, single-sourced in the implementation, so enumerator and lint cannot drift. Enforcement family: the repo's existing source-lint family — `libs/mnemra-host/tests/lint_workspace_clause.rs` and `libs/mnemra-host/tests/no_test_seams.rs` (trybuild compile-fail); the earlier "grep-shape" label mischaracterized both and is retired. The family citation is placement-and-technique precedent, NOT semantics inheritance: `lint_workspace_clause.rs`'s `run_lint` (`:150-212`) substring-matches the syn-parsed **function-block** token string of top-level free fns only — it strips no string literals (its SELECT classifier in fact matches literal contents), and doc attributes never reach its match surface only because the block-scoped `quote!(#func_block)` scan excludes attributes — so obligations (ii) and (iii) are NOT delivered by reusing it as-is, and R-0032's scan must additionally cover construction inside `impl` methods and nested items, which a top-level `Item::Fn` walk alone would miss. Recipe placement (under `verify-lint` or a sibling `verify-*` recipe) is implement-time; it MUST run under `just ci`. *Anchors: Frame F-3 part 2 (PD3 — the rule ships with its checker; P-ShiftLeft D1 — scope known mechanically; P-LockContract — the fixture seam is the enumeration point).*

### R-0033 — Source-derived membership enumerator + partition reconciliation

**Decision.** A CI-gated reconciliation check SHALL compute group membership **source-derived**: for each test binary, walk its sources **transitively** (the root `tests/*.rs` file, its `#[path]`-included `tests/common/` modules, and any `#[path]` includes those modules make in turn — today's tree is single-level; the transitive statement closes the future-nesting gap) for fixture references, and union the named production-bootstrap caller list (R-0032's same list). The enumerator SHALL NOT be derived from, seeded by, or filtered against `PG_TEST_FLAGS` — for the **enumerator**, the flag list appears only as clause (a)'s compared operand (clause (b) separately, and legitimately, reads it as one of the partition cells); an enumerator echoing it would compare the list to itself and could not fail (vacuity). The partition is defined over **full-binary execution cells** — invocation sites that run a binary's complete test set: `PG_TEST_FLAGS` (`justfile:96`), `NONPG_TEST_FLAGS` (`justfile:107`), and the named full-binary standalone gates (today exactly one: `smoke_e2e` via `verify-smoke`, `justfile:229`). **Filtered supplementary invocations** — a name-filtered subset of a binary run as an additional gate (today exactly one: `verify-signing-root`'s `--exact root_pin_gate_matches_embedded` run of `build_gate`, `justfile:266-270`) — are NOT cells; the check carries them as a separately-enumerated supplementary list outside the partition. The check computes, for every `libs/mnemra-host/tests/*.rs` integration binary, the count of cells executing its full set, and FAILS iff **(a)** computed membership differs from the `PG_TEST_FLAGS` contents in either direction, or **(b)** any binary's cell count ≠ 1 — count 0 is a never-run binary (a filtered supplementary invocation does not rescue it: its full set still never runs); count ≥ 2 is a genuinely twice-executed one. Under these semantics `build_gate` as it exists today PASSES (one cell, `NONPG_TEST_FLAGS`; its signing-root invocation is listed as supplementary, not counted). **Diagnostic contract (review r1):** on FAIL the check SHALL name each offending binary and which clause fired — (a) membership mismatch, with the computed-membership set printed, or (b) cell count ≠ 1, with the count; a bare PASS/FAIL is not an acceptable output surface, since the seeded proofs below must be attributable to a specific clause. *Anchors: Frame F-3 parts 1+3 (intake hard constraint — transitive-construction-aware membership, review-r1 M-1; P-TrustworthySignal — a membership list that silently omits binaries under-reports; PD3; P-ShiftLeft D1).*

**Acceptance Criteria:**
- [ ] The check exists, runs under `just ci`, and passes at the refactored HEAD with computed membership = the 17 enumerated members.
- [ ] Source-derivation is structural: the check's membership computation takes test sources + the named-caller list as its only inputs; `PG_TEST_FLAGS` appears only as the compared operand of clause (a) (reviewable in the check's implementation; proof 3 below is the mechanical demonstration).
- [ ] **Seeded can-fail proof (2 of 3) — partition:** a seeded binary present in neither flag list makes the check FAIL on clause (b) with cell count 0 (seed, failing output, removal recorded in PR evidence).
- [ ] **Seeded can-fail proof (3 of 3) — transitive enumeration, isolated to clause (a):** a seeded member that reaches the fixture **only** through a `#[path]`-included `tests/common/` shim, **placed in exactly one other execution cell** (e.g. added to `NONPG_TEST_FLAGS`) so clause (b) independently PASSES for it (cell count 1), while `PG_TEST_FLAGS` omits it. The check FAILS **on clause (a) specifically**: the failing output names the seeded binary and clause (a), and the computed-membership diagnostic showing the shim-mediated member was walked is captured in the PR evidence (seed, failing output, removal recorded). Anti-vacuity: with clause (b) satisfied by construction, no vacuous enumerator (e.g. one echoing `PG_TEST_FLAGS`) can produce this FAIL — only genuine transitive `#[path]` enumeration can. This proof closes the intake's M-1 verification loop: the enumerator demonstrably fires on `/common/`-mediated construction rather than inheriting tier-1's grep blind spot.
- [ ] The supplementary-invocation list exists in the check, contains `verify-signing-root`'s `build_gate` invocation, and is excluded from cell counting (`build_gate` passes with cell count 1).

### R-0034 — Group serialization preserved; uniform across the three CI recipes

## Out of Scope

Each exclusion below is either a permanent cut or a deferral with a named, mechanically-fired tripwire (decide-and-lock anti-hedge discipline). Intake Non-goals are carried verbatim in substance.

- **Mocking or externalizing Postgres.** Permanent (R-0018-b: real embedded PG + pgvector; serialize/share, never mock or externalize — locked).
- **Changing R-0006 isolation *semantics*.** Permanent for this work: per-test fresh-DATABASE on the shared engine preserves clean-DB-per-test; R-0037 is the in-scope guard.
- **Any wasmtime-side remedy for the #1852 abort** (engine singleton/reuse, `Config::macos_use_mach_ports(false)`, wasmtime upgrade, `Engine::unload_process_handlers`) — the per-test `PluginPool`/wasmtime lifecycle is untouched. **Deferred** (Frame F-4, all three elements): *decision content* — choose among the four remedies or an upstream fix for the machports handler-thread abort; *deferral anchor* — P-Defer (under R-0034's serialization the abort has no firing surface; the remedy shape should be chosen against the evidence that re-exposes it) + P-MinBlastRadius (a production-config change to fix a suppressed test symptom is disproportionate; machports backs wasmtime's hardware-trap handling — guest hardware faults become recoverable traps — foundational to the sandbox's fail-safe posture, P-0007-adjacent) + Honesty (the discriminator could not pin which of the two `machports.rs` abort branches fires; a targeted remedy would aim at an unconfirmed branch); *tripwires (mechanically fired)* — (i) the F-2 split tripwire fires (R-0036 > 300s) and option b is pursued: its mandated N ≥ 20 proof on un-serialized wasmtime-adjacent members is exactly the measurement that would re-surface the abort; or (ii) R-0035 reports any ABORT — which fails acceptance outright and forces the remedy conversation. (Filing upstream against wasmtime 45.0.2 `machports.rs` is an out-of-repo orchestrator action recorded in the discriminator dispatch's follow-ups, not spec scope.)
- **Test-body parallelism / the PG-only vs wasmtime group split (option b).** **Deferred** (Frame F-2, all three elements): *decision content* — un-serialize the PG-only members (those with no reachable wasmtime `Engine` construction), keep wasmtime-touching members serialized, with per-member flake-runner proof N ≥ 20 on the new configuration; *deferral anchor* — P-Defer (the evidence does not exist until the post-collapse wall-clock is measured; parallel bodies are mechanism ahead of evidence); *tripwire (mechanically fired)* — R-0036's mandatory CI-runner measurement against the 300s threshold: above → the split decision routes to the maintainer as a new intake (the collapse ships regardless; the measurement is mandatory, so the comparison cannot silently not-happen).
- **cargo-nextest adoption.** **Don't-adopt** (Frame F-5 — a lock, not a deferral; no tripwire because the conflict is with a locked decision, not missing evidence): nextest's process-per-test execution model boots the "shared" engine once per test again, structurally defeating R-0026's in-process sharing. Re-opening is self-announcing: adopting nextest would require re-deriving the fixture architecture, which re-opens the Frame through the normal amendment path (P-LockContract; S2 — installed-base popularity is not a fit claim).

- **smoke_e2e restructuring.** It boots the engine out-of-process via the real binary's production startup; different surface, stays as-is — its own full-binary execution cell in R-0033's partition (`verify-smoke`).
- **Disk-full / `ci.yml` free-space work** (tier-1 parked operational item, #1860 family). **Deferred** — tripwire unchanged from tier-1: fired when CI hits disk pressure; tracked as its own item.

## Scenarios

**S1 — Fixture cold-start and reuse within a binary.** [R-0026]
*Given* a fixture-consuming member binary whose engine has not yet booted,
*When* the first test acquires storage through the fixture and a later test in the same binary acquires again,
*Then* exactly one embedded engine boots (first acquisition), the second acquisition reuses it without a second boot, and each test receives its own fresh database on that engine.

**S2 — Per-test database isolation on the shared engine.** [R-0027]
*Given* two tests in the same member binary, each holding its fixture-provisioned database and app-role pool,
*When* test A writes and commits rows and test B lists/reads through its own pool,
*Then* the two database names are distinct and test B observes none of test A's rows.

**S3 — Seeded guard-lint violation fails CI (can-fail proof 1).** [R-0032]
*Given* a seeded test file in `libs/mnemra-host/tests/` that constructs an embedded engine directly (e.g. calls `EmbeddedEngine::start`) outside the fixture module and is not on the by-name exception list,
*When* the guard lint runs,
*Then* the check FAILS, and the failing output is captured in the PR evidence before the seed is removed.

**S4 — Seeded partition violation fails CI (can-fail proof 2).** [R-0033 clause (b)]
*Given* a seeded integration binary present in neither `PG_TEST_FLAGS` nor `NONPG_TEST_FLAGS` nor any named standalone gate,
*When* the reconciliation check runs,
*Then* the check FAILS with that binary's cell count = 0, captured in the PR evidence before the seed is removed.

**S5 — Seeded transitive member fails reconciliation on clause (a) alone (can-fail proof 3; M-1 closure).** [R-0033 clause (a)]
*Given* a seeded member binary that reaches the fixture **only** through a `#[path]`-included `tests/common/` shim, placed in exactly one other execution cell (e.g. `NONPG_TEST_FLAGS` — so clause (b) independently PASSES for it with cell count 1) while `PG_TEST_FLAGS` omits it,
*When* the reconciliation check runs,
*Then* the computed membership includes the seeded binary (source-derived, transitive), the check FAILS naming the binary and clause (a) specifically, and the failing output plus the computed-membership diagnostic are captured in the PR evidence before the seed is removed — a FAIL no vacuous enumerator (one echoing `PG_TEST_FLAGS`) could produce, since clause (b) is satisfied by construction.

**S6 — Deterministic teardown at binary exit.** [R-0028]
*Given* a member-binary run that completes normally — including the all-tests-passed path,
*When* the process exits,
*Then* no fixture-spawned postmaster survives: the live `bin/postgres` count is back at the pre-run baseline (`BASE_PM`).

**S7 — The tripwire firing path: above-threshold measurement ships AND routes.** [R-0036]
*Given* the post-collapse CI-runner measurement sums above 300 seconds across the three recipes' PG-group invocations,
*When* the PR evidence is assembled,
*Then* the collapse still ships (no acceptance criterion conditions on the number) AND the evidence carries a routing note putting the option-b split decision to the maintainer as a new intake.

**S8 — Determinism proven, not asserted.** [R-0034, R-0035]
*Given* the refactored group under all three `verify-*` recipes,
*When* `scripts/flake-runner.sh` runs N ≥ 20 per member locally and the temporary Linux CI proof job runs over all members,
*Then* ABORT classifications = 0 everywhere, and the recorded numbers ship with the PR evidence.

**S9 — Attribution-first wiring of the never-run binaries.** [R-0031]
*Given* `artifact_list_paging` and `artifact_list_paging_whitebox` at the pre-refactor parent commit,
*When* each is run standalone and the result recorded before the migration and wiring land,
*Then* any latent red surfaced by the CI wiring is attributable to the wiring evidence, not to the fixture refactor.
