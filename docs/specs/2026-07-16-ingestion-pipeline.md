# Spec: Ingestion pipeline — ELT harness · loader seam · dogfood-critical surfaces

> **Spec for:** task #2439 (design run); register entry `1.2.0` (V0.1) + #324 (binary-asset extraction); board W2-2.
> **Date:** 2026-07-16
> **Design ref:** the locked Frame [`docs/intent/ingestion-pipeline-frame.md`](../intent/ingestion-pipeline-frame.md), locked 2026-07-17, blob `f56b3685b3422144a0290114d11537238a09ed9a` — the closed world for this spec.

## Purpose / context

After `0.14.0`'s one-shot corpus migration, content keeps arriving — markdown notes and binary documents (PDF/PPTX/XLSX/DOCX/CSV/JSON) — and nothing ingests it: content enters the corpus only when a human notices and re-runs something by hand, and every derived view of it is recomputed per session. This spec designs the **ingest subsystem** inside the mnemra-core host: a source registry, reconciliation-first arrival drivers, the shared loader seam the `0.14.0` migrator and the `1.2.0` watcher both drive, durable raw staging, a transform harness on a minimal `JobQueue` port, egress-denied binary extraction (#324), recency instrumentation, and the quarantine/failure path — so content arriving in the corpus enters durably and automatically, and derived context is (re)computed from what was persisted rather than re-ingested.

### R-0100 — Loader seam: typed stage chain; driver differences stay in drivers; the admission chain is the D3 insertion point

*Locks Frame IP-2 (intake Hard constraint 1). Anchors: intake HC1 + HC4; D3 (differentiation-trio record, ratified 2026-07-06); P-LockContract — the seam is the intrinsic contract; P-MinBlastRadius — one chain, one slot per concern; [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-2. [Frame IP-2; QA-1]*

- **R-0100-a (the seam contract):** The loader seam SHALL be a typed stage chain with one contract: **bytes + source-metadata in → a durably-staged raw artifact with [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md)-envelope provenance out, a transform job enqueued.** The ordered stages SHALL be: `parse/normalize` (format detection, text normalization; binary kinds pass through raw) → `frontmatter-map` (structured frontmatter extraction for markdown kinds) → `provenance-stamp` ([P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) envelope; trust-class mapping per R-0108; PE-2 not-set permissive defaults applied on this sanctioned write path) → `admission chain` → `write-through`. *(Binary-observable: a markdown fixture driven through the seam produces one staging row with a populated provenance envelope and one enqueued job; a construction audit finds the five stages in order — [acceptance test] + [construction + review audit].)*
- **R-0100-b (write-through is transactional; enqueue is post-commit best-effort):** `write-through` SHALL be a transactional staging insert; the transform-job enqueue SHALL follow **post-commit, best-effort** — the durable staged row is the guarantee, and the reconciliation scan re-enqueues any staged-but-untransformed orphan (R-0101-d / R-0112-b). The `JobQueue` port SHALL carry no SQL-transactional coupling; the Postgres adapter MAY enqueue inside the staging transaction as an adapter-local optimization, never as a port guarantee. *(Binary-observable: a fixture that commits a staging row then fails the enqueue leaves a durable staged row that the next reconciliation scan re-enqueues; the `JobQueue` port trait signature carries no transaction handle — [acceptance test] + [construction + review audit].)*
- **R-0100-c (the admission chain is the D3 classify-at-write insertion point):** The admission chain SHALL be an ordered set of registered admission stages; the V0.1 content is the fail-closed validation set (size, format, path, registration checks — R-0112 / R-0115). A classification stage SHALL be insertable by registering one more admission stage — no schema change, no path reshape. Each admission stage SHALL implement the typed contract `(candidate artifact + provenance) → admit | quarantine(reason) | annotate(policy fields)`, where `annotate` writes only PE-2 policy columns that already exist. The stage itself SHALL NOT be designed here (Frame Non-goal 7; Tier-2-gated per D3). *(Binary-observable: a pass-through admission stage registered at the D3 point requires zero schema or path change and the staging DDL is unchanged — [construction + review audit]; a fixture stage returning `quarantine(reason)` quarantines the row without a partial write — [acceptance test].)*
- **R-0100-d (driver differences stay in drivers):** The seam SHALL admit both drivers with no duplicated loader logic. The `0.14.0` migrator SHALL own kill-and-resume manifest, byte-equivalence verification, and source-read-only posture; the `1.2.0` watcher SHALL own arrival detection, dedup consultation, and recency stamping. Building the migrator driver is out of scope (Frame Non-goal 1); the seam SHALL NOT preclude it. *(Binary-observable: a construction audit finds no migrator-only or watcher-only concern (kill-and-resume, byte-equivalence, dedup, recency) inside the seam's stage code — [construction + review audit].)*

### R-0101 — Arrival: reconciliation-first; events accelerate latency; the scan reconciles three joins

*Locks Frame IP-3 (2a decision 3). Anchors: 2a decision 3 (LOCKED); intake SC4/SC11; P-GuaranteeByMechanism — correctness is carried by the scan mechanism, not by OS event-delivery guarantees; P-TrustworthySignal — the recovery path runs constantly and cannot silently rot. [Frame IP-3; QA-2]*

- **R-0101-a (reconciliation-first):** The periodic/startup reconciliation scan SHALL be the primary arrival mechanism. The scan SHALL walk registered roots and join observed state (path, size/mtime, content hash on candidates) against staging identity (R-0109) — reconciliation on data, not on event labels — so a missed filesystem event costs latency, never correctness. Startup reconciliation SHALL be the same scan as periodic reconciliation. *(Binary-observable: QA-2 — with fs events disabled entirely, a file added under a registered root is staged on the next scan — [acceptance test].)*
- **R-0101-b (events are an accelerator only):** Filesystem-event subscription (a cross-platform fs-notification dependency, Green-tier — R-0115 / § Numeric calibrations) SHALL trigger targeted early scans of changed paths and SHALL NOT be a correctness dependency. Event floods SHALL coalesce structurally into scan batches under an **event→scan debounce/coalescing window** (§ Numeric calibrations); identical-re-arrival event emission SHALL be rate-bounded to at most one arrival-event row per staging identity per **identical-re-arrival suppression window** (§ Numeric calibrations — the debounce window doubles as this bound at V0.1, sized so the two never disagree at dogfood scale), so a touch-flood writes a bounded arrival-event row count rather than one row per event. *(Binary-observable: a thousand fs events against one unchanged file resolve to at most one scan observation and at most one arrival-event row inside the suppression window — [acceptance test].)*
- **R-0101-c (the size/mtime pre-filter is bounded by a full-content rehash cadence):** The scan's size/mtime candidate pre-filter SHALL be a fast-path heuristic, not an integrity boundary. Because an in-place, size-preserving edit with a reset mtime evades it, the scan SHALL additionally re-hash every file within a **periodic full-content-rehash cadence** (§ Numeric calibrations) regardless of the pre-filter, bounding heuristic evasion to that window's staleness. The full-rehash pass SHALL be I/O-paced (a bounded per-pass work budget) so its bulk read load does not contend with the `arrival→staged` critical path (R-0111) on a larger corpus; the pace is a § Numeric calibrations tunable, the existence of pacing the lock. *(Binary-observable: a size-preserving content edit with the mtime reset to its prior value is not staged on a pre-filter-only scan but IS staged on the next full-rehash pass — [acceptance test].)*
- **R-0101-d (the scan reconciles staging→jobs, re-enqueuing true orphans):** The scan SHALL reconcile a second join alongside filesystem→staging — **staging→jobs**: a staging row in `status = staged` whose post-commit best-effort enqueue (R-0100-b) never landed a transform job, evaluated past the enqueue-orphan window (§ Numeric calibrations), SHALL be re-enqueued. A row is a **true orphan** (re-enqueue target) only when **no** transform job exists for it in **any** state — the re-enqueue guard set is `state ∈ {pending, claimed, completed, dead_letter}`, `dead_letter` **included** so a poison-message row that has already dead-lettered is **never** re-enqueued into a fresh retry cycle (the crash-window between a job's terminal `dead_letter` transition and the R-0104-f staging-quarantine write is closed by the R-0101-f third join, not by re-enqueue). The exact orphan predicate is rendered in [P-0024](../src/adrs/P-0024-ingest-pipeline-shape.md). *(Binary-observable: a staging row committed with no corresponding job row is re-enqueued on the next scan and transforms to `status = transformed`; a staged row that already has a live, completed, **or dead-lettered** job is not re-enqueued — [acceptance test] + [operator SQL surface].)*
- **R-0101-e (scan interval config-tunable):** The scan interval SHALL be config-tunable, defaulted against the recency bound (R-0110 / § Numeric calibrations). *(Binary-observable: the scan interval is read from config at startup; a shortened test-environment value is honored — [acceptance test].)*
- **R-0101-f (the scan reconciles staging↔dead-letter, completing interrupted terminal-failures):** The scan SHALL reconcile a **third join** — **staging↔dead-letter**: a staging row in `status = staged` whose only transform job is in the terminal `dead_letter` state (no job in `{pending, claimed, completed}`) is **not** an orphan to re-enqueue but an **interrupted terminal-failure** — the R-0104-f staging-quarantine write did not land before the harness lost the row (crash between the job's `dead_letter` transition and the quarantine write). The scan SHALL drive that staging row to its designed terminal state — `status = quarantined`, `rejection_class = transform-failure` — **idempotently** (a row already `quarantined` is a no-op; no canonical row is written). This closes the poison-loop gap in the R-0112-c working-state → working-state guarantee (a dead-lettered row must never re-enter a fresh retry cycle) using the same reconciliation mechanism as the other two joins — one scan, three joins, no cross-resource transaction (the R-0103-c / JQ-3 no-SQL-coupling discipline is preserved). The predicate is rendered in [P-0024](../src/adrs/P-0024-ingest-pipeline-shape.md) IPS-4. *(Binary-observable: a staging row left `staged` with a single `dead_letter` transform job and no live/completed job is completed to `quarantined` with `rejection_class = transform-failure` on the next scan — not re-enqueued — and a re-run of the scan over an already-quarantined row makes no further change — [acceptance test] + [operator SQL surface].)*

### R-0102 — Durable raw staging: logged, one superset table, in-substrate blob home, per-source-kind retention

*Locks Frame IP-4 (intake Hard constraint 2). Anchors: intake HC2 (ELT, LOCKED) + SC2/SC3/SC5; [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT reconstructability; [P-0016](../src/adrs/P-0016-edge-schema.md) ES-1 one-superset-table precedent; Simplicity; P-Defer/DF1 on externalization + attachments wiring. [Frame IP-4; QA-6]*

- **R-0102-a (durable, never `UNLOGGED`):** Staging tables SHALL be ordinary logged tables. `UNLOGGED` SHALL NOT be used for staging — a raw store Postgres truncates on crash recovery cannot be the ELT source-of-record (raw = truth). *(Binary-observable: the staging table's DDL carries no `UNLOGGED` qualifier — [operator SQL surface].)*
- **R-0102-b (one superset staging table):** Staging SHALL be **one** superset table ([§ Data Model](#data-model)), not per-kind table proliferation; per-kind variance SHALL be data (registry fields + a `source_kind` column + payload polarity), not schema forks. The payload SHALL be `raw_text | raw_bytes` with **exactly one non-null**, CHECK-enforced. *(Binary-observable: exactly one staging table exists for all source kinds; a row with both `raw_text` and `raw_bytes` non-null, or both null, is rejected by a CHECK constraint — [operator SQL surface].)*
- **R-0102-c (in-substrate blob home at V0.1):** Retained binaries SHALL live in-substrate (`raw_bytes`, TOAST-backed), bounded by the per-file size cap (R-0115), so the staging row and its payload commit in one transaction (no fs/db two-phase residue). Blob externalization SHALL NOT be built at V0.1; it is deferred with the mechanical trip-wire of a spec-named staging-storage-bytes or backup-duration threshold in the run records (§ Numeric calibrations). *(Binary-observable: a binary fixture stages with its bytes in `raw_bytes` in the same transaction as the row; no filesystem blob path is written — [acceptance test] + [construction + review audit].)*
- **R-0102-d (per-source-kind retention; every V0.1 kind is `long-lived`):** Retention SHALL be a per-source-kind closed enum `{long-lived, transient}` on the registry. Every V0.1 source kind SHALL be `long-lived` (watched markdown and retained binaries alike — SC3 requires re-derivation with the arrival source absent, so the fs copy is never the system's raw). The `transient` value SHALL exist in the vocabulary (validator-first) though no in-scope kind uses it. *(Binary-observable: the retention enum admits both values; every registered V0.1 source kind resolves `long-lived`; a `transient` registration is accepted by the validator though unused — [acceptance test] + [operator SQL surface].)*

### R-0103 — The `JobQueue` port: minimal, at-least-once, single-claimant; Postgres SKIP LOCKED adapter

*Locks Frame IP-5 (2a decision 1). Anchors: 2a decision 1 (LOCKED); the two-adapter test as a design-time swap-readiness check ([P-0010](../src/adrs/P-0010-storage-substrate-engine.md) D5 pattern); [P-0022](../src/adrs/P-0022-coordination-cluster.md) D-STORAGE operational-table precedent; P-LockContract; Simplicity. Rendered at ADR precision in [P-0025](../src/adrs/P-0025-job-queue.md). [Frame IP-5]*

- **R-0103-a (the port operations):** This bundle SHALL specify a minimal `JobQueue` port with operations expressed in delivery semantics, never engine idioms: `enqueue(kind, payload, scheduled_for?)`, `claim(kinds, lease)`, `complete(job)`, `fail(job, retryable?)` (bounded attempts, backoff policy), dead-letter disposition for exhausted retries, and `schedule` via `scheduled_for`. The harness is its first consumer. *(Binary-observable: the port trait exposes exactly these operations; a job kind with no registered handler does not enqueue — validator-first — [construction + review audit] + [acceptance test].)*
- **R-0103-b (at-least-once, single live claimant):** Delivery SHALL be at-least-once with a single live claimant per job — `claim` carries a lease/visibility window (its duration defaulted in § Numeric calibrations, sized to **exceed the longest lane's wall-clock bound plus margin** so a still-running job is never re-delivered to a second claimant), and an expired claim re-delivers. A **retryable `fail()` and a lease expiry SHALL both return the job to `pending`** (re-delivered after backoff, incrementing `attempts`); a non-retryable `fail()` or an attempts-exhausted retry SHALL move the job to the terminal `dead_letter` state — so there is **no non-terminal `failed` resting state** (the job state enum is `{pending, claimed, completed, dead_letter}`). Because the staging-row quarantine on retry exhaustion (R-0104-f) is a **separate resource write** from the job's `dead_letter` transition and the two are not transactionally coupled (R-0103-c / JQ-3 forbid it), the re-enqueue orphan guard (R-0101-d) counts `dead_letter` as a job-that-exists — guard set `state ∈ {pending, claimed, completed, dead_letter}` — so a dead-lettered row is never re-enqueued, and a `staged` row stranded with only a `dead_letter` job (quarantine write lost to a crash) is completed to `quarantined` by the R-0101-f third join rather than resurrected. Handlers SHALL therefore be idempotent (R-0104-e). Job identity SHALL be UUID v7; payloads JSONB; job kinds a code-registered closed vocabulary. *(Binary-observable: two concurrent claimants of the same job observe exactly one live claim; a job whose lease expires mid-flight is re-delivered and its idempotent handler converges; a retryable-failed job reads `pending`, not `failed`; a dead-lettered job is not re-enqueued by the reconciliation scan — [acceptance test].)*
- **R-0103-c (Postgres adapter on SKIP LOCKED; two-adapter test at drafting):** The V0.1 adapter SHALL implement `claim` with `FOR UPDATE SKIP LOCKED`. The port surface SHALL be satisfiable by an SQS-class or Temporal-class adapter — **no `LISTEN/NOTIFY` in the contract** (a low-latency wake hint is an optional adapter capability; polling is the portable baseline), no SQL-transactional coupling in the port. A design-time two-adapter conformance test SHALL demonstrate the port's semantics over the Postgres adapter and a second in-memory or stub adapter. *(Binary-observable: the two-adapter conformance suite passes over both adapters; the port trait references no `LISTEN/NOTIFY` and no transaction handle — [acceptance test] + [construction + review audit].)*

### R-0104 — Transform-harness conventions: transform-as-job, update-with-audit re-derivation, lineage, declared determinism, lineage-addressed canonical id

*Locks Frame IP-6 (ADR-16 materialized). Anchors: intake SC2/SC3/SC6 + fold M2; overview open questions 8/9 dispositioned; [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT lineage; [P-0001](../src/adrs/P-0001-storage-layout.md) system-versioned history pattern; [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-10 citation stability; P-TrustworthySignal — a declared property carries a validator. [Frame IP-6; QA-6]*

- **R-0104-a (transform-as-job):** The loader SHALL enqueue one transform job per staged row (enqueue off the critical path — R-0111). A transform SHALL be a registered, versioned unit `(source_kind, transform_id, version, declared_deterministic: bool)` with signature `staged raw in → canonical candidate rows out`, holding no capability beyond bytes-in/content-out (extraction transforms additionally run sandboxed, R-0105). *(Binary-observable: the transform registry is a closed code registration; a staged row of a given `source_kind` enqueues its registered transform — [construction + review audit] + [acceptance test].)*
- **R-0104-b (canonical writes carry `derived_from` lineage):** Transform outputs SHALL land as ordinary content rows ([P-0001](../src/adrs/P-0001-storage-layout.md) C1) each carrying a `derived_from` soft ref to the staging row ([P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT). The staging row is the source-of-record for re-derivation; the canonical row is D-SoT's source of truth for the artifact's served data. *(Binary-observable: every canonical row a transform writes carries a `derived_from` ref resolving to its staging row — [operator SQL surface].)*
- **R-0104-c (re-derivation policy = update-with-audit):** A re-run — a new transform version, or a re-arrival superseding content at the same identity — SHALL **update the existing canonical row** at the same artifact identity, with the [P-0001](../src/adrs/P-0001-storage-layout.md) system-versioned history pattern as the audit trail. New-rows-append SHALL NOT be used: minting a new artifact identity per derive-run would break [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-10 citation stability and push supersession bookkeeping into every consumer. *(Binary-observable: re-running a transform at a higher version over the same staging identity updates the same canonical row id and appends a history record; no second canonical id is minted — [acceptance test] + [operator SQL surface].)*
- **R-0104-d (lineage-addressed canonical identity; the one-to-many discriminator):** "Same artifact identity" SHALL be carried by a **stable, lineage-addressed canonical id**, deterministically derived from `(workspace_id, source_id, source_path, transform_id, output_key)` — the R-0109 staging lineage minus its content hash, plus the transform identity, plus a transform-assigned **`output_key`** discriminating each logical output where one transform emits many canonical rows (a document extracting into several artifacts). The canonical id SHALL be **stable across content-changing re-derivations** (update-with-audit's requirement — so `content_hash` and `transform_version` are NOT part of the identity) and **reconstructable from staging alone** after delete-derived (SC3, including for a `declared_deterministic: false` transform). Content-hash addressing SHALL NOT be used for canonical identity. The spec pins this scheme; [P-0024](../src/adrs/P-0024-ingest-pipeline-shape.md) renders it and reconciles [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-10's "content-addressed, per the V0 substrate" citation wording against it (PE-10 citation stability preserved by the stable-id-with-history model, not by content-hash addressing). *(Binary-observable: two derive-runs over the same staging lineage with changed content resolve the same canonical id; one transform emitting N outputs mints N distinct ids stable across re-runs; a delete-derived → re-run reconstructs the same id set from staging alone — [acceptance test].)*
- **R-0104-e (re-run is first-class; idempotent convergence, not byte-identity):** Delete-derived → re-run SHALL repopulate canonical from staging alone with no read of the arrival source (SC2/SC3). Deterministic byte-identity of outputs SHALL NOT be a harness requirement; the harness SHALL require **idempotent convergence** — re-running an unchanged transform over unchanged raw converges to a stable canonical state. Per-transform determinism SHALL be a declared property (`declared_deterministic`), tested by seeded re-run byte-compare **where declared `true`**; the V0.1 extraction transforms declare `true`. *(Binary-observable: delete every canonical row and re-run → canonical repopulates from staging with the arrival file deleted; a `declared_deterministic: true` transform passes the seeded byte-compare re-run — [acceptance test].)*
- **R-0104-f (transform failure quarantines, never partial-writes):** A transform failure after retry exhaustion SHALL quarantine the staging row with the designed error (R-0112); it SHALL NOT write a partial canonical set (the canonical write is transactional per staging row). Because this staging-quarantine write is not transactionally coupled to the job's `dead_letter` transition (R-0103-c / JQ-3), a crash landing between the two leaves a `staged` row with only a `dead_letter` job; the R-0101-f staging↔dead-letter reconciliation completes that row to `quarantined` idempotently, so the failure terminal is durable across the crash-window and the row is never re-enqueued into a fresh retry cycle. *(Binary-observable: a transform seeded to fail after its retry budget leaves the staging row `quarantined` with `rejection_class = transform-failure` and zero canonical rows; a `SIGKILL` fired after the job dead-letters but before the quarantine write still resolves to `quarantined` on the next scan, never to a re-enqueue — [acceptance test].)*
- **R-0104-g (retire-on-shrink: a re-run that stops emitting an `output_key` retires that canonical row):** Because the canonical id scheme is one-to-many (R-0104-d) and re-derivation is update-with-audit (R-0104-c), a re-run — a new transform version or a re-arrival superseding content at the same staging identity — SHALL **retire** each canonical row whose `output_key` is present in the prior run's set but **absent from the new run's set** — every `output_key` in the non-empty set-difference `prior_output_keys − new_output_keys`, **regardless of whether the new run also emits added `output_key`s the prior run lacked**. A pure shrink (a PDF that extracted into three sections now extracts into two, `{a,b,c} → {a,b}`) and a **drop-and-add** (`{a,b,c} → {a,b,d}`: `c` dropped, `d` added, so the new set is **not a subset** of the prior) both retire the dropped `output_key`s; the drop-and-add additionally mints the added id. Retirement happens in the **same re-derivation transaction** that updates the surviving ids and mints any added ones. The retire diff `prior − new` is computed **only on the transform's success path**: a re-run that *completes* emitting an empty `output_key` set legitimately produces nothing and retires **all** prior canonical rows for the lineage (`prior − ∅ = prior`), whereas a transform **failure** never reaches the diff — it diverts through retry-exhaustion to R-0104-f quarantine (`rejection_class = transform-failure`) before any retire is computed, retiring and writing nothing; an empty output set is thus never on its own the retire trigger, only a *successful* empty completion is. Retire SHALL be a **soft-delete / mark-superseded** recorded through the [P-0001](../src/adrs/P-0001-storage-layout.md) system-versioned history (a retired row leaves an audit trail and stops serving; its id is not re-minted), never a hard delete and never a new identity — so PE-10 citation stability holds for the retired id (a citation resolves to a superseded-terminal, not a dangling reference). Absent this rule the update-with-audit path would update present outputs and leave a vanished output's stale canonical row serving indefinitely; the full delete-derived → re-run path (R-0104-e) already drops vanished outputs by reconstructing the id set from staging alone and is unaffected. *(Binary-observable: a transform that emitted `output_key`s {a,b,c} re-run to emit {a,b} updates a and b in place, retires c (marked superseded in history, no longer served, id not re-minted), and mints no new id; a **drop-and-add** re-run {a,b,c} → {a,b,d} likewise retires c and additionally mints d, even though {a,b,d} is not a subset of {a,b,c}; a re-run that **successfully completes** emitting ∅ retires all of {a,b,c}, whereas a **failed** transform retires nothing (routed to R-0104-f quarantine, zero canonical writes); a subsequent delete-derived → re-run reconstructs exactly the surviving `output_key` set from staging alone — [acceptance test] + [operator SQL surface].)*

### R-0106 — Watcher: in-process supervised task; least authority; TOCTOU-safe open; per-file failure containment

*Locks Frame IP-13. Anchors: the architecture-overview load-shedding contract (verified current, Frame §8); intake Frame-carry N2; P-TrustworthySignal; D1 — recency measured, not promised. [Frame IP-13; QA-7]*

- **R-0111-a (staging protected; transforms off the critical path):** Under pressure, the staging insert (Load) SHALL be the protected critical path; enqueue and transform execution SHALL be off it. The queue depth SHALL be the absorption buffer, measured (queue-depth/backlog gauges in the run records), unbounded at V0.1 by policy (bounded physically by the R-0115 intake caps — content that never stages never backlogs). *(Binary-observable: QA-7 — under an arrival burst exceeding transform throughput, staging latency stays within its bound while `staged→canonical` degrades measurably; the backlog is visible in the queue-depth gauge, and the concurrently-executing transform count never exceeds the R-0111-b pool bound — [acceptance test].)*
- **R-0111-b (transform execution runs a bounded, host-owned worker pool):** The queue-depth buffer (R-0111-a) is unbounded by policy, but transform **execution concurrency** SHALL be **bounded**: the harness SHALL run a fixed-size **transform-worker pool** — a configured maximum number of concurrent claim-loops/workers each executing at most one claimed job at a time — so an arrival burst absorbs into queue depth (measured), never into an unbounded number of simultaneously-executing extraction instances. This pool is **host-owned and sized by this bundle**; it is **not** [P-0007](../src/adrs/P-0007-plugin-resource-limits.md)'s 3–5 plugin-pool slots — extraction is a host-owned transform unit, not a [P-0002](../src/adrs/P-0002-core-plugin-partition.md) plugin (R-0105-a) — and it is a **distinct mechanism** from P-0007's per-instance fuel/epoch/memory kill-and-replace, which governs a single Lane-A extraction's WASM-instance lifecycle *inside* one worker (the "pool-slot poison-and-replace" R-0115-c names for Lane A), not how many transforms run at once. The pool size is a § Numeric calibrations dogfood-scale tunable; its **existence and boundedness are the lock**. The aggregate execution-resource ceiling under any burst is therefore bounded and stated: **`worker_pool_size × per-lane execution budget`** (memory/CPU per R-0115-c) — this product, not the queue depth, is what a QA-7 test asserts staging latency against, making QA-7 falsifiable (an unbounded worker count would let a burst spin up unbounded 512-MiB extraction instances and itself threaten the staging latency R-0111-a protects). *(Binary-observable: under an arrival burst the count of concurrently-executing transform jobs never exceeds the configured pool size; peak host extraction-memory stays at or below `pool_size × per-lane memory budget` — [acceptance test] + [operator SQL surface].)*

### R-0112 — Failure path: closed rejection vocabulary, fail-closed, working-state → working-state

*Locks Frame IP-14 (intake SC9/SC10). Anchors: intake SC9/SC10 + the W2-1 determinacy ruling as intake-scoped; P-SecurityLayered fail-closed; P-ShiftLeft D2 — closed enums carry validators; P-TrustworthySignal. [Frame IP-14; QA-3, QA-5]*

- **R-0112-a (closed rejection vocabulary):** Rejection classes SHALL be a closed enum (extension is a spec amendment): `malformed-file`, `unparseable-binary`, `oversized-input`, `decompression-bomb`, `path-traversal-name`, `unregistered-source`, `transform-failure`, `unextractable-here`. Each rejection SHALL be a distinct designed error with a per-class error-path test; each SHALL either quarantine (staging row `status = quarantined` + `rejection_class` + diagnostic) or refuse pre-staging (`unregistered-source`, `path-traversal-name` — nothing to stage), never partially write. *(Binary-observable: an error-path test exists per rejection class; the rejection-class column admits only the eight values — [acceptance test] + [operator SQL surface].)*
- **R-0112-b (no partial canonical write; staged-durability-plus-reconciliation is the delivery guarantee):** The canonical transform write SHALL be transactional per staging row (failure rolls back whole). The staging write SHALL be transactional; the enqueue SHALL follow post-commit best-effort (R-0100-b), and a staged-but-unenqueued orphan SHALL be re-enqueued by the reconciliation scan (R-0101-d). *(Binary-observable: a kill between staging commit and enqueue leaves a durable staged row and zero partial canonical rows; the next scan re-enqueues it — [acceptance test].)*
- **R-0112-c (working-state → working-state):** Every operation SHALL move working-state → working-state: watcher crash → supervised restart + reconciliation scan; kill mid-load → the transaction never half-commits and the next scan re-observes the source; transform kill/failure → job lease expires, at-least-once re-delivery, idempotent handler converges. Residue of any interrupt SHALL be inert (an unclaimed job, an un-transformed staged row — both re-processable, neither corrupting). *(Binary-observable: QA-3 — a `SIGKILL` at each pipeline stage (scan / load / transform) leaves no indeterminate state; a re-run after each interrupt converges with no manual repair and no partial canonical rows — [acceptance test].)*
- **R-0112-d (quarantine is inert, visible, bounded):** Quarantined rows SHALL never reach transforms or canonical; they SHALL be listable via the admin CLI and counted in run records; release-from-quarantine SHALL be a re-scan-driven re-arrival, not a mutation path. The quarantine store SHALL carry an **aggregate bound** (a size/count bound with an eviction/rotation policy — § Numeric calibrations); the per-file cap bounds one file, not a malformed-file flood. Quarantine listing and inspection SHALL be **metadata-only** (path, `rejection_class`, diagnostic — never a payload render); any payload retrieval SHALL be an explicit operator action that re-enters the sandboxed extraction path. *(Binary-observable: a malformed-file flood past the aggregate bound triggers the eviction/rotation policy rather than an unbounded write; `quarantine inspect` returns metadata with no payload body — [acceptance test].)*

### R-0113 — Reliability posture: contractual recovery behavior now; numeric target deferred on a named instrument

*Locks Frame IP-15. Anchors: the Reliability supporting value (2026-07-04) + its own trip-wire semantics; intake Frame-carry N1; P-Defer/DF1; P-InstrumentBefore — measure before you promise. [Frame IP-15; QA-2, QA-3]*

- **R-0113-a (no numeric availability/recovery target at V0.1):** No numeric availability or recovery target SHALL be adopted at V0.1. What binds instead is **contractual recovery behavior**: SC11's reconcile-on-restart (R-0101-a / QA-2) and SC9's working-state → working-state (R-0112-c / QA-3), both binary observables; the recency metric (R-0110) is the future target's calibration baseline. *(Binary-observable: the spec states no availability SLI/number; QA-2 and QA-3 pass as the binding recovery observables — [acceptance test] + [construction + review audit].)*
- **R-0113-b (the deferral's firing instrument is named):** The deferred numeric target (a stated recovery/availability target plus its SLI) SHALL fire on either the mechanical trip-wire (the recency/run-record measures breaching the spec's stated bound in dogfood) **or** the self-announcing one (the first adopter deployment). *(Binary-observable: the deferral names both firing conditions; neither is "later"/"eventually" — [construction + review audit].)*

### R-0114 — Seam build order: migrator-first, rendered as design obligation

*Locks Frame IP-16 (2a decision 4). Anchors: 2a decision 4 (LOCKED); intake HC1/HC9 + SC1; P-MinBlastRadius MB5; P-LockContract. [Frame IP-16; SC1]*

- **R-0114-a (contract specified now; migrator consumes it):** The seam contract (R-0100), staging shape (R-0102), queue port (R-0103), and harness conventions (R-0104) SHALL be specified in this bundle so the `0.14.0` migrator build consumes a locked contract rather than improvising one. The seam's first commit series SHALL introduce the seam (MB5); the watcher driver SHALL land behind it, never in lock-step across modules. No migrator-internal mechanism (kill-and-resume, byte-equivalence, source-read-only) SHALL be specified here. *(Binary-observable: this spec fully specifies the seam/staging/queue/harness contracts; it specifies no migrator-internal mechanism — [construction + review audit].)*

### R-0115 — Resource-exhaustion bounds at intake (TB-2)

*Locks Frame §5 TB-2. Anchors: intake Risk profile (resource exhaustion) + the W2-1 fetch/unpack analogue; [P-0007](../src/adrs/P-0007-plugin-resource-limits.md) mechanisms; P-GuaranteeByMechanism; Security | Simplicity default-to-Security. [Frame §5 TB-2; QA-5]*

- **R-0115-a (per-file size cap before read-in):** A per-file size cap (§ Numeric calibrations) SHALL be enforced **before read-in** — the size decision reads file **metadata only** (`stat`/directory-entry size), never the payload; an oversized file SHALL be refused/quarantined `oversized-input` without being read into memory. *(Binary-observable: a file above the cap is quarantined `oversized-input` — [acceptance test]. The "never fully read into memory" internal property is checked through two channels, since it is not a pure black-box outcome: a construction audit confirms the size gate consumes only metadata before the payload read, and a driven **latency proxy** confirms rejection latency stays flat as file size grows well beyond the cap — a full read would scale with size — [construction + review audit] + [acceptance test].)*
- **R-0115-b (decompression/expansion budget as explicit counters):** Archive-shaped formats (ZIP-of-XML) SHALL be bounded by **explicit counters in the extraction wrapper** — decompressed-total-bytes, entry-count, nesting-depth — independent of any lane's memory ceiling (a memory limit does not catch a bomb decompressing to Lane B's scratch/disk surface). A breach SHALL quarantine `decompression-bomb`. *(Binary-observable: QA-5 — a decompression bomb is killed within the counter budgets and quarantined `decompression-bomb`; host memory/CPU stay within the P-0007-class budgets — [acceptance test].)*
- **R-0115-c (per-transform execution budget, per lane):** Lane A SHALL ride [P-0007](../src/adrs/P-0007-plugin-resource-limits.md) mechanisms (fuel + epoch + memory, kill-and-replace) with **extraction-calibrated** numeric bounds (§ Numeric calibrations) — P-0007's plugin-calibrated V0 values (64 MiB / 10B fuel / 5 s) are NOT inherited. Lane B SHALL name its concrete bound set — `RLIMIT_AS` + `RLIMIT_CPU` + `RLIMIT_FSIZE` + wall-clock `SIGKILL` + a bounded asynchronous output read (the host reads child output async under a hard timeout, so a wedged child filling its stdout pipe cannot deadlock the host reader) — and SHALL assert the no-host-hang property Lane A gets from P-0007 as an explicit Lane-B observable. The two lanes are NOT bound-equivalent (no fuel determinism, no pool-slot poison-and-replace in Lane B) — recorded, not papered. *(Binary-observable: a Lane-A extraction exceeding its fuel/epoch/memory budget is killed-and-replaced within budget; a wedged Lane-B child filling stdout is `SIGKILL`ed at the wall-clock bound without hanging the host reader — [acceptance test].)*
- **R-0115-d (Lane-B disk expansion + quarantine aggregate — the two at-spec placements the Frame names):** The spec SHALL bound Lane B's scratch/disk expansion (`RLIMIT_FSIZE` or a bounded scratch write) and SHALL place the aggregate quarantine size/count bound with its eviction/rotation policy (R-0112-d). Scan batching SHALL bound the arrival rate structurally (reconciliation-first coalesces event floods — R-0101-b). *(Binary-observable: a Lane-B extractor writing past the scratch bound is killed; the quarantine aggregate bound has an enforced eviction/rotation policy — [acceptance test].)*

## Scenarios

Given/When/Then for the behaviors this spec locks. Each maps to its Frame QA row and the R-IDs it exercises.

### Scenario: watched-corpus arrival, staged and canonicalized (R-0100/R-0102/R-0104/R-0110; QA-1; SC4/SC7)

**Given** a registered `first-party` markdown source root and a running watcher
**When** a new markdown note lands under the root
**Then** the loader stages a raw artifact with a [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md)-envelope provenance stamp, enqueues a transform job, and the transform commits a canonical row carrying `derived_from`; the `arrival→staged` and `staged→canonical` spans are recorded within their stated bounds.

### Scenario: missed-arrival catch-up on restart (R-0101; QA-2; SC11)

**Given** the watcher is stopped
**When** files are added/modified under a registered root and the service then restarts
**Then** the startup reconciliation scan stages the missed content with no re-drop and zero manual repair.

### Scenario: kill-mid-operation determinacy (R-0112; QA-3; SC9)

**Given** the pipeline is mid-load / mid-transform / mid-scan
**When** the process receives `SIGKILL`
**Then** no indeterminate state remains; residue is inert; a re-run converges, with no partial canonical rows and no manual repair.

> This bundles **three materially distinct recovery mechanisms** and a red-phase test-writer SHALL split it into three (one scenario per interrupt point, distinct Then): **mid-scan** → the next scan re-observes the source; **mid-load** → the staging transaction never half-commits and the next scan re-observes it; **mid-transform** → the job lease expires, at-least-once re-delivery fires, and the idempotent handler converges (and a kill *after* dead-letter but before quarantine resolves via the R-0101-f third join — see the dedicated crash-window scenario below).

### Scenario: crafted-payload extraction produces no egress (R-0105; QA-4; fold M3)

**Given** a zero-egress configuration and a crafted DOCX carrying an external-entity payload
**When** the file is ingested and extraction runs
**Then** extraction produces no outbound connection (structural egress-deny plus XXE-safe-by-omission parsers); the content is extracted or quarantined `unparseable-binary`; the zero-egress config's guarantee holds through ingest. *(Only the **zero-egress** property is the load-bearing assertion; extraction-vs-`unparseable-binary` is a legitimately-either outcome a test SHALL NOT pin.)*

### Scenario: fail-closed extraction terminal on a host without the Lane-B enclosure (R-0105-c; QA-4b; Frame §15 item 4)

**Given** a native (non-containerized) host on which Lane B's structural egress-denying enclosure is not verifiably present (e.g. a native macOS dogfood host)
**When** a Lane-B-only format (PPTX, or DOCX if its spike fails) is ingested
**Then** the format quarantines `unextractable-here` and runs no extractor — never an unsandboxed extraction path; the exposure is availability (the format is unavailable on that host), never egress.

### Scenario: staging→jobs orphan re-enqueue (R-0101-d/R-0112-b; FIV-1)

**Given** a staging row committed in `status = staged` whose post-commit best-effort enqueue did not land a transform job
**When** the reconciliation scan runs and the row is past the enqueue-orphan window
**Then** the scan re-enqueues the orphan and the row transforms to `status = transformed`; a staged row that already has a live or completed job is not double-enqueued.

### Scenario: resource-bound rejection (R-0115; QA-5)

**Given** a decompression bomb or an oversized file arrives
**When** it is admitted
**Then** it is refused pre-read (size) or killed in-sandbox within the counter/lane budget (expansion) and quarantined with its distinct designed class; host memory/CPU stay within the P-0007-class budgets and the pipeline continues.

### Scenario: re-derivation after extractor upgrade, arrival source gone (R-0102/R-0104; QA-6; SC3)

**Given** a binary retained in staging whose original drop location has since been deleted, extracted by a `declared_deterministic: true` transform
**When** the extractor is upgraded and re-derivation runs
**Then** canonical content re-derives from the staging binary alone at the same canonical id (update-with-audit), with no re-fetch of the arrival source; the re-derivation **appends a [P-0001](../src/adrs/P-0001-storage-layout.md) system-versioned history record** (R-0104-c), and the seeded byte-compare re-run over the unchanged raw passes (R-0104-e, since the fixture declares `deterministic: true`).

### Scenario: load-shedding under arrival burst (R-0111; QA-7)

**Given** an arrival burst exceeding transform throughput
**When** the pressure is sustained
**Then** staging latency stays within its bound while `staged→canonical` degrades measurably (the deliberate, observable trade); the backlog is visible in the queue-depth gauge.

### Scenario: external-class trust integrity (R-0108; QA-8; PE-7)

**Given** a registered `external`-class source whose ingested frontmatter asserts `supersedes` a canon artifact *(testable at V0.1 today: `trust_class = external` is a registerable enum value on any driver including the in-scope watched-markdown driver — R-0099-d/R-0108-a — so this drives through the existing watcher with no clipper/webhook infra; the parked external-class connectors of Non-goal 5 are not needed)*
**When** it is staged and its declared relation is written
**Then** the relation records as an `origin = declared` edge via the ES-6 sanctioned path, and the trust predicate — keyed on the declaring artifact's `external` trust class (`{{P-0015-A1-ingest-trust-class}}`) — does not fire; the target's trust is unaffected; the curatorial signal fires as the R-0108-f mechanism (`curatorial_signal_count` increments by one on the run record **and** one redacted named-level log line is emitted); no trust flip without a PE-6 promotion.

### Scenario: staging↔dead-letter crash-window reconciliation (R-0101-f/R-0104-f; the B1≡M2 poison-loop closure)

**Given** a staging row in `status = staged` whose transform job has reached terminal `dead_letter` (retries exhausted) but whose R-0104-f staging-quarantine write was lost to a `SIGKILL` before it committed
**When** the reconciliation scan runs
**Then** the scan reads the row as an interrupted terminal-failure (a single `dead_letter` job, no live/completed job) and drives it to `status = quarantined`, `rejection_class = transform-failure` — **not** a re-enqueue; a second scan over the now-`quarantined` row makes no further change (idempotent); the poison message never re-enters a fresh retry cycle, so R-0112-c working-state → working-state holds across the crash-window.

### Scenario: canonical-id one-to-many, N outputs stable across re-runs, and retire-on-shrink (R-0104-d/-e/-g; F2+M1)

**Given** a `first-party` binary (a PDF/DOCX) whose extraction transform emits **three** logical outputs, minting three distinct `output_key`-discriminated canonical ids {a, b, c}
**When** the transform is re-run at a higher version over changed content that now emits only {a, b}
**Then** the three ids are stable across the content-changing re-run (a and b resolve the **same** ids, no new id minted for changed content — R-0104-d); the vanished `output_key` c is **retired** — soft-deleted / marked superseded via system-versioned history, no longer served, id not re-minted (R-0104-g); and a subsequent delete-derived → re-run reconstructs exactly {a, b} from staging alone (SC3), including where the transform declares `declared_deterministic: false` (whose *content* may vary run-to-run while its *identity set* {a, b} does not — R-0104-d/-e).

### Scenario: retire-on-shrink under drop-and-add — the non-subset re-run (R-0104-g; H1)

**Given** the same `first-party` binary whose extraction transform emitted three `output_key`-discriminated canonical ids {a, b, c}
**When** the transform is re-run at a higher version whose output **drops** `c` and **adds** a new `output_key` `d` — new set {a, b, d}, which is **not a subset** of the prior {a, b, c}
**Then** a and b are updated in place at their same ids (R-0104-d); the dropped `output_key` c is **retired** — soft-deleted / marked superseded via system-versioned history, no longer served, id not re-minted (R-0104-g); and the added `output_key` d **mints a new canonical id** — all in the one re-derivation transaction; and a subsequent delete-derived → re-run reconstructs exactly {a, b, d} from staging alone (SC3).

### Scenario: tenant-isolation negative test — the single construction-site chokepoint (R-0107-b; §5 RLS; Integrity priority #1)

**Given** the ingest system principal constructed at its single `WorkspaceCtx` site, bound to source S's registered `workspace_id = W`
**When** a seeded write attempts to commit a row whose `workspace_id ≠ W`
**Then** the write is **rejected** at the construction-site chokepoint (the cross-workspace-write-capable principal cannot commit outside its source's registered workspace); the negative test the spec commits to rendering is satisfied.

### Scenario: admission-control CLI — control-plane authorization and registration allow-lists (R-0099-c/-d; TB-3)

**Given** the admin CLI control-plane and a `read_observer` token
**When** the `read_observer` token invokes any ingest control-plane operation (`source register`/`list`/`retire`, `quarantine list`/`inspect`)
**Then** it is refused at the host-fn boundary, and the attempt is audited.
**And when** `source register` is invoked with (i) `trust_class = "trusted"`, (ii) a non-directory root, or (iii) a root outside every configured allowed prefix
**Then** each is refused with a structured error naming the violated allow-list rule; a well-formed registration succeeds and its row carries the admin-attested `trust_class`.

### Scenario: watcher path-containment — symlink, FIFO, traversal (R-0106-c; TB-3; groups F1's `path-traversal-name`)

**Given** a registered root and a running watcher
**When** the scan encounters (i) an in-root symlink pointing outside the root, (ii) a FIFO under the root, and (iii) a `../`-bearing candidate path
**Then** (i) the symlink target is **not opened or staged** (open-time containment, not check-then-open); (ii) the FIFO is **skipped, not read** (`S_ISREG`-only), and the `skipped` run-record counter increments so the skip is a positive trace; (iii) the traversal path is **refused `path-traversal-name`** pre-staging. No special file is ever read as content.

### Scenario: per-class rejection paths — `malformed-file`, `unregistered-source`, `transform-failure` (R-0112-a/R-0104-f; F1)

**Given** the admission and transform paths
**When** (i) a structurally malformed markdown/frontmatter file is admitted, (ii) content is presented for a path under no registered source, and (iii) a transform is seeded to fail past its retry budget
**Then** (i) quarantines `status = quarantined`, `rejection_class = malformed-file` with a diagnostic and no partial write; (ii) is **refused pre-staging** `unregistered-source` (nothing to stage); (iii) leaves the staging row `quarantined`, `rejection_class = transform-failure`, **zero canonical rows** (steady-state — distinct from the crash-window third-join scenario above). Each rejection class is a distinct designed error with its own error-path test.

### Scenario: dedup no-op is a recorded arrival event (R-0109-b; TB-6)

**Given** a file already staged at its identity `(workspace_id, source_id, source_path, content_hash)`
**When** the identical file is re-dropped at the same path
**Then** **no new staging row** is written but **an arrival-event record is written** (suppression is visible, never silent — TB-6); and the **same content at a different path** stages a **second, distinct** row (path-scoping bounds dedup poisoning — a planted duplicate cannot suppress a legitimate arrival elsewhere).

### Scenario: telemetry redaction (R-0110-c; Confidentiality)

**Given** a fixture file at a known path with a known frontmatter title
**When** ingest emits a telemetry/run-record line for it
**Then** the emitted line contains the **path's hash, not the path**, and **no frontmatter title verbatim** (derived IDs only); the stored run-record row MAY hold the real path (admin-read), but the emission does not.

### Scenario: quarantine aggregate bound + metadata-only inspection (R-0112-d/R-0115-d; TB-2)

**Given** the quarantine store at its aggregate bound
**When** a malformed-file flood pushes past the size/count aggregate bound
**Then** the **eviction/rotation policy** (oldest-first) fires rather than an unbounded write; and `quarantine inspect` returns **metadata only** (path, `rejection_class`, diagnostic) with **no payload body** — any payload retrieval is an explicit operator action re-entering the sandboxed extraction path.

### Scenario: Lane-B no-host-hang — a wedged child filling stdout (R-0115-c; explicit Lane-B observable)

**Given** a Lane-B extraction whose child process wedges and fills its stdout pipe (a non-terminating failure distinct from a decompression bomb or an oversized file)
**When** the wall-clock bound elapses
**Then** the child is `SIGKILL`ed at the wall-clock bound and the host reader (bounded asynchronous output read) **does not deadlock** — the no-host-hang property Lane A gets from P-0007 is met explicitly by Lane B.

### Scenario: mtime-evasion caught by the full-content rehash (R-0101-c; Frame §5 residual 6)

**Given** a file already staged, then edited in place at the **same size with its mtime reset** to the prior value
**When** a pre-filter-only scan runs, then the next full-content-rehash pass runs
**Then** the edit is **not** staged on the pre-filter-only scan (the size/mtime fast-path is evaded) but **is** staged on the full-rehash pass — bounding the evasion to one rehash-cadence window.

## Threat model

The formal typed DFD (EE-/P-/DS-/DF- typed anchors + trust-boundary subgraphs) is authored in **[P-0024](../src/adrs/P-0024-ingest-pipeline-shape.md) § Typed DFD** (Frame §9 routes the artifact there); this section carries the attack-surface → owning-control map so the spec's security surface is legible without leaving it. Attack surface, each with its owning control: extraction outbound fetch / XXE (structural egress-deny both lanes + XXE-safe-by-omission parsers — R-0105-a/-e); resource exhaustion at intake (size cap pre-read + decompression counters + per-lane execution budgets — R-0115); path traversal / symlink escape on watched roots (TOCTOU-safe open-time containment + `S_ISREG`-only scan — R-0106-c); fs-write-on-a-watched-root as silent write-through (admission is admin-attested registration with an attested `trust_class` — R-0099-c/-d, TB-3); adversarial corpus text flipping trust (trust-predicate keying on the stored `trust_class`; `extracted` never enters a trust predicate — R-0108-c, PE-7); standing-service authority (least-authority system principal, writes only through the seam, no edge-write capability — R-0106-b/R-0107); dedup poisoning (path-scoped BLAKE3 identity; observable no-ops — R-0109); cross-workspace write (single construction-site invariant + the negative test — R-0107-b); quarantine as an attacker-controlled unbounded write (aggregate bound + metadata-only inspection — R-0112-d); poison-message re-processing (a planted document whose transform permanently fails, dead-lettered then resurrected into an infinite retry cycle by the orphan sweep — closed by counting `dead_letter` in the re-enqueue guard and completing the stranded staged row to `quarantined` via the R-0101-f staging↔dead-letter reconciliation — R-0101-d/-f); ingested-content-as-injection-payload-to-downstream-agents (named, deferred to the D3 admission hook — R-0100-c, out of scope). Residual risks are the Frame §5 set, accepted at single-operator dogfood scope with named re-open conditions — carried, not re-litigated.

## Numeric calibrations (spec-tier; dogfood-scale defaults; config-tunable)

Each default below is config-backed and validated at startup; the **existence and enforcement** are the lock (the requirement clause), the number a tunable default sized for single-operator dogfood scale. A test environment running shortened/relaxed values is a supported configuration (the ACs that wait out a cadence run at CI speed against the configured test value, never a wall-clock sleep).

| Calibration | Default (dogfood) | Requirement | Rationale |
|---|---|---|---|
| Reconciliation scan interval | 60 s | R-0101-e | Primary arrival mechanism; sized under the `arrival→staged` recency bound. Events accelerate; the scan is the correctness floor. |
| Event→scan debounce/coalescing window | 2 s | R-0101-b | Coalesces event floods into scan batches; a touch-flood resolves to one scan observation. **Doubles as the identical-re-arrival suppression window at V0.1** (at most one arrival-event row per identity per window — F11), sized so the two never disagree at dogfood scale; a distinct bound splits out only if a real workload separates them. |
| Full-content-rehash cadence | 24 h | R-0101-c | Bounds the size/mtime-fast-path evasion window (a size-preserving edit with reset mtime) to one cadence period. The pass is **I/O-paced** by a per-pass work budget (a tunable; existence of pacing is the lock) so its bulk read load does not contend with the `arrival→staged` critical path on a larger corpus (B6). |
| Enqueue-orphan window (staging→jobs re-enqueue) | 2 × scan interval | R-0101-d | A staged row with no job past this window is an orphan the scan re-enqueues; wide enough that a normally-enqueued job is never falsely swept. |
| `arrival→staged` recency bound | 90 s | R-0110-d | Arrival-detection-to-durable-staging target; the QA-1 observable. |
| `staged→canonical-committed` recency bound | 5 min | R-0110-d | Staging-to-canonical target under normal (non-shedding) load. |
| Per-file size cap (pre-read) | 256 MiB | R-0115-a | Above it → `oversized-input`, never read into memory. Sized to the largest routine corpus document. |
| Decompression budget (total bytes / entry-count / nesting-depth) | 1 GiB / 10 000 / 16 | R-0115-b | Explicit counters in the extraction wrapper, independent of any lane memory ceiling. Breach → `decompression-bomb`. |
| Lane-A extraction budget (fuel / epoch / memory) | 60 s / 50B fuel / 512 MiB | R-0115-c | Extraction-calibrated, NOT P-0007's plugin values (5 s / 10B / 64 MiB) — large-document parsing needs its own numbers. |
| Lane-B budget (`RLIMIT_AS` / `RLIMIT_CPU` / wall-clock `SIGKILL` / `RLIMIT_FSIZE`) | 512 MiB / 60 s / 90 s / 512 MiB | R-0115-c/-d | The native lane's non-fuel bound set; the wall-clock `SIGKILL` + async bounded output read give the no-host-hang property. |
| Job retry budget / backoff | 5 attempts / exponential | R-0103-a | Exhausted retries → dead-letter; a transform failure quarantines `transform-failure` after exhaustion (R-0104-f). |
| Job lease / visibility window | 120 s | R-0103-b | Must **exceed the longest lane's wall-clock bound (Lane B, 90 s) plus margin** so a still-running job is never re-delivered to a second claimant while the first is working; an abandoned `claimed` job recovers when this expires. |

## Intent self-report
