# Spec: Translation pipeline — EXPLAIN/STRIP passes + hash-gated regeneration

## Goal

Add a publish-time translation pipeline that produces a human-form tree (`docs/_published/human/`) and an agent-form tree (`docs/_published/agent/`) from `docs/src/`. Each page's *opposite-audience* form is generated by an in-session Agent-tool dispatch against a prompt at `docs/prompts/`; the *canonical-audience* form is a verbatim copy. Translation is hash-gated via `docs/_published/.translation-manifest.json` so unchanged pages skip the LLM call. `just docs-check` enforces non-stale.

**Pipeline shape.** Three surfaces collaborate:

- **`scripts/docs-translate.py`** — deterministic gating (hashing, manifest IO, verbatim copies, orphan cleanup, plan/finalize/check modes). No LLM calls.
- **`docs/prompts/explain-pass.md` + `docs/prompts/strip-pass.md`** — prompt content (the substantive translation rules).
- **`.claude/commands/docs-translate.md`** — slash command Peter runs inside a Claude session. Invokes the script in plan mode, dispatches one Agent per stale page (parallelizable), writes returned translations to disk, then invokes the script in finalize mode to update the manifest.

After this step:

- `mdbook build` still reads `docs/src/` (step 7 re-points it at `_published/human/`; out of scope here).
- `scripts/docs-llms.py` switches its default source root from `docs/src/` to `docs/_published/agent/` so `llms.txt`/`llms-full.txt` reflect the agent-form. The format contract from step 5 is unchanged.
- G-0029's "Mechanism details (TBD placeholder)" section gets updated with the resolved mechanism (separate commit after the pipeline stabilizes; in-scope for this step's closing work).

## Out of scope (do not extend into these)

- Re-pointing `book.toml` at `_published/human/` — step 7.
- Pre-push git hook running `just docs-check` — step 9 (optional).
- Workspace promotion of the slash command + skill — step 8, gated on a second adopter.
- Structured-delta glossary form — gated on the structured-delta tooling, not this step.
- `llms-ctx.txt` / `llms-ctx-full.txt` variants — deferred.
- `robots.txt` / `sitemap.xml` — deferred.
- LLM-output schema validation beyond non-empty-content — deferred.
- A pure-CLI `just docs-translate` that performs translations without a Claude session — out of scope by design (see "Why dispatch, not subprocess" in Risks).

## Acceptance criteria

1. **Prompts exist at `docs/prompts/`.** Two files, hand-authored in this step:
   - `docs/prompts/explain-pass.md`
   - `docs/prompts/strip-pass.md`

   Both are plain Markdown prompts (no frontmatter). Each MUST contain the literal tokens `{{GLOSSARY}}` and `{{PAGE}}` exactly once each — these are the substitution points. The script substitutes glossary content and page content into these tokens before emitting a plan entry. Initial prompt content is authored by Puck during spec preparation and committed as part of this step; Peter iterates after first end-to-end run.

2. **Generator exists at `scripts/docs-translate.py`.** Python script invoked via `uv run`. Three modes:
   - `--plan`: compute the work needing translation; emit a JSON plan to stdout; perform deterministic filesystem work (verbatim copies of canonical-side pages, orphan file deletions); write a pending sidecar at `docs/_published/.translation-pending.json` describing the work the slash command must complete and the orphan manifest entries to remove. **Does NOT mutate the manifest.** NEVER calls an LLM. NEVER writes translated outputs.
   - `--finalize`: validate that the slash command produced every translated output the pending sidecar lists; apply manifest mutations atomically — add/update entries for newly-translated pages AND remove entries for orphans listed in the sidecar. Delete the pending sidecar on success. Errors if any output is missing or empty (manifest and sidecar both unchanged on error).
   - `--check`: read-only inspection. No copies, no orphan removal, no sidecar writes, no manifest mutation. Exits 0 if the manifest matches reality; exits 1 with a drift report to stderr otherwise.

   **Manifest mutation discipline:** ONLY `--finalize` writes the manifest. `--plan` and `--check` are manifest-readers.

3. **Audience routing.** For each `docs/src/**/*.md` (excluding `SUMMARY.md`):
   - Agent-primary pages: `_published/agent/<path>` is a verbatim copy of source; `_published/human/<path>` is the EXPLAIN-pass output.
   - Human-primary pages: `_published/human/<path>` is a verbatim copy of source; `_published/agent/<path>` is the STRIP-pass output.
   - Pages with no `primary-audience` in frontmatter: exit 1 naming the page. No default-by-directory inference inside the generator (required-field discipline per `feedback_validatability_lens_schema_design.md`).

4. **`SUMMARY.md` special case.** Treated as structural, never translated. Copied verbatim to both `_published/human/SUMMARY.md` and `_published/agent/SUMMARY.md` so each subtree is internally navigable. No frontmatter required on SUMMARY.md.

5. **`.translation-manifest.json` schema.** Located at `docs/_published/.translation-manifest.json`. Structure:

   ```json
   {
     "schema_version": 1,
     "entries": {
       "<relative-path-from-docs-src>": {
         "source_sha256": "<64-hex>",
         "prompt_path": "explain-pass.md|strip-pass.md",
         "prompt_sha256": "<64-hex>",
         "primary_audience": "agent|human",
         "translated_at": "<ISO-8601 UTC, e.g. 2026-05-21T12:34:56Z>"
       }
     }
   }
   ```

   Field-by-field mechanical validators (each enforced by `--check` / `--finalize` and at write-time):
   - `schema_version`: integer, exactly `1`. Wrong value → exit 1, message names the file and expected version.
   - `entries.<path>.source_sha256`: 64-char lowercase hex. SHA-256 of the source file's full bytes (frontmatter included). Translation does NOT write back to source, so no self-loop.
   - `entries.<path>.prompt_path`: literal string `"explain-pass.md"` or `"strip-pass.md"`. Records *which* prompt produced this entry. Any other value → exit 1.
   - `entries.<path>.prompt_sha256`: 64-char lowercase hex. SHA-256 of the prompt file named by `prompt_path`. A prompt edit invalidates every entry that used it.
   - `entries.<path>.primary_audience`: literal string `"agent"` or `"human"`. Must agree with `prompt_path` (`agent` → `explain-pass.md`, `human` → `strip-pass.md`). Disagreement → exit 1.
   - `entries.<path>.translated_at`: parseable as ISO-8601 UTC with trailing `Z`. Used for human inspection of the manifest, not for gating decisions.

6. **Hash-gated regeneration.** On `--plan` (and `--check`):
   - Compute current `source_sha256` for each `docs/src/**/*.md` (excluding `SUMMARY.md`).
   - Compute current `prompt_sha256` for both prompt files.
   - For each page: if manifest entry exists AND `source_sha256` matches AND the relevant `prompt_sha256` matches AND both `_published/{human,agent}/<path>` exist → SKIP. Otherwise → include in plan.

7. **Plan JSON contract.** `--plan` emits JSON to stdout with this shape:

   ```json
   {
     "schema_version": 1,
     "items": [
       {
         "source_path": "docs/src/intent/mnemra-core.md",
         "output_path": "docs/_published/human/intent/mnemra-core.md",
         "primary_audience": "agent",
         "prompt_path": "docs/prompts/explain-pass.md",
         "source_sha256": "<hex>",
         "prompt_sha256": "<hex>",
         "assembled_prompt": "<full prompt text, with {{GLOSSARY}} and {{PAGE}} already substituted>"
       }
     ]
   }
   ```

   `assembled_prompt` is the substance the slash command hands to the dispatched Agent — the script does the substitution so the slash command has no templating responsibility. Empty `items` array means nothing to translate.

8. **Slash command at `.claude/commands/docs-translate.md`.** Plain-text instructions (no executable code itself); the body tells Claude to:
   1. Run `uv run scripts/docs-translate.py --plan` via Bash, capture stdout, parse JSON.
   2. If `items` is empty: report "no translation needed" and stop.
   3. For each item: dispatch an Agent (`subagent_type: general-purpose`, `mode: default`) whose prompt is the item's `assembled_prompt`. **Parallelism is mandatory and explicit:** dispatch agents in batches of 4 by sending exactly 4 Agent tool calls in a single message — Claude only parallelizes when multiple Agent calls share a single tool-use message. Serial dispatch (one Agent call per message) is a defect. Each Agent's job is to return the translated Markdown as its final message; the slash command's parent Claude session writes the result to `output_path` using the Write tool after the batch returns.
   4. After all items complete: run `uv run scripts/docs-translate.py --finalize` via Bash. If finalize exits non-zero: report which outputs are missing and stop without manifest update.
   5. On success: report a one-line summary (`N pages translated, manifest updated`).

   The slash command frontmatter is standard `description:` only. No `argument-hint` needed (no args in v1). The slash command body MUST contain the explicit "4 Agent calls per message" parallelism instruction verbatim — under-specifying produces serial execution.

9. **Agent dispatch contract.** Each Agent dispatched by the slash command:
   - Receives the assembled prompt as its task input.
   - Has NO Write/Edit access (mode: default, no bypassPermissions). The Agent's output is text only — the parent session does the file write.
   - Must return ONLY the translated Markdown body, no preamble, no commentary. Enforced by prompt content + a contract line in the slash command.
   - If the Agent returns empty content or content starting with explanation prose: the parent session retries once with a stricter prompt prefix. Second failure → report the failing page and abort the run (parent re-runs `--finalize` with the missing output absent, which surfaces it as drift).

10. **Glossary injection.** The full text of `docs/src/glossary.md` (stripped of its frontmatter) is substituted into `{{GLOSSARY}}` in both prompts on every `--plan` invocation. No per-page glossary scoping in v1. The `prompt_sha256` field hashes only the prompt file itself — not the assembled prompt — so glossary edits do NOT invalidate translations. (Tradeoff accepted: glossary edits propagate only when the source page itself changes. If this proves too lax, step 8 or a separate amendment can add `glossary_sha256` as a sibling field.)

11. **Orphan cleanup.** When `docs/src/X.md` is deleted, the next `--plan` (or `--check`) run:
   - Removes `_published/human/X.md` and `_published/agent/X.md` if they exist (`--plan` only; `--check` reports).
   - Records the orphan in the pending sidecar (`--plan` only); the corresponding manifest entry is removed by `--finalize` per the single-writer invariant (AC #5).
   - Prints a one-line stderr notice (`removed orphan: <path>`).

    `--check` reports orphans as drift (exits 1 with stderr listing each orphan).

12. **`docs-llms.py` source root flip.** `scripts/docs-llms.py` changes its default source root from `docs/src/` to `docs/_published/agent/`. The format contract (`llms.txt` shape, SUMMARY ordering, `summary:` frontmatter source) is identical; only the input tree changes. Step-5 tests parameterize source via `--src`, so they stay green; the script's default-source-root change is the only material edit.

13. **`just docs-check` recipe.** Added to `justfile`:
    ```just
    docs-check:
        uv run scripts/docs-translate.py --check
        uv run scripts/docs-llms.py --check
    ```
    Runs both drift gates. Either failing exits the recipe.

14. **`just check` integration.** Existing `check` recipe gains `uv run scripts/docs-translate.py --check` (added before the existing `docs-llms.py --check` line so translation drift surfaces first).

15. **No `just docs-translate` recipe.** Translation requires a Claude session; the entry point is the `/docs-translate` slash command, not a justfile recipe. A `just docs-translate` recipe would imply CLI-callability and mislead. Document this in the `justfile` itself as a comment:
    ```just
    # Translation runs from a Claude session via /docs-translate.
    # See .claude/commands/docs-translate.md.
    ```

16. **`.gitignore` update.** Add `docs/_published/.translation-pending.json` to `.gitignore` (it's ephemeral, removed by `--finalize` on success; mid-run state should not be tracked). Confirm `docs/_published/` is NOT otherwise excluded; `docs/_published/.translation-manifest.json` IS tracked.

17. **CI workflow unchanged.** `.github/workflows/docs.yml` stays as-is. The `mdbook build` still reads `docs/src/` (re-pointing is step 7). The `Stage llms.txt + llms-full.txt` step still copies from `docs/_published/`. No new CI step in this PR.

18. **End-to-end first run produces a complete tree.** After implementation, Peter runs `/docs-translate` in a Claude session at the mnemra-core repo root. All 7 current pages produce both `_published/human/` and `_published/agent/` entries. `just docs-check` exits 0. `just check` exits 0. PR includes the regenerated `_published/{human,agent}/` tree plus the new manifest.

## Prompt structure (initial drafts authored by Puck pre-Forge)

Both prompts share an XML-tagged skeleton (per Anthropic's prompt-engineering best practices; XML boundaries disambiguate prompt frame from Markdown content the model processes):

```
<role>
  Role framing.
</role>

<task>
  Task framing. References <glossary> and <source-page>.
</task>

<rules>
  - EXPLAIN-specific or STRIP-specific transformation rules.
</rules>

<prose-style>  <!-- EXPLAIN only -->
  - Anti-AI-writing constraints for human-facing prose generation.
</prose-style>

<output-format>
  Emit only the translated Markdown. No preamble, no trailing commentary.
</output-format>

<glossary>
{{GLOSSARY}}
</glossary>

<source-page>
{{PAGE}}
</source-page>
```

The `{{GLOSSARY}}` and `{{PAGE}}` tokens still appear exactly once each per AC #1. The XML tags are part of the prompt file contents; the script does not strip or transform them.

**Asymmetry between EXPLAIN and STRIP:** EXPLAIN's `<prose-style>` block injects the workspace's `skills/no-ai-writing.md` core constraints (em-dashes, filler transitions, inflated vocabulary, marketing phrases, burstiness, triple-lists, "phrase: comma-list" shorthand) because the output goes to human readers who notice AI-flavored prose. STRIP omits the block because the output is consumed by LLM agents reading `llms-full.txt`, where AI-detection concerns do not apply; STRIP's existing rules already handle the relevant tightening for agent consumption. Initial prompts may grow to ~55 lines (EXPLAIN) and ~30 lines (STRIP); iteration after first run is expected.

Specific rules per pass (encoded in the prompt files):

**EXPLAIN pass** (agent-primary → human form):
- Resolve workspace shorthand (ADR codes like `G-0027`, principle codes like `P-Defer`, lifecycle tiers, R-codes) by reference to the glossary; first use in a page gets an inline gloss or a glossary link, subsequent uses can stay terse.
- Add narrative connective tissue where source jumps between dense claims.
- Preserve frontmatter verbatim (translation does NOT re-author `title:`, `summary:`, `primary-audience:`).
- Preserve all Markdown structural elements (tables, code blocks, ADR cross-reference paths).
- Do NOT add new sections; do NOT remove existing sections.

**STRIP pass** (human-primary → agent form):
- Remove pedagogical voice ("let's", "you'll see", "here we'll").
- Tighten prose to facts and contracts.
- Add explicit cross-references to ADRs/specs by file path where the source narrative implies but does not name them.
- Preserve frontmatter verbatim.
- Preserve all Markdown structural elements.

The initial prompts are intentionally short — under 40 lines each. Iteration after first end-to-end run is expected and budgeted; prompt edits trigger a full retranslation via the `prompt_sha256` change.

## Generator implementation notes (non-binding guidance — Forge may differ)

- Script header: `# /// script\n# requires-python = ">=3.11"\n# dependencies = ["pyyaml"]\n# ///` (same uv-script inline metadata pattern as `docs-llms.py`).
- CLI: `python scripts/docs-translate.py {--plan | --finalize | --check} [--src docs/src] [--out docs/_published] [--prompts docs/prompts]`. Exactly one of `--plan` / `--finalize` / `--check` is required (mutually exclusive).
- Module layout: keep it single-file; mirror `docs-llms.py`'s style.
- Hashing: `hashlib.sha256(path.read_bytes()).hexdigest()`. Lowercase hex by default — assert in tests.
- Manifest read: `json.loads(path.read_text())`. Manifest write: `json.dumps(..., indent=2, sort_keys=True) + "\n"` for stable diffs.
- Frontmatter parsing: reuse the same regex pattern as `docs-llms.py` (`^---\n(.*?)\n---\n`, `re.DOTALL`); use the FIRST frontmatter block only (handles `adrs/template.md`'s second-block embedded MADR template correctly).
- Plan JSON to stdout; log lines to stderr — keep them separate so the slash command can `jq` stdout safely.
- `--finalize` reads the manifest's pending-write state from a sidecar `docs/_published/.translation-pending.json` written by `--plan`. The pending file lists exactly the items the plan emitted; `--finalize` reads each `output_path`, hashes it for the manifest's downstream consumers (no — actually the manifest tracks source+prompt hashes only, not output hashes; finalize just confirms the file exists and is non-empty, then updates the manifest entry and deletes the pending sidecar). On any missing/empty output, finalize exits non-zero and leaves both the pending sidecar and manifest unchanged.

## Test plan

Tests live at `tests/test_docs_translate.py`. **Glitch authors red-phase first per `feedback_test_writer_separation.md`**, then Forge implements.

Required cases (all deterministic; no LLM mocking needed because the script never calls an LLM):

1. **Manifest schema validation:**
   - Valid manifest round-trips through read/write unchanged.
   - `schema_version: 2` (or any non-1) → exit 1.
   - Missing required field on any entry → exit 1.
   - Malformed `source_sha256` (not 64 hex chars) → exit 1.
   - `primary_audience: "other"` → exit 1.
   - `primary_audience: "agent"` paired with `prompt_path: "strip-pass.md"` → exit 1.

2. **Hash-gated logic (via `--plan` output):**
   - Page with matching `source_sha256` + matching `prompt_sha256` + both published files present → NOT in plan items.
   - Page with mismatched `source_sha256` → IN plan items with correct `output_path` + `assembled_prompt`.
   - Page with mismatched `prompt_sha256` (prompt was edited) → IN plan items.
   - Page with matching hashes but missing `_published/<dir>/<path>` → IN plan items.

3. **Audience routing:**
   - `primary-audience: agent` page → source verbatim to `_published/agent/`, plan entry targets `_published/human/<path>` with `prompt_path: docs/prompts/explain-pass.md`.
   - `primary-audience: human` page → source verbatim to `_published/human/`, plan entry targets `_published/agent/<path>` with `prompt_path: docs/prompts/strip-pass.md`.
   - Missing `primary-audience` frontmatter → exit 1 naming the page.

4. **Prompt substitution:**
   - `{{GLOSSARY}}` is replaced with glossary file content (frontmatter stripped).
   - `{{PAGE}}` is replaced with page content (frontmatter included).
   - Prompt missing `{{GLOSSARY}}` or `{{PAGE}}` token → exit 1 with a message naming the prompt file.

5. **`--check` mode:**
   - Clean tree, manifest matches → exit 0.
   - Source edited since last manifest → exit 1, stderr names the stale page.
   - Prompt edited since last manifest → exit 1, stderr names the prompt + count of stale pages.
   - Missing `_published/<dir>/<path>` for a manifested entry → exit 1.
   - `--check` does NOT mutate filesystem (no copies, no orphan removal).

6. **`--finalize` mode:**
   - All planned outputs present + non-empty → manifest updated, pending sidecar removed, exit 0.
   - Any planned output missing → exit 1, manifest unchanged, pending sidecar preserved.
   - Any planned output present but empty (zero bytes) → exit 1, manifest unchanged.
   - Sidecar lists orphans to remove → manifest entries for those orphans are deleted as part of the same write transaction.

7. **Manifest single-writer invariant:**
   - `--plan` with stale source + missing manifest entry → plan emitted but manifest file contents are byte-identical before and after.
   - `--check` with any drift state → manifest file contents are byte-identical before and after.
   - Only `--finalize` mutates the manifest. Tests assert this explicitly on at least one stale-source case for each non-finalize mode.

8. **`SUMMARY.md` special case:** present in `docs/src/`, copied verbatim to both `_published/human/SUMMARY.md` and `_published/agent/SUMMARY.md` during `--plan`, never appears in manifest or plan items.

9. **Orphan cleanup:**
   - Source page deleted → `--plan` removes `_published/{human,agent}/<path>` filesystem entries + lists the orphan in the pending sidecar; manifest is unchanged by `--plan`; `--finalize` deletes the manifest entry.
   - `--check` mode reports orphans as drift (exit 1) without removing them.

10. **Property-based (one PBT, optional):** for any well-formed manifest + source-tree pair where all hashes match, `--plan` produces zero plan items. Tests the gating invariant directly. Use `hypothesis` only if it's already in dev deps; otherwise skip.

Branch-delta gate: tests cover every code path in `docs-translate.py`. No path requires a network or LLM mock per `feedback_testing_philosophy.md`.

- None new at the Python layer (`pyyaml` already pinned via `docs-llms.py` inline metadata; `hashlib`, `json`, `pathlib`, `argparse` are stdlib).

## Risks + tradeoffs

- **Why dispatch, not subprocess.** Early draft used `claude --print` as a subprocess per page. Pivoted to in-session Agent-tool dispatch for three reasons:
  1. **Billing-model alignment (2026-06-15).** Per Anthropic's published policy (<https://support.claude.com/en/articles/15036540-use-the-claude-agent-sdk-with-your-claude-plan>), `claude -p` and Agent SDK usage move off plan usage limits on 2026-06-15 and draw a separate monthly Agent SDK credit ($20 on Pro). Interactive Claude Code — including Agent-tool dispatches inside an interactive `claude` session — explicitly continues to use subscription usage limits "exactly as before." A per-page `claude -p` subprocess would drain the Agent SDK credit on every retranslation; in-session Agent dispatch stays on the interactive subscription. See [[reference-agent-sdk-credit]] in workspace memory.
  2. **Per-page startup cost.** Subprocess spawned a fresh Claude per page (~3-5s setup × 7 pages = 30s baseline). Agent dispatch in-session has near-zero spawn cost.
  3. **Workspace idiom alignment.** Slash command + Agent dispatch matches how the workspace factors LLM work elsewhere; subprocess would have been an outlier requiring its own conventions.

- **Translation is local-only.** Requires a maintainer with a Claude Code session at the repo root. CI cannot retranslate. Accepted constraint — matches G-0029 §4.

- **Prompt-edit blast radius.** A one-character change to `explain-pass.md` retranslates every agent-primary page. Intentional (prompts are part of the contract) but produces large diffs. Mitigation: prompt edits go in their own PR, not bundled with content changes.

- **Non-deterministic LLM output.** Hash gating prevents re-rolls on unchanged source/prompt, but the FIRST translation after any source change is non-deterministic. Reviewers cannot diff against a previous translation in a meaningful way for the changed pages. Accepted per G-0029.

- **Manifest as merge-conflict surface.** Two concurrent doc PRs both regenerate the manifest; merge conflict on JSON. Mitigation: `sort_keys=True` produces deterministic ordering; conflicts are mostly resolvable by re-running `/docs-translate` post-merge. Accepted.

- **Partial-run behavior.** If translation aborts mid-run (Claude session closed, Agent dispatch failure), the pending sidecar persists on disk but the manifest is untouched (per single-writer rule). Next `/docs-translate` runs `--plan` again, which rewrites the sidecar from current state — hash gating means any pages already written and unchanged are excluded from the new plan, so no work is duplicated. Wasted work is bounded by "pages whose translation was written before the abort but whose source changed in the interim" — empirically zero on a single-maintainer repo. Resume-from-sidecar is NOT implemented in v1; the abort-and-retry-from-scratch path is the supported recovery.

- **Glossary edits don't invalidate translations.** AC #10 design choice. Glossary changes propagate when the source page itself changes. If glossary churn drifts translations meaningfully, add a `glossary_sha256` sibling field in an amendment; out of scope for v1.

- **Agent return discipline.** Agents may include commentary alongside the translated Markdown despite the prompt's "no preamble" rule. One automatic retry + abort on second failure (AC #9). If this misfires in practice, tune the prompt prefix or add a post-processor (out of scope here).

- **Parent-context budget at scale.** AC #7 embeds the full assembled prompt (instructions + glossary + page content) in the plan JSON, which the parent Claude session reads to dispatch agents. With 7 pages this is well within context. At 30+ pages, the plan JSON alone could be 200K+ tokens before any dispatch happens. Revisit when the doc tree grows or at workspace promotion (step 8): possible refactors include streaming the plan (NDJSON one item per line) or having the slash command read each item's prompt from disk via a path reference rather than inlined text.

- **Subscription-vs-API switch (future work).** The (C) design assumes a Claude subscription session. Users with only an API key cannot run `/docs-translate` as-spec'd — the slash command + interactive-Claude-Code path is subscription-shaped. A future switch (config field, env var, or invocation flag) selecting between in-session Agent dispatch (subscription) and `claude -p` / Agent SDK subprocess (API key) would let the same pipeline serve both audiences. Out of scope for step 6; tracked separately. The architecture choice here doesn't block it — both modes can converge on the same plan JSON + prompts + manifest, differing only in HOW translations are obtained.

- **Python choice is provisional.** Workspace CLI standard is Rust (`brain` binary, etc.). Step 6 uses Python for symmetry with step 5's `docs-llms.py` (already shipped Python). Rewriting both as a Rust binary is known future-work tracked separately. Tolerated for now because (a) step 5 set the precedent, (b) the script is small and easily ported, (c) the upcoming `/brief` skill is intended to catch stack-choice mismatches at spec-drafting time but isn't yet in force.
