## Purpose / context

Multiple concurrent orchestrator instances operate in one workspace and cannot coordinate safely: resource claims are advisory files with no atomic acquisition and no liveness binding; handoffs carry no delivery or consumption state; every convention decays silently under agent authorship (the intake's four evidence items; P-GuaranteeByMechanism names the failure class). This spec locks the designed tier of the coordination cluster that replaces that file substrate with substrate-enforced coordination: durable actor identity, leases with exactly-one-holder atomicity and TTL/stale/takeover semantics, and addressed messages whose delivery and consumption state are facts in a store.

The closed world is the locked Frame (blob `4d1a8c2`). This spec maps Frame directions C1–C13 and trust-boundary resolutions TB-1..TB-5 to requirement slots **R-0063–R-0077** at spec precision; nothing outside the Frame's locked directions, the locked intake, and cited canon enters here. The Frame's ratified elicitation directions (its §2, maintainer-ratified 2026-07-06) are transitively off-limits to re-open. Where the Frame delegated a calibration to spec tier (TTLs, action spellings, the resource-identifier scheme, the disposition vocabulary, the founding-type schemas, the role-declaration mechanism, the SC8 dry-run flip plan, the typed DFD, log-field hygiene), this spec pins it — those pins are the Stage-3 obligations Frame §9 enumerates, discharged below.

### R-0063 — Core host residence; exactly two agent-visible tools; the surface is closed

*Locks Frame C1 + the surface half of C2. Anchors: ratified direction 1; [P-0002](../src/adrs/P-0002-core-plugin-partition.md) — the verb-on-content walk (coordination verbs are state transitions on operational substrate, not CRUD on content; actors ARE the core entity; bootstrap ordering); [P-0013](../src/adrs/P-0013-plugin-invocation-model.md) — host-served verbs are not plugin exports, the deferred-domain-verb tripwire stays unfired (the retrieval host-placement precedent); [P-0012](../src/adrs/P-0012-plugin-runtime-and-mcp-sdk.md) — the locked `rmcp` handler contract; [P-0005](../src/adrs/P-0005-v0-signing-chain.md) — satisfied vacuously-by-residence, stated. [Frame C1, C2]*

- **R-0063-a (host placement):** The coordination cluster — session plane (actor resolution + attachment), lease subsystem, message subsystem, message-type registry, observability emission — SHALL be host code inside the mnemra-core host process, beside (not inside) the retrieval subsystem. It SHALL NOT export anything through the plugin ABI: no WIT-surface change, no plugin-exported coordination verb, no P-0013 domain-verb dispatch machinery. No new signed plugin artifact is minted — the subsystem ships inside the host binary's existing trust boundary; P-0005's plugin-load verification surface is unchanged. *(Binary-observable: both tools answer on the stdio server with no plugin loaded beyond the V0 set — [acceptance test]; the WIT world and plugin export surface are unchanged by this cluster's diff — [construction + review audit].)*
- **R-0063-b (two tools, closed):** Exactly two agent-visible MCP tools SHALL ship with this cluster — **`claim`** (actions: `acquire`, `renew`, `release`, `takeover`, `list`) and **`message`** (actions: `send`, `poll`, `list`, `ack`, `disposition`) — registered on the existing single stdio MCP server as host-served verbs, exactly as the retrieval verbs are. Each action set SHALL be a closed enum in the tool schema; an out-of-enum action is refused with a structured error. Additional agent-visible coordination tools SHALL NOT ship in this cluster; in particular, **registration SHALL NOT be a third tool** — register/resolve-attach is a session-plane primitive below the tool surface (R-0064-e). *(Binary-observable: the MCP tool list contains `claim` and `message` and no other coordination-family tool; an out-of-enum action value is refused — [acceptance test].)*
- **R-0063-c (handler contract):** Both tools SHALL follow the locked handler contract: authentication check before routing; a single `WorkspaceCtx` construction site; a per-action capability check before dispatch (P-0012, [P-0006](../src/adrs/P-0006-v0-tenant-enforcement.md), [P-0009](../src/adrs/P-0009-rls-admin-token.md) verb-category gating per action — R-0073-b). *(Binary-observable: an unauthenticated coordination request is refused before any coordination code runs — [acceptance test]; every coordination host function takes `WorkspaceCtx` as its first parameter — [construction + review audit].)*
- **R-0063-d (verb-budget bound):** This cluster SHALL NOT grow the agent-visible MCP tool count beyond the two tools above. The cluster-wide budget position is recorded in [§ API Contract](#api-contract) for the committed-tier pickup; the ratified fallback if the budget tightens is collapsing `claim` + `message` into one `coordinate` tool (−1), preserved as a named option, not built. *(Binary-observable: the pinned tool census in §API Contract equals the shipped MCP tool list — [acceptance test].)*

### R-0064 — Actor grain, registration, and attachment

*Locks Frame C3. Anchors: ratified item 6 / intake forks 4+7; intake SC1; [P-0018](../src/adrs/P-0018-core-entity-manifest.md) D-ACTOR — one `actors` table, closed `actor_type`; [P-0019](../src/adrs/P-0019-plugin-contract.md) — the host-derived-context rule generalized to the acting principal; P-GuaranteeByMechanism — the type restriction is enforced, not implied; P-LockContract — the attachment invariants lock, the realization varies. [Frame C3; QA-2; TB-3]*

- **R-0064-a (role-instance = one durable actor row):** Each role-instance SHALL be exactly one durable `actors` row with `actor_type = agent`, at the P-0018 grain — no actor↔session intermediate entity. The resolution key SHALL be the role-instance identifier unique per workspace (the `actors` entity's per-workspace unique name); a session presenting a role-instance identifier SHALL **resolve-or-create** that row through the identity-builtin path (`P-builtin-agents`), never minting a duplicate for an existing identifier. The role-instance identifier SHALL be validated against **one host-registered identifier rule** at every entry point that can mint a row — the bind `poll` and `send`'s addressee resolve-or-create alike (non-empty; length-bounded; no whitespace or control characters; the exact grammar is implementation-tier within that binding shape — one rule, one validator, shared by both paths); a failing identifier is refused `invalid_role_instance` and no row is minted. *(Binary-observable: two sequential sessions declaring the same role-instance resolve the same `actor_id`; a `poll` or `send` naming an identifier that fails the rule is refused `invalid_role_instance` with no row minted; a schema/code audit finds no session↔actor join entity carrying identity — [acceptance test] + [construction + review audit].)*
- **R-0064-b (host-derived principal):** The acting principal on every coordination operation SHALL be derived from host-held attachment state. Neither tool schema SHALL carry an acting-actor parameter on any action; a caller-supplied principal SHALL NOT exist as a write input anywhere on the surface. *(Binary-observable: the tool schemas contain no acting-actor field — [acceptance test]; the coordination write path reads the principal only from session state — [construction + review audit].)*
- **R-0064-c (one live attachment; agent-type-only):** At most one live session SHALL be attached per actor row. Attach SHALL validate the resolved row's `actor_type` and refuse `human` or `system` rows (refusal `wrong_actor_type`), so a session cannot launder writes as host or human action. Attaching to an actor with a live attachment SHALL be refused mechanically (refusal `actor_live_attached`). *(Binary-observable: a second concurrent session declaring the same role-instance receives the structured refusal while the first attachment is live; an attach naming a `human` row is refused — [acceptance test].)*
- **R-0064-d (attachment liveness; succession is an audited takeover):** Attachment liveness SHALL ride the same stale/TTL discipline as leases: the attachment carries an expiry, renewed implicitly by every successful coordination operation of the attached session and by `poll`. Succession over a stale (expired) attachment SHALL be automatic at the successor's bind call and SHALL be recorded as an **audited takeover at TB-2 parity** — the audit record carries the prior attachment holder (session), the **successor session's identifier**, and the staleness/expiry evidence. A same-session re-bind after an idle-gap expiry (a session that outlived its own attachment TTL — human-paced gate waits make this ordinary, not exceptional) rides this same audited path; the record's two session identifiers being equal is the mechanical discriminator between an idle-gap resume and a genuine succession — accepted audit-surface noise at dogfood scale, filterable by construction rather than by guesswork. A live attachment SHALL NOT be succeedable. The attachment TTL is **security-load-bearing** (it sets the TB-3 impersonation-after-death window against successor-lockout); its default is pinned in §Numeric calibrations under that lens. *(Binary-observable: kill an attached session, wait past the attachment TTL (the configured test-environment value — §Numeric calibrations, test calibration), bind a successor — the successor resolves the same `actor_id` and the audit surface holds a succession record with prior-holder session, successor session, and expiry evidence; binding before expiry is refused — [acceptance test].)*
- **R-0064-e (bind moment: poll):** The canonical — and only — bind call SHALL be the `message` tool's `poll` action carrying a `role_instance` parameter (the session role-declaration mechanism, pinned: a first-call parameter, not session metadata — the smallest mechanism on the locked `rmcp` stdio surface). The host resolves, attaches (or succeeds a stale attachment per R-0064-d), audits the attach event, and returns the actor's queue plus the workspace's live leases — parity with today's session-start injection in one call. Every other action on either tool SHALL be refused with the structured refusal `not_attached` when the session holds no live attachment. *(Binary-observable: a `claim acquire` from a fresh session with no prior `poll` returns `not_attached`; the same call after a successful `poll` proceeds; a session attached as one role-instance (e.g. `merger`) polling again with a different `role_instance` (e.g. `design-lane`) is refused `attachment_mismatch` — [acceptance test].)*
- **R-0064-f (realization: invariants locked, mechanism preferred):** The four attachment invariants — one live attachment per actor; agent-type-only; stale-recoverable under TTL; succession audited with prior-holder evidence — are the locked contract. The implementation SHOULD realize attachment under the lease mechanism itself (an attachment is held intent on the role-instance; one liveness mechanism, two uses — the Frame's preferred realization, reserved `actor` resource family, barred from the entire `claim` surface per R-0067-a/-c). A dedicated attachment linkage (columns or table, [§ Data Model](#data-model)) is admissible instead, provided every R-0064 acceptance criterion passes unchanged — the ACs bind the invariants, not the realization. *(Binary-observable: the R-0064-c/-d acceptance tests pass against whichever realization ships — [acceptance test].)*

### R-0065 — Leases: the duration line; substrate-enforced atomic acquisition

*Locks Frame C4. Anchors: ratified items 3+4 (the duration-line refinement, maintainer-walked); intake SC2/SC3; P-GuaranteeByMechanism — the invariant lives in the substrate, never a convention; [P-0010](../src/adrs/P-0010-storage-substrate-engine.md) — transactions absorb atomic writes; P-LeastAuthority on holder-only mutation. [Frame C4; QA-1; QA-2]*

- **R-0065-a (the duration line):** An **atomic write** to in-system state SHALL be a storage transaction — the engine's semantics absorb it; no claim entity SHALL be involved in or created for single-transaction writes. **Held intent** — a resource held across time, inside or outside the system — SHALL be a **lease**: a first-class entity with holder (`actor_id` FK), structured resource identifier (R-0067), acquisition time, TTL/expiry, and explicit state ([§ Data Model](#data-model)). *(Binary-observable: no code path creates a lease for a single-transaction storage write — [construction + review audit].)*
- **R-0065-b (atomic acquisition; exactly one live holder):** Lease acquisition SHALL be substrate-enforced atomic — a uniqueness-constrained insert or compare-and-set inside a single transaction on the coordination store. Application-level check-then-act SHALL NOT be the mutual-exclusion mechanism. At most one live lease per `(workspace_id, resource)` SHALL be able to exist, enforced by a schema-level constraint (partial unique index or equivalent) — not by serving-path filtering. *(Binary-observable: QA-1 — a concurrent-acquisition test (two sessions, same resource, both `acquire` calls released from a shared start barrier so they reach the storage layer together — genuine contention, not scheduler-serialized calls; repeated runs) never observes two live holders — [acceptance test]; the uniqueness constraint exists in the schema — [operator SQL surface].)*
- **R-0065-c (structured refusal to the loser):** The losing contender SHALL receive a structured, machine-readable refusal (reason `resource_held`, carrying the holding actor and the lease expiry — the workspace-visible facts, per the C8 read default), never an error string, never empty-success. The refusal SHALL be logged in the operational log (R-0075-a). *(Binary-observable: QA-1 — the loser's response parses to the refusal shape with `reason_code = resource_held`; the op log holds the refusal entry — [acceptance test].)*
- **R-0065-d (holder-only renew/release; bounded requested duration):** `renew` and `release` SHALL be holder-only (refusal `not_holder` otherwise). `acquire` SHALL declare its requested duration, defaulted and bounded by the policy maximum (§Numeric calibrations); a request above the maximum is refused (`invalid_duration`). Renewal SHALL extend expiry from the renewal moment by the lease's duration; renewal applies to **live leases only** — `renew` on an expired lease (taken over or not) is refused (`lease_not_found`): an expired hold is not revivable, and the path back is the explicit, audited `takeover` (R-0066), for the prior holder and any contender alike — one recovery path, always audited. Renewal is **uncapped at V0** — a live-and-renewing holder can hold indefinitely. That absence of a total-hold bound is deliberate and named (nuisance-class at trusted single-team scope, visible in the op log); it re-examines with the force-break deferral ([§ Out of Scope](#out-of-scope)). *(Binary-observable: a non-holder `renew`/`release` is refused `not_holder`; an over-maximum `acquire` is refused `invalid_duration`, as is a zero or negative `duration_seconds` (an omitted field takes the configured default); a renewed lease's expiry moves forward; `renew` on an expired-but-untaken lease is refused `lease_not_found`; a second `release` of the same lease, a `renew` after the holder's own `release`, and a `renew`/`release` naming a fabricated `lease_id` that never existed are each refused `lease_not_found` — [acceptance test].)*
- **R-0065-e (stale semantics; no manual-wipe limbo):** An expired lease SHALL be takeover-eligible via the explicit `takeover` action (R-0066) — every dead-holder path SHALL end in a mechanical recovery rule; no coordination state SHALL require a maintainer's manual store wipe to recover. Expiry SHALL be evaluated against the stored `expires_at` at operation time, with the comparison pinned: **a lease is expired iff `now >= expires_at`** — at the boundary instant the lease is expired (takeover-eligible, not renewable), so the renew/takeover tiebreak at the instant itself resolves in takeover's favor by definition (the atomic acquisition transaction still decides any same-instant race, R-0065-b). Both sides of every expiry/TTL comparison SHALL read **one clock — the coordination store's transaction-time clock, evaluated inside the deciding transaction**; host-local clocks SHALL NOT decide expiry (a non-issue at the colocated V0 topology; pinned so a future network-separated store cannot reintroduce a skew race into exactly the guarantee this cluster makes substrate-enforced). No background reaper daemon is required at V0. *(Binary-observable: QA-2 — kill a holder, wait past TTL, a successor acquires via `takeover` with no manual intervention — [acceptance test]; the expiry predicate in code implements the pinned `now >= expires_at` comparison and reads the store's transaction-time clock — [construction + review audit].)*

### R-0066 — Takeover: a distinct, privileged, always-audited action

*Locks Frame TB-2. Anchors: P-LeastAuthority — takeover authority scoped to the recovery job, never standing break authority; intake SC3; P-Defer/DF1 on the force-break deferral. [Frame §5 TB-2; QA-2]*

- **R-0066-a (expired-only; no force-break):** `takeover` SHALL succeed only against a lease whose TTL has expired; any workspace actor may take over an expired lease. A live (unexpired) lease SHALL NOT be takeover-able through the tool surface (refusal `not_expired`); no force-break verb SHALL exist at V0. A genuinely wedged live lease waits out its TTL; an emergency break is a maintainer operation at the store, audited. `takeover` SHALL NOT be a flag or side effect on `acquire` — **unqualified**: expired-row cleanup SHALL NOT be folded into `acquire`'s transaction; recovery of an expired lease flows through the explicit `takeover` action only, one client-visible contract and one audited path (the draft's folded-cleanup admissibility was withdrawn at the r1 fold — §Decisions of note). *(Binary-observable: `takeover` against a live lease returns `not_expired`; `takeover` on a resource with no lease row at all is refused `lease_not_found`; `acquire` against an expired-but-untaken lease returns `resource_held` (carrying the holder and the already-past expiry) — the expired row is dispositioned only by `takeover` — [acceptance test].)*
- **R-0066-b (audited with evidence):** Every takeover SHALL emit to the audit surface (R-0075-b) a record carrying: the prior holder, the new holder, the lease's expiry timestamp, and the takeover timestamp (the expiry evidence — Frame routine decision 5). *(Binary-observable: QA-2 — the audit row exists with all four fields after a takeover — [acceptance test] + [operator SQL surface].)*
- **R-0066-c (deposed-holder backstop):** A deposed holder (its lease taken over after expiry) SHALL discover the deposition at its next coordination write, which fails closed per R-0074 — two actors SHALL NOT both be able to proceed as holder under any interleaving. *(Binary-observable: after a takeover, the prior holder's `renew` **and its `release`** each return a structured refusal (`not_holder`/`lease_not_found` family) — never success — [acceptance test].)*

### R-0067 — Resource identifiers: a registered structured scheme

*Discharges the Frame §9 namespace obligation. Anchors: Frame C4 (a registered structured-string scheme); P-ShiftLeft D2 — validator before field: an identifier family without a mechanical validator does not register; MB3 vocabulary consistency. [Frame C4]*

- **R-0067-a (grammar + closed V0 family set):** A resource identifier SHALL be a structured string `<family>:<qualifier>`, where `family` is drawn from a closed, host-registered set and `qualifier` is a non-empty family-validated string. The V0 family set SHALL be exactly: **`repo-lane`** (a git repository's serialized lane; qualifier = repo identifier + lane name, e.g. `repo-lane:<repo>/main-merge` — the merger lane's main-merge lease, the first designed consumer), **`file`** (a workspace file or directory path held across time), **`surface`** (a named configuration or tool surface), and **`actor`** (reserved for the R-0064-f attachment-as-lease realization; **barred from the entire `claim` surface** — R-0067-c). *(Binary-observable: an `acquire` with an out-of-family or malformed identifier is refused with `invalid_resource` naming the rule; each in-family well-formed identifier acquires; the reserved-family bar has its own per-action criteria — R-0067-c — [acceptance test].)*
- **R-0067-b (extension is a spec amendment):** Adding a resource family SHALL be an amendment to this spec (with its validator defined in the same amendment), never a config knob or a freeform passthrough. *(Binary-observable: the family set in code is a closed enum matching this list — [construction + review audit].)*
- **R-0067-c (the reserved family is unreachable through `claim` — every action, not just `acquire`):** The `actor:` family SHALL be barred from **every** `claim` action: `acquire` and `takeover` refuse on a request-side family check of `resource`; `renew` and `release` refuse on a family check of the resolved lease row (the case that exists under the attachment-as-lease realization); the refusal code is the dedicated **`reserved_family`** — distinct from `invalid_resource` (malformed / out-of-family), so a typo and a reserved-family probe are distinguishable to the caller. `claim list` SHALL NOT return `actor:`-family rows in any response, a `list` filtered to the `actor` family is refused `reserved_family`, and the `poll` live-lease listing (R-0072-a) SHALL equally exclude them — the realization choice stays invisible on the tool surface. Attachment lifecycle transitions flow **only** through the session plane — the bind `poll` and R-0064-d audited succession — never through the `claim` tool, regardless of which R-0064-f realization ships: the audited-succession-at-bind control is the only identity-slot mutation path. *(Binary-observable: `acquire`, `takeover`, and a family-filtered `list` naming `actor:<x>` each refuse `reserved_family`; under the attachment-as-lease realization, `renew`/`release` naming an attachment row's `lease_id` (obtained via the operator SQL surface in the fixture) refuse `reserved_family`, and no `claim list` or `poll` response contains an `actor:`-family lease — [acceptance test] + [operator SQL surface].)*

### R-0068 — Messages: rows; post-send immutability; append-only consumption

*Locks Frame C5's substance. Anchors: ratified item 3 (row-shape; the artifact-shape alternative rejected at ratification for its content-machinery dependency — intake Hard constraint 1); intake SC4; [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT — coordination rows are their own source of truth; P-GuaranteeByMechanism — SC4's "structurally impossible" is structural, not procedural. [Frame C5; QA-3]*

- **R-0068-a (rows in a coordination-owned table):** A message SHALL be a row in a coordination-owned table: sender `actor_id` FK, addressee `actor_id` FK, message-type name + schema-version, schema-validated JSON payload, lifecycle state, and timestamps ([§ Data Model](#data-model)). Addressing SHALL be role-instance (the actor row), so a queue survives session death exactly as leases do. *(Binary-observable: a message addressed to a role-instance whose session died is returned to that role-instance's successor session on its bind poll — [acceptance test].)*
- **R-0068-b (post-send immutability):** A message's payload, sender, addressee, type, and schema-version SHALL be immutable after send. No edit, recall, or retraction path SHALL exist at any layer — not merely no verb: no host function, no substrate-layer helper in the shipped system's runtime code mutates these fields post-insert. *(Binary-observable: the message write surface exposes insert + lifecycle-transition operations only — [construction + review audit]; QA-3 — no reachable call path mutates payload or addressing post-send — [acceptance test] over the tool surface.)*
- **R-0068-c (monotonic lifecycle; append-only consumption records):** The lifecycle SHALL be monotonic — no reverse transition, no re-transition; a disposition SHALL NOT be overwritten or reverted. Consumption records (actor + timestamp at delivery, acknowledgement, and disposition) SHALL be append-once fields: written exactly once, at their transition, never cleared. *(Binary-observable: a second `disposition` on a dispositioned message is refused with `invalid_transition`; an `ack` on an acknowledged message is refused; the stored timestamps survive every subsequent operation — [acceptance test] + [operator SQL surface]; no host function or substrate-layer helper issues a write that clears or rewrites `delivered_at`/`acknowledged_at`/`dispositioned_at` once set — [construction + review audit] (the R-0068-b split applied to the append-once half: the runnable test asserts survival across the operations it exercises; the construction audit carries the universal claim).)*
- **R-0068-d (deletion is not a lifecycle operation):** Message deletion SHALL NOT exist at V0 — deleting a pending message is retraction by another name; deleting a dispositioned one erases the consumption trail. Any future retention/archival policy is a spec amendment (or event-bus-ADR-tier work) and SHALL preserve consumption records under the same append-only shape. *(Binary-observable: no delete operation exists on either tool or on the coordination host-fn surface — [construction + review audit].)*
- **R-0068-e (messages are not content):** Message rows SHALL carry no provenance envelope, SHALL NOT be indexed or retrievable via `get_context_for` or any retrieval verb at V0 (coordination is not context — the re-open is a named tripwire, [§ Out of Scope](#out-of-scope)), and SHALL NOT reference content tables (R-0076-d). *(Binary-observable: a `get_context_for` call over a corpus containing coordination rows returns none of them — [acceptance test], runnable only where retrieval is deployed; the schema audit half rides R-0076-d.)*

### R-0069 — Message lifecycle state machine + the disposition vocabulary

*Renders Frame C5's state machine at spec precision; discharges the Frame §9 disposition-vocabulary obligation. Anchors: intake SC4 — every consumption leaves a record, every item has a disposition path; Frame routine decision 4 — ack and disposition are distinct steps (receipt vs terminal outcome); P-TrustworthySignal — delivery is a recorded fact, not an implication of send; MB3 — one closed vocabulary. ### R-0074 — Availability contract: fail-closed, no queue, no fallback

*Locks Frame C12. Anchors: intake Hard constraint 4 + SC7 (the ratified availability matrix); intake Non-goal 5 — no local write queue, ever (reconciliation = split-brain by construction); P-TrustworthySignal — structured states, the retrieval cluster's structured-unavailable precedent; P-SecurityLayered fail-closed as the integrity backstop. - **R-0074-a (the client contract):** The availability matrix SHALL render as client-contract semantics, exactly:

*Locks Frame §8. #### SC8 dry-run flip plan (spec-tier observable)

Per-function ritual instantiated for coordination: **statement → drain → backup → remove** (the migrate step is vacuous by R-0077-b). Preconditions common to every pass: scheduled window; **operator-only-live** (only the operator instance running at the statement); the coordination cluster deployed and its acceptance suite green.

1. **Pass 1 — leases.** *Drain check:* zero live file claims (`.claude/`-convention claim files) — released or expired; verified by listing the claim-file directory. *Statement:* dated, explicit; from this instant the `claim` tool is the sole claim authority and the file-claim convention is frozen (writes to it are a protocol violation, not a fallback). *Backup:* archive snapshot of the claim-file directory. *Remove:* claim files and their convention docs retired. No dual-authority window: the drain check ran before the statement; the frozen substrate accepts no writes after it.
2. **Pass 2 — messaging.** *Drain check:* the file handoff inbox processed to empty — every item consumed or archived; verified by listing the inbox tree. *Statement:* dated, explicit; `message` becomes the sole handoff medium; the inbox convention freezes. *Backup:* archive snapshot of the inbox tree. *Remove:* inbox directories and convention docs retired.
3. **Pass 3 — stash-carrier.** A consumer-pattern change over the already-flipped message primitives (intake Non-goal 7): it adds no coordination surface and needs no coordination-side flip mechanics; it rides its own later cutover pass under the same ritual, out of this cluster's scope.

**Rollback: none by design — stated, not inferred.** After a pass's *remove* step there is no path back to the file substrate: re-introducing it would recreate the dual-authority window Hard constraint 5 forbids, in reverse, and would resurrect exactly the convention-decay failure class the wedge exists to close. Recovery from a coordination-cluster defect discovered post-remove is **fix-forward against the new substrate**; the *backup* step's archive is forensic reference — readable, never re-activatable. The operator's 2am move is fix-forward, not restore.

Each pass is independently schedulable (R-0077-a); in-flight coordination state is drained, never migrated (R-0077-b); the per-pass flip *executions* belong to the program frame and the cutover statements themselves — this plan is the design-property demonstration SC8 requires, not the operational schedule.

## Scenarios

Given/When/Then for the behaviors this spec locks; each renders an intake success criterion or a Frame QA scenario / trust-boundary resolution.

### Scenario: exactly one holder under contention (R-0065; QA-1; SC2)

**Given** two attached sessions A and B and no lease on `repo-lane:mnemra-core/main-merge`
**When** both concurrently invoke `claim acquire` on that resource — both calls released from a shared start barrier so they reach the storage layer together (genuine contention, not scheduler-serialized calls) — repeatedly across runs
**Then** exactly one acquisition succeeds per round; the loser receives the structured refusal `resource_held` carrying the holder and expiry; the refusal appears in the op log with its reason code; no post-round read ever observes two live leases for the resource — the between-observations guarantee is carried by the schema-level uniqueness constraint (R-0065-b's [operator SQL surface] half), not by the test's observation cadence. [SC2]

### Scenario: dead-holder recovery, same actor identity (R-0064/R-0065/R-0066; QA-2; SC1, SC3)

**Given** session S1 attached as role-instance `merger` holding a lease, and S1 is killed without release
**When** the lease TTL and attachment TTL elapse (the configured test-environment values — the sanctioned config override, §Numeric calibrations test calibration; never a wall-clock ten-minute wait), and a successor session S2 binds with `poll(role_instance = "merger")` then invokes `claim takeover` on the resource
**Then** S2 resolves the same `actor_id` S1 held (no duplicate row); the attachment succession is audited with prior-holder and staleness evidence; the takeover succeeds and its audit record carries prior holder, new holder, expiry timestamp, and takeover timestamp; no manual state wipe occurred. [SC1, SC3]

### Scenario: takeover of a live lease is refused (R-0066; TB-2)

**Given** an attached session holding an unexpired, actively-renewed lease
**When** another attached actor invokes `claim takeover` on that resource
**Then** the refusal `not_expired` returns; the lease is unchanged; the attempt is in the op log. [SC3]

### Scenario: full consumption trail; non-addressee refused (R-0068/R-0069; QA-3; SC4)

**Given** a `handoff` message sent from actor A to actor B
**When** B polls, acks, and dispositions it `completed`, and C (a third attached actor) attempts `ack` and `disposition` on the same message
**Then** the message's row carries `delivered_at`, `acknowledged_at`, `dispositioned_at` timestamps and the disposition member; C's attempts are refused `not_addressee` and logged; no path exists that mutates the payload or addressing post-send or reverts the disposition. [SC4]

### Scenario: disposition without acknowledgement is refused (R-0069)

**Given** a message in state `delivered` to its addressee
**When** the addressee invokes `disposition` directly
**Then** the refusal `invalid_transition` returns naming the required prior state; after an `ack`, the same `disposition` succeeds. [SC4]

### Scenario: founding-type schema audit; out-of-schema send refused (R-0070/R-0071; QA-4; SC5)

**Given** the locked merge-request particulars (the merger-role input contract) and the registered `merge-request` v1 schema
**When** a schema audit walks the particulars against the schema, and a fixture sends one conformant payload and one payload missing `base_pin`
**Then** every particular maps to a named schema field with zero prose fallbacks; the conformant send round-trips intact through poll; the non-conformant send is refused `schema_violation` naming the missing field, and no row lands. [SC5]

### Scenario: standalone cluster (R-0076; QA-5; SC6)

**Given** a fresh deployment holding only core-entity rows and empty coordination tables — zero content tables populated
**When** the full walk runs: bind poll → acquire → send (`merge-request`) → addressee bind poll → ack → disposition → release
**Then** every step succeeds with its response asserted against its documented §API Contract shape (`release` names the same `lease_id` `acquire` returned; the closing read shows `state: "dispositioned"`) — never merely the absence of an error (R-0076-d); the schema and code audit finds no dependency from the coordination cluster on content tables or corpus population. [SC6]

### Scenario: fail-closed write mid-session (R-0074; QA-6; SC7)

**Given** an attached session mid-unit, and the coordination store becomes unavailable
**When** the session attempts any coordination write (`renew`, `send`, `ack`)
**Then** the write returns the structured unavailable error (distinct from a refusal) within its timeout; nothing is queued; the write is provably absent from the store after recovery; the client contract renders the failure as a stop. [SC7]

### Scenario: unattached call refused; read_observer excluded (R-0064-e/R-0073-b)

**Given** a fresh session that has not polled, and separately a `read_observer`-token session
**When** each invokes `claim acquire`
**Then** the first receives `not_attached`; the second is refused pre-dispatch at the token-role gate. [SC2 precondition discipline]

### Scenario: visible reads; list transitions nothing (R-0072-b/R-0073-a; TB-1/C8)

**Given** actor A with a `sent` (never-polled) message in its queue and a held lease
**When** actor B invokes `message list` filtered to A's queue and `claim list`
**Then** B reads the message (payload included) and the lease (holder, resource, expiry); the message's state and timestamps are unchanged by B's list; B gains no write authority — B's `ack` against A's message still refuses `not_addressee`. [TB-1 rendered]

## API Contract

Payload field names below are binding at contract level; exact JSON serialization is implementation-tier. Both tools are host-served on the existing stdio MCP server (R-0063). Every response is structured; there is no prose-only success or failure anywhere on the surface.

### Error taxonomy (one grammar across the surface)

- **Refusal** — the operation was understood, authorized processing ran, and the substrate's rules refuse it: `{ refused: true, reason_code, detail }`. `reason_code` is a closed enum: `resource_held`, `not_holder`, `not_expired`, `lease_not_found`, `invalid_duration`, `invalid_resource`, `reserved_family`, `not_attached`, `actor_live_attached`, `attachment_mismatch`, `wrong_actor_type`, `invalid_role_instance`, `not_addressee`, `invalid_transition`, `invalid_disposition`, `schema_violation`, `unknown_type`, `message_not_found`. Out-of-enum codes SHALL NOT be emitted; extension is a spec amendment (`reserved_family` — R-0067-c — and `invalid_role_instance` — R-0064-a — entered by exactly that route at the r1 fold, §Changelog). Every refusal is logged (R-0075-a).
- **Unavailable / write-failure** — the substrate could not verify the operation: a structured error state distinct from a refusal, rendering the R-0074 stop contract. Never empty-success, never silent retry.

### Tool: `claim` — actions as a closed enum parameter

- **`acquire`** — Request: `{ action: "acquire", resource, duration_seconds? }` (defaults + bound per §Numeric calibrations). Response: the lease object `{ lease_id, resource, holder: {actor_id, role_instance}, acquired_at, expires_at }`. Refusals: `resource_held` (carrying holder + expiry), `invalid_resource`, `reserved_family` (R-0067-c), `invalid_duration`, `not_attached`.
- **`renew`** — Request: `{ action: "renew", lease_id }`. Holder-only. Response: `{ lease_id, expires_at }`. Refusals: `not_holder`, `lease_not_found` (expired — taken over or not — released, or a `lease_id` that never existed; R-0065-d), `reserved_family` (an attachment row's id — R-0067-c), `not_attached`.
- **`release`** — Request: `{ action: "release", lease_id }`. Holder-only. Response: `{ lease_id, released: true }`. Refusals: `not_holder`, `lease_not_found` (incl. already-released and never-existed ids), `reserved_family` (R-0067-c), `not_attached`.
- **`takeover`** — Request: `{ action: "takeover", resource }`. Expired leases only; always audited (R-0066). Response: the new lease object plus `{ superseded: { prior_holder, expired_at } }`. Refusals: `not_expired`, `lease_not_found` (no lease on the resource — use `acquire`), `reserved_family` (R-0067-c), `not_attached`.
- **`list`** — Request: `{ action: "list", family?, resource_prefix? }`. Read-only; workspace-visible (R-0073-a). Response: `{ leases: [lease object …] }` — never including `actor:`-family rows (R-0067-c). Refusals: `not_attached`, `reserved_family` (a filter naming the `actor` family).

### Tool: `message` — actions as a closed enum parameter

- **`poll`** — Request: `{ action: "poll", role_instance }` — the bind call (R-0064-e): resolves-or-creates the actor, attaches (or audited-succeeds a stale attachment), then delivers. Subsequent polls in the same session carry the same shape (the `role_instance` must match the session's attachment; mismatch refuses `attachment_mismatch` — one session, one role-instance). Response: `{ actor: {actor_id, role_instance}, messages: [message object …], live_leases: [lease object …] }` — the actor's non-terminal messages (`sent` rows transitioning to `delivered`, R-0072-a) plus the workspace's live leases (never including `actor:`-family rows — R-0067-c). Refusals: `actor_live_attached`, `wrong_actor_type`, `attachment_mismatch`, `invalid_role_instance` (R-0064-a).
- **`send`** — Request: `{ action: "send", to_role_instance, type, schema_version, payload }`. Sender is the attached actor, host-resolved — no sender parameter exists. Response: `{ message_id, state: "sent", sent_at }`. Refusals: `unknown_type`, `schema_violation` (naming the violated rule), `invalid_role_instance` (R-0064-a), `not_attached`. (An unknown `to_role_instance` resolves-or-creates the addressee actor row through the identity-builtin path — under the same one host-registered identifier rule and validator as the bind path (R-0064-a; a failing identifier refuses `invalid_role_instance`, no row minted), and the row-minting is audited as a registration event (R-0075-b). **Ordering is pinned: the addressee principal is minted only after the send passes every validation gate** — a send refused for any reason (`not_attached`, `unknown_type`, `schema_violation`, `invalid_role_instance`) mints no actor row and emits no registration event. Addressing a role-instance that has not yet bound is legal; its queue waits for its first bind.)
- **`list`** — Request: `{ action: "list", addressee_role_instance?, state?, type? }`. Read-only; workspace-visible; transitions nothing (R-0072-b). Response: `{ messages: [message object …] }`. Refusals: `not_attached`.
- **`ack`** — Request: `{ action: "ack", message_id }`. Addressee-only; `delivered → acknowledged`. Response: `{ message_id, state: "acknowledged", acknowledged_at }`. Refusals: `not_addressee`, `invalid_transition`, `message_not_found`, `not_attached`.
- **`disposition`** — Request: `{ action: "disposition", message_id, disposition, note? }`. Addressee-only; `acknowledged → dispositioned`; audited (R-0075-b). Response: `{ message_id, state: "dispositioned", disposition, dispositioned_at }`. Refusals: `not_addressee`, `invalid_transition`, `invalid_disposition`, `message_not_found`, `not_attached`.

The **message object** in responses: `{ message_id, sender: {actor_id, role_instance}, addressee: {actor_id, role_instance}, type, schema_version, payload, state, sent_at, delivered_at?, acknowledged_at?, dispositioned_at?, disposition?, disposition_note? }`.

## Numeric calibrations (spec-tier; dogfood-scale defaults; config-tunable)

Each default below is config-backed and validated at startup; the *existence and enforcement* are the lock (the requirement clause), the number is a tunable default sized for single-team dogfood scale.

| Calibration | Default | Requirement | Rationale |
|---|---|---|---|
| Lease default duration (when `acquire` omits it) | **15 minutes** | R-0065-d | Long enough for a typical held unit (a doc edit, a merge-lane hold between renewals); short enough that a dead holder's resource frees within one coffee break. Holders doing longer work renew. |
| Lease maximum duration (per-acquire policy bound) | **4 hours** | R-0065-d | Sized to the largest external unit of work a lease guards (a long batch land / migration window) — the bound the Frame's residual 5 sizes the expiry-during-outage exposure by. Above it, the request is refused `invalid_duration`. |
| Attachment TTL (idle expiry; implicitly renewed per R-0064-d) | **10 minutes** | R-0064-d | **Security-load-bearing (TB-3):** the ceiling on successor-lockout after a session dies mid-work, and the floor on the impersonation-after-death window (succession before expiry is impossible; after it, succession is audited with evidence). At single-team scope with integrity > confidentiality, fast recovery wins; the audited-succession record is the compensating control on the shortened window. Calibrated under exactly the C3 tension, as the Frame requires. |
| Coordination-write client timeout (the R-0074-b structured-failure bound) | **10 seconds** | R-0074-b | A coordination write is a single-row transaction; 10 s distinguishes "down" from "slow" decisively at dogfood scale while keeping the fail-closed stop prompt. One attempt; no automatic retry. The bound covers the **whole call, storage-connection acquisition included** — the shared storage pool's own acquire-timeout (a 60 s placeholder tuned for the retrieval/reporting workload) neither satisfies nor replaces it; the committed-tier plan reconciles the mechanism (a wrapping race that cancels the storage call, or a dedicated short-acquire sub-pool for coordination writes) so the two independently-authored numbers never interact emergently. |

**Test calibration is sanctioned, not a hack.** Every default above is config-set at startup; an acceptance-test environment running shortened values (second-scale lease and attachment TTLs) is a **supported configuration** — the lock is each calibration's existence and enforcement, never its number — so the ACs that wait out a TTL (R-0064-d, R-0065-e, QA-2) are authored against the configured test-environment value and run at CI speed, not against a ten-minute sleep. No client-settable per-request override exists for the attachment TTL: the security-load-bearing knob stays deployment-scoped (lease duration remains request-parameterized within the policy bound, as R-0065-d already provides). The sanction carries a production guard: **production startup SHALL validate the attachment-TTL config value against an expected production range or an explicit environment tag, and refuse to start on a mismatch** — a test-calibrated second-scale value cannot silently ship to a production deploy. (The calibrations are already config-backed and validated at startup; this names that validation's production teeth on the one security-load-bearing knob.) *(Binary-observable: a production-tagged startup with a second-scale attachment TTL refuses to start with a structured configuration error — [acceptance test].)*

**Poll cadence (guidance, non-normative).** Push is deferred (R-0072-c), so polling is the delivery path; the intended cadence is once at session start (the bind) plus at natural work-unit boundaries. A tight poll loop is not the intended parity substitute for push — timely-delivery pressure is the push tripwire's signal, not a load profile to normalize.

- Engine-agnostic: all storage through the `Storage` trait (P-0010 D5); no engine-specific feature outside the trait's seam.
- No new dependencies without the workspace tier review; the cluster is expected to need none beyond the substrate's existing set. ## Decisions of note

Each locked decision + its canon anchor. No tied trade-off survived canon-anchoring; **zero escalations** — every decision below is application of a locked Frame direction, the locked intake, or spec work the Frame explicitly delegated (Frame §9's Stage-3 obligations). The Frame's own routine decisions (its §14) carry forward as already-recorded and are not restated.

- **The lifecycle gains an explicit `sent` state ahead of the Frame's `delivered → acknowledged → dispositioned` chain** (R-0069-a). *Anchors: P-TrustworthySignal — under poll transport (C7), delivery happens at the addressee's poll, not at send; a state model that conflates them would report a delivery that has not occurred; intake JTBD — "addressed messaging with delivery state".* The Frame's three-state consumption chain is preserved verbatim as the monotonic tail; `sent` renders the row's pre-delivery existence at spec precision. Not a re-open: the Frame locked the chain's shape and delegated state-machine rendering to spec tier.
- **Strict no-skip chain: `disposition` requires prior `ack`** (R-0069-a). *Anchors: intake SC4 — every consumption leaves a record: the receipt timestamp is always present on every terminal message; Simplicity — one path, no skip branch; the intake's stale-addressed-item evidence needs delivered-but-unresolved distinguishable.*
- **Disposition vocabulary pinned at `{completed, declined, obsolete}`** (R-0069-c). *Anchors: the Frame delegated the members to spec; MB3 — one closed vocabulary; the intake's unarchivable-item evidence demands a terminal path for no-longer-actionable items (`obsolete`); extension is a spec amendment (P-ShiftLeft D2 — each member has a mechanical validator by virtue of the closed enum).*
- **The bind mechanism is a `role_instance` parameter on `poll`, not session metadata** (R-0064-e). *Anchors: the Frame delegated the mechanism to spec; Simplicity — the smallest mechanism on the locked `rmcp` stdio surface (P-0012), which has no designed session-metadata channel at V0; C3's canonical-bind-at-poll direction rendered literally.* Consequence stated: `poll` is the mandatory first coordination call of every session.
- **Attachment realization: invariants are the requirement; attachment-as-lease is SHOULD, dedicated linkage admissible** (R-0064-f). *Anchors: the Frame's explicit latitude ("the invariants are locked, the realization is not" — P-LockContract applied by the Frame itself); Simplicity — one liveness mechanism preferred.* The spec binds the ACs to the invariants so either realization is verified identically; the `actor` resource family is reserved either way (R-0067-a).
- **Attachment renewal is implicit on activity** (R-0064-d). *Anchors: Simplicity — no attach-renew verb, no heartbeat daemon; C3's liveness-rides-lease-discipline direction; the two-tool surface (R-0063-b) forecloses a third liveness verb.* Every successful coordination operation renews; `poll` is the explicit heartbeat when idle.
- **Token-role composition: the coordination surface is unreachable by `read_observer` tokens at V0** (R-0073-b). *Anchors: P-0009 — attachment is a write, and every coordination action (the Frame matrix's "any resolved actor") executes under an attachment; stating the consequence rather than carving a read-only bind path keeps the matrix verbatim and the mechanism smallest.* Monitoring reads for read-only consumers are a future need that announces itself (it would arrive as its own intake against the C8 default anyway).
- **No message-type registry table at V0** (R-0070-a). *Anchors: Frame C6 — types are code-registered, shipped with the host; Simplicity — a table for a closed code-owned set is a second source of truth (P-0017 D-SoT anti-pattern).* This resolves Frame C13's conditional "message-type registry state" to none.
- **Resource-identifier families pinned at `{repo-lane, file, surface}` + reserved `actor`, with the reserved family barred from the entire `claim` surface** (R-0067; R-0067-c added at the r1 fold). *Anchors: the Frame delegated the scheme to spec; P-ShiftLeft D2 — each family ships its validator; the merger lane's `repo-lane:<repo>/main-merge` is the first designed consumer (C4).* The r1 security + operational reviews converged on the same gap — the reservation was enforced only against `acquire`, leaving `takeover`/`renew`/`release`/`list` reachable against the identity substrate under the attachment-as-lease realization, outside the audited-succession-at-bind control; R-0067-c closes it as enforcement-completeness on the settled reservation (the family set itself is untouched), with the dedicated `reserved_family` code so the bar is distinguishable from a malformed identifier.
- **Expired-lease recovery is `takeover`-only: the folded-cleanup admissibility was withdrawn at the r1 fold** (R-0066-a). *Anchors: P-LeastAuthority — takeover is a distinct privileged action, never a side effect, now unqualified; P-TrustworthySignal — every takeover fires the R-0066-b audit record on one named path; the audit record is the non-negotiable.* The draft admitted folding expired-row cleanup into `acquire`'s transaction provided the op log distinguished the takeover path; the r1 reviews showed that latitude (a) made "takeover" ambiguous against R-0066-b/R-0075-b's audit-mandatory semantics — a literal fold-path implementer could satisfy the op-log clause and never emit the audit record — and (b) left the AC's pass/fail shape branching on an implementation choice deferred past red-phase test authoring. The latitude was spec-authored (not Frame-locked), so withdrawing it re-opens nothing: one client-visible contract, one audited recovery path.
- **Refusal grammar amended at the r1 fold: `reserved_family` + `invalid_role_instance` (16 → 18 codes; the enum stays closed, extension stays spec-amendment-only)** (§API Contract; R-0067-c, R-0064-a). *Anchors: P-ShiftLeft D2 — each new code ships with its own binary-observable emission criteria; MB3 — one grammar across the surface; P-TrustworthySignal — a reserved-family probe, a typo, and a malformed principal identifier are distinguishable signals.*
