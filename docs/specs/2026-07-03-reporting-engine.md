## Purpose / context

Operators and agents working over mnemra-core's stored context and measurement data need repeatable, named, parameterized reports available at runtime without a code change, rebuild, or redeploy ‚Äî so recurring data cuts stop being hand-rolled one-off queries that drift and sometimes compute the wrong answer (intake JTBD). The engine is a **host-core read-only service** over the substrate: a registry with two report kinds (compiled built-ins and declarative user reports), a dedicated Postgres execution context that makes read-only and workspace/policy isolation structural on the one path where the host cannot thread its own `WHERE` clause (caller-authored SQL), and a uniform structured output surface exposed as MCP verbs.

The closed world for this spec is the locked Frame (blob `871441d`). This spec maps the Frame's directions 2a-1..2a-3 and decisions D1‚ÄìD8 to requirement slots **R-0040‚ÄìR-0062** at spec precision; nothing outside the Frame's locked directions, the locked intake, and cited canon enters. Where a requirement binds a contract the companion ADRs own (the execution-context security model, the definition schema and write-time validation), the ADR is the authority and the requirement cites it.

### R-0040 ‚Äî Host-core placement; the reporting engine is a host subsystem with no plugin surface

*Elaborates Frame D1 (2a-1). Anchors: 2a-1 (ratified); [P-0019](../src/adrs/P-0019-plugin-contract.md) ¬ßCorrection 1 + D2.5 ‚Äî plugins are leaves, host-mediated IO, host-derived `WorkspaceCtx`; [P-0002](../src/adrs/P-0002-core-plugin-partition.md) ‚Äî cross-plugin aggregation is a projection concern; [P-0014](../src/adrs/P-0014-retrieval-architecture.md) Considered-Option-3 precedent; [P-0013](../src/adrs/P-0013-plugin-invocation-model.md) ‚Äî host placement keeps the deferred plugin domain-verb machinery unbuilt.*

- **R-0040-a (host placement):** The reporting engine ‚Äî registry, executor, definition-validator, and renderer ‚Äî SHALL be host code inside the mnemra-core host process. Report SQL of **both** kinds executes host-side over the plugin-namespaced table families; no plugin reads a sibling's tables because no plugin reads anything new. *(Binary-observable: the engine's reads occur above the plugin boundary; no plugin export interface changes in this feature's diff ‚Äî [construction + review audit].)*
- **R-0040-b (no new plugin/WIT surface):** This feature SHALL add no WIT surface (`wit/host.wit` untouched at this tier) and SHALL NOT introduce any plugin-exported report verb. Report verbs are host-served (R-0055), not plugin exports. *(Binary-observable: the WIT world is unchanged by this feature's diff; the plugin export surface is unchanged ‚Äî [construction + review audit].)*

### R-0041 ‚Äî Report registry: two kinds, one lookup, built-in preference, write-time name reservation

*Elaborates Frame D6. Anchors: D6 (registry mechanism); [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-SoT ‚Äî the content row is the source of truth, caches are derived; carried worked-reference R81/R83.*

- **R-0041-a (two kinds, one lookup):** The registry SHALL hold two report kinds behind one lookup keyed by report name: **built-in** (compiled host code, reserved names, registered at startup) and **user** (definition artifacts resolved from the definition store, R-0042). A lookup for a name SHALL resolve to exactly one report or a structured not-found naming the name. *(Binary-observable: a lookup for a reserved built-in name resolves to the built-in; a lookup for a registered user-definition name resolves to that definition; a lookup for an absent name returns a structured not-found ‚Äî [acceptance test].)*
- **R-0041-b (built-in preference; reservation enforced at write time):** A built-in SHALL win any lookup collision; a user definition SHALL NOT shadow a built-in. Reserved-name enforcement SHALL occur at definition **write time** ‚Äî a create/update naming a reserved built-in name is rejected with a structured error naming the collision, and the definition SHALL NOT register. *(Binary-observable: an attempt to author a definition whose name equals a reserved built-in name is rejected at write with the collision named; the built-in remains reachable ‚Äî [acceptance test].)*
- **R-0041-c (uniqueness):** Definition names SHALL be unique per `(workspace_id, name)` among user definitions. A second create of an existing `(workspace_id, name)` is rejected with a structured error; there is no cross-directory precedence tie-break (the filesystem discovery model does not exist here ‚Äî R-0082 drop, SC4 map). *(Binary-observable: a duplicate `(workspace_id, name)` create is rejected ‚Äî [acceptance test].)*
- **R-0041-d (store is source of truth; write-then-invoke, no restart):** The registry SHALL read the definition store (R-0042) as the authoritative source; any in-memory registry cache SHALL be a rebuildable derivation, never written as authoritative (P-0017 D-SoT). A user definition that lands through the write path SHALL be resolvable on the next lookup with no process restart (intake SC1 runtime extensibility). *(Binary-observable: a definition authored via the write verb is invocable in the same process lifetime without a restart ‚Äî [acceptance test].)*

### R-0042 ‚Äî The report-definition content type, schema, and write-time validation

*Elaborates Frame D2 (2a-2); the requirement side of [`{{P-ReportDefinitionType}}`](../src/adrs/P-0021-report-definition-type.md) (P-0021), which is the schema/validation authority. Anchors: 2a-2 (ratified); [P-0001](../src/adrs/P-0001-storage-layout.md) C1 + [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-CM/D-SoT (`reference` cluster); [P-0015](../src/adrs/P-0015-provenance-envelope-source-roles.md) PE-2/PE-6; [P-0018](../src/adrs/P-0018-core-entity-manifest.md) D-BOUNDARY (a host-core content type that is NOT a core FK-target entity); P-LockContract (the schema is the lock; storage mechanics vary behind the `Storage` seam).*

- **R-0042-a (content artifact through the existing write machinery):** A report definition SHALL be a content artifact ‚Äî one row in its own per-artifact-type table under the C1 layout (JSONB frontmatter + body + system fields), host-core-owned, classified `reference` (P-0017 D-CM). It SHALL be written through the existing content-write internals (validation, `WorkspaceCtx` threading, workspace scoping, P-0015 policy columns with their DDL defaults, audit emission). This feature SHALL NOT add a new writer class or a bespoke definition store. It SHALL NOT make the definition type a [P-0018](../src/adrs/P-0018-core-entity-manifest.md) hard-FK-target core entity. *(Binary-observable, per half: the schema shape ‚Äî exactly one new `reference`-cluster per-artifact-type table carrying the P-0015 policy columns ‚Äî is directly checkable via `information_schema.tables`/`columns` ‚Äî [operator SQL surface]; no new writer path and no new hard FK target referencing the definition table ‚Äî [construction + review audit].)*
- **R-0042-b (definition schema):** A definition's frontmatter SHALL carry: `name` (string, REQUIRED, matching `^[a-z0-9][a-z0-9-]*$`, not a reserved built-in name); `description` (string, REQUIRED); zero or more typed parameter declarations (R-0057); and a default output format (R-0056). The body SHALL be **exactly one read statement**. Any unknown top-level frontmatter key SHALL be ignored (forward-compat) and SHALL NOT invalidate the definition. *(Binary-observable: a definition with `name`/`description`/one-SELECT-body validates and registers; a definition whose `name` is `Foo Bar` is rejected naming the field and the rule; a definition with an unknown extra key still validates ‚Äî [acceptance test].)*
- **R-0042-c (write-time validation, before the row lands):** The write path SHALL run, before the definition row is committed: (1) schema validity at declared types; (2) name rules + reserved-built-in-name reservation (R-0041-b); (3) the **single-read-statement check** ‚Äî the body parses, is exactly one statement, and is a read (`SELECT` / `WITH‚Ä¶SELECT`) with **every CTE itself read-only**: a data-modifying CTE (e.g. `WITH t AS (DELETE ‚Ä¶ RETURNING *) SELECT ‚Ä¶`) is `WITH‚Ä¶SELECT`-shaped yet writes and SHALL be rejected at write time; (4) placeholder‚Üîparameter reconciliation ‚Äî a `:param`-style placeholder with no declaration is a hard error naming the placeholder; a declared-but-unused parameter is a warning naming the parameter, and the definition still registers; (5) each parameter `default` value passes its own type validation. An invalid definition SHALL NOT register. *(Binary-observable: a two-statement body is rejected; a `WITH t AS (DELETE ‚Ä¶ RETURNING *) SELECT ‚Ä¶` body is rejected at write naming the data-modifying CTE; a body referencing `:until` with no `[params].until` is rejected naming `:until`; a `date` parameter with `default = "not-a-date"` is rejected ‚Äî [acceptance test].)*
- **R-0042-d (policy-record inheritance on definitions):** A definition SHALL inherit the full P-0015 policy record with unchanged semantics: `visibility` governs who can list/read/invoke the definition (its body text can embed sensitive filter literals ‚Äî the intake motivation); `dont_use` is the curatorial kill switch (a killed definition resolves per the PE-4 stub disposition ‚Äî R-0053-b); `visibility: owner-only` follows PE-4 V0 semantics (serves no one until per-user identity lands); `tenant_share` is the structural workspace-only constant. Every policy-field write on a definition SHALL ride the PE-6 admin-gated path with an attributable audit row (R-0054). *(Binary-observable: a `dont_use`-killed definition returns the metadata stub on invoke; an `admin-only` definition is absent from a `read_observer`'s list; a policy-field flip by a `read_observer` token is refused ‚Äî [acceptance test].)*

### R-0043 ‚Äî Dedicated read-only execution context: the two-independent-write-block pair [GUARDRAIL]

*Elaborates Frame D3 (2a-3); scope owned by [`{{P-ReportExecutionContext}}`](../src/adrs/P-0020-report-execution-context.md) (P-0020). Anchors: 2a-3 (ratified); worked-reference R85 defense-in-depth pair, re-derived for Postgres; P-SecurityLayered (each layer independently load-bearing); the `Security ‚áÑ Simplicity` conflict edge, resolved default-to-Security.*

- **R-0043-a (report role ‚Äî SELECT-only privileges):** All report SQL (user AND built-in) SHALL execute under a dedicated Postgres role holding `SELECT`-only privileges ‚Äî no `INSERT`/`UPDATE`/`DELETE`, no DDL, no sequence usage. *(Binary-observable: a write attempted as the report role fails with a privilege error ‚Äî [acceptance test].)*
- **R-0043-b (READ ONLY transaction):** Every report transaction SHALL be `READ ONLY` (`SET TRANSACTION READ ONLY` / `default_transaction_read_only`). *(Binary-observable: a write attempted inside the report transaction fails with a read-only-transaction error even absent the privilege block ‚Äî [acceptance test].)*
- **R-0043-c (each write-block independently sufficient):** The two mechanisms in R-0043-a/-b SHALL each alone block a write. The threat-test suite (¬ßThreat-test suite) SHALL prove each independently ‚Äî a write blocked by the role privileges with the transaction hypothetically read-write, and a write blocked by the read-only transaction with the privileges hypothetically granted. *(Binary-observable: two negative tests, one isolating each mechanism, each demonstrably blocking the write ‚Äî [acceptance test].)*

### R-0044 ‚Äî Grant-set allow-lists (tables + functions), default-deny exclusions, reconciliation, and provisioning

*Elaborates Frame D3; scope owned by P-0020. Anchors: D3 (the Postgres re-derivation of the worked-reference authorizer allow-set); P-TrustworthySignal TS2 (an execution-gating enumeration ships with its reconciliation check); OC-10.*

- **R-0044-a (table grant allow-list ‚Äî a derivation rule over two canonical sources, not a hand list):** The report role's readable surface SHALL be an **allow-list of `SELECT` grants** whose membership is *derived*, mechanically, from two canonical sources under rules locked in [P-0020](../src/adrs/P-0020-report-execution-context.md): **(i) content-artifact tables** ‚Äî every per-artifact-type content table created through the host content machinery (the C1 generator; enumerable from the content-type registry) is in the grant set, and receives its report-role RLS policies (R-0050-a) at table creation through that same machinery ‚Äî the report-definition table itself included. **Primary tables only**: the content machinery's history/shadow tables are NOT in the set (default-deny; *deferral:* decision content ‚Äî whether historical versions become report-readable; anchor ‚Äî P-Defer + the policy-envelope question historical rows raise; instrument ‚Äî the first report request needing versioned rows). **(ii) product-measurement tables** ‚Äî enumerated by name in the provisioning DDL (R-0044-e), scoped to the dispatch/skill-run/activity **measurement** families; the retrieval-cluster operator-side instrumentation/audit tables are NOT "product-measurement tables" under this clause (they are R-0044-b exclusions). Changing the membership *rules* (adding a family, granting a core-entity or substrate table, removing an exclusion) SHALL be an amendment to P-0020, never a config knob; a new content type flowing through the content machinery is the existing rule applying, not an extension ‚Äî and it cannot become report-readable ahead of its RLS policies, because the reconciliation asserts the R-0050 matrix per table (R-0044-d) at its standing trigger (R-0044-e). *(Binary-observable: the report role's computed granted-table set (pg_catalog ACLs) equals the set derived from the two canonical sources minus the R-0044-b exclusions ‚Äî [operator SQL surface], the reconciliation's table-axis assertion.)*
- **R-0044-b (default-deny exclusions ‚Äî nameable-complete):** The following SHALL be excluded from the report-role grant set by default-deny (no grant = unreadable), enumerated so the reconciliation (R-0044-d) and T8 assert against names, not categories: **the identity/auth/session tables** ‚Äî `admin_tokens` (foremost), `users`, `sessions`, and `workspaces` (readable workspace rows would disclose tenant existence cross-tenant); **the P-0015 policy-audit stores** ‚Äî `policy_write_audit`, `policy_decision_audit`; **the retrieval-cluster operator-side instrumentation/audit tables** ‚Äî `egress_events` (its `payload_audit` carries actual egressed content ‚Äî the consumed retrieval spec groups it with `policy_decision_audit` as credential-gated sensitive), `retrieval_runs`, `traversal_log`, `index_builds`; **the content machinery's history/shadow tables** (R-0044-a(i)); and **the operator-internal substrate tables** ‚Äî `state_config`, `schema_migrations`. These are operator-side signals whose exposure through caller SQL would itself be an existence-disclosure or content-egress channel. The core-entity tables (`projects`, `agents`, and kin) are simply *not granted* at V0 ‚Äî not sensitive-class; adding one is a P-0020 amendment, not an exclusion-list change. *(Binary-observable: a report selecting from `admin_tokens`, `policy_write_audit`, `policy_decision_audit`, or `egress_events` fails with a permission error ‚Äî [acceptance test], T8.)*
- **R-0044-c (function-EXECUTE surface ‚Äî the two-part lock; PUBLIC-default revocation):** Because Postgres grants `EXECUTE` to `PUBLIC` by default, a table-only allow-list is default-*allow* on the function axis. The report role's callable-function surface SHALL therefore be locked as a **two-part structure** (the canonical, auditable form of the allow-list at this tier): **(1) non-`pg_catalog` functions are default-deny** ‚Äî none is executable by the report role except an explicit per-function grant enumerated in the provisioning DDL; this enumeration is **EMPTY at V0**, and adding an entry is a P-0020 amendment. Any in-schema `SECURITY DEFINER` function is categorically outside the permittable set. **(2) `pg_catalog` built-ins are callable except the enumerated revocation classes**, revoked for the report path: (a) **GUC/session mutators** ‚Äî `set_config(text,text,boolean)` and any `pg_catalog` function that sets a configuration parameter (the R-0046 session-key threat); (b) **the server-info / statistics / file-access family** ‚Äî the `pg_stat_*` view+function family, `pg_ls_*`, `pg_read_file`/`pg_read_binary_file`/`pg_stat_file`, `pg_current_logfile`. A literal positive enumeration of `pg_catalog`'s thousands of built-ins is deliberately NOT the mechanism (unauditable, version-drift-prone); the locked invariant is that **no revocation-class function and no non-`pg_catalog` function is executable by the report role**, reconciliation-asserted (R-0044-d). Concretely permitted, so the positive threat-test case is constructible without inference (T9): `count`, `sum`, `avg`, `min`, `max`, `percentile_cont`, `percentile_disc`, `coalesce`, `date_trunc`, `now`, and `pg_sleep` (harmless ‚Äî bounded by `statement_timeout`; the R-0048-a fixture's seeding function). Revocation mechanics (`REVOKE EXECUTE ‚Ä¶ FROM PUBLIC` + explicit re-grants, or a non-`PUBLIC`-inheriting report role) are spec-tier within the P-0020 D-GRANT shape. *(Binary-observable: `SELECT set_config('x','y',false)`, `SELECT pg_read_file('x')`, and `SELECT * FROM pg_stat_activity` executed as the report role fail; `SELECT percentile_cont(0.5) WITHIN GROUP (ORDER BY duration_ms)` over a granted table executes ‚Äî [acceptance test]; the revocation classes, the SECURITY DEFINER sweep, and the empty non-`pg_catalog` inventory are reconciliation-asserted ‚Äî [operator SQL surface].)*
- **R-0044-d (reconciliation check ‚Äî OC-10; the full assertion matrix):** The grant set (tables and functions) and its RLS coverage SHALL ship with a reconciliation check in the same change, demonstrably failing on a seeded violation. The check SHALL assert, **per granted table**: RLS enabled AND **the R-0050 per-family matrix present** ‚Äî for a content-artifact table, BOTH the workspace-RLS policy AND the P-0015 policy-envelope predicates (R-0050-a); for a measurement table, the workspace-RLS policy (R-0050-b) ‚Äî AND the host's primary path provably bypasses via ownership-without-`FORCE` (R-0045-b) AND the report role provably does not; AND **relkind is constrained**: every granted relation is a base table or a `security_invoker` view ‚Äî a non-`security_invoker` view in the grant set is a violation (it would evaluate base-table RLS as the view's owner, silently bypassing the report-role policies; a materialized view cannot carry RLS and fails the RLS-enabled assertion fail-closed). On the **function axis**: no revocation-class function is executable by the report role, no non-`pg_catalog` function is executable beyond the (empty at V0) provisioning inventory, no `SECURITY DEFINER` function is executable (a `pg_proc.prosecdef` sweep over the executable set), and ‚Äî the R-0046 inside-set assertion ‚Äî **the executable surface is GUC-setter-free**. *(Binary-observable: the reconciliation check passes against the true configuration and fails on each seeded violation: an extra table grant; a granted table missing its workspace-RLS policy; a content-artifact table carrying workspace-RLS but missing its policy-envelope predicate; a non-`security_invoker` view seeded into the grant set; a host path that does not bypass; a seeded extra executable function ‚Äî [operator SQL surface] + [acceptance test], T3.)*
- **R-0044-e (provisioning mechanism; standing reconciliation trigger ‚Äî fail-closed; rollback + drift-recovery runbook):** The role / grant / revocation / RLS DDL SHALL ship **migration-borne**, through the host's forward-only migration runner (the P-0009 precedent: RLS activation is additive `CREATE POLICY` DDL riding migrations, no schema migration), **idempotent across fresh init and upgrade-in-place** (guarded `CREATE ROLE`, idempotent `GRANT`/`REVOKE`, guarded `CREATE POLICY`); grants + policies for content types registered *after* this feature lands attach at table-generation time through the content machinery (R-0044-a(i)) ‚Äî the same change, never a separate manual step. The OC-10 reconciliation (R-0044-d) SHALL have a **standing trigger**, not a one-time ship-time gate: it runs (i) as an acceptance/CI gate in this feature's change, and (ii) **on every migration-apply completion and every artifact-table generation** (including the startup apply path), failing loud with a structured error naming each violated assertion (relation/function + the assertion that failed) ‚Äî so a future migration touching a granted family, or a content-type registration that misses its policies, fails at apply time, not at leak time (P-TrustworthySignal TS2: drift fails the check, never the tenant). **Failure semantics ‚Äî fail-closed at every firing point (locked):** a reconciliation failure SHALL abort the operation that fired it ‚Äî a failing CI-gate run fails the change; a failing migration-apply (including the startup apply path) fails the apply, and the host SHALL refuse to start rather than serve with the drift live; a failing artifact-table generation fails that generation (the new table never becomes report-readable). Log-and-continue is prohibited, and there is deliberately NO skip-reconciliation switch ‚Äî a bypass flag would be the fail-open backdoor this clause exists to exclude (the `Security ‚áÑ Simplicity` edge, resolved default-to-Security: availability yields to isolation on this surface). **Drift-lockout recovery (a runbook entry distinct from intentional teardown):** the recovery path for an operator whose host now refuses to start is **diagnose-then-reapply** ‚Äî inspect the named violations via the reconciliation's own [operator SQL surface] queries under a privileged direct connection; re-run the idempotent provisioning DDL (which restores any *missing* grant/policy/role state); reverse any *surplus* state the idempotent re-apply cannot remove (`REVOKE` the extra grant, drop the seeded object); restart ‚Äî the startup reconciliation re-runs and passes. The last-resort escape hatch is the ordered teardown below followed by re-provisioning ‚Äî never a reconciliation bypass. The provisioning change SHALL include a documented **ordered rollback procedure** (`DROP POLICY` per table ‚Üí `REVOKE` grants ‚Üí `ALTER TABLE ‚Ä¶ DISABLE ROW LEVEL SECURITY` ‚Üí `DROP ROLE`) as an operator runbook artifact, with the drift-lockout recovery entry in the same artifact (¬ßOperator deployment requirements) ‚Äî the forward-only migration system does not auto-reverse security DDL, and an out-of-order teardown has two failure modes this ordering forecloses: stranding RLS enabled without its host-bypass precondition, and ‚Äî the reason `REVOKE` precedes `DISABLE ROW LEVEL SECURITY` ‚Äî a transient window in which the report role still holds `SELECT` grants with RLS already off (a cross-tenant read window: after `DROP POLICY` the role is fail-closed to zero rows, and revoking before disabling keeps it fail-closed until its access is gone). *(Binary-observable: fresh-init and upgrade-in-place both converge to the same reconciliation-passing state ‚Äî [acceptance test]; the reconciliation is invoked on migration-apply completion ‚Äî [construction + review audit]; **a violation seeded through the standing-trigger path ‚Äî a fixture migration adding a granted-family table without its RLS policy ‚Äî fails the migration-apply call itself with the structured reconciliation error** (the fail-closed consequence proven end-to-end, not the call site's existence audited) ‚Äî [acceptance test], T3; the ordered rollback runbook + drift-lockout recovery entry exist in the same change ‚Äî [construction + review audit].)*

### R-0045 ‚Äî The scoped RLS exception; the ownership-without-FORCE host-bypass precondition; convergence with R-0001

*Elaborates Frame D3; scope owned by P-0020. - **R-0045-c (the general R-0001 deferral stands; convergence):** The general V0 RLS deferral SHALL stand unchanged: application-layer `WorkspaceCtx` discipline remains the enforcement on every host-constructed query, and the R-0001 trip-wire ‚Äî first deployment serving more than one workspace in production, OR read-path column-shape lint coverage dropping below 100%, OR a third-party plugin loaded at runtime (P-0009's three OR-conditions, restated in full) ‚Äî is unchanged. Definition authoring SHALL surface as host-served MCP **write** verbs riding the existing content-write internals; whether they are type-parameterized content verbs or dedicated report-authoring verbs is an implementation choice within the R-0042-a content-write path. *(Binary-observable: the three read verbs and an authoring write path answer on the stdio MCP server ‚Äî [acceptance test].)*
- **R-0055-c (admin CLI is a convenience frontend, never a second path):** The admin-CLI `report` subcommand SHALL be a convenience frontend over the identical registry/executor/renderer ‚Äî never a second execution or enforcement path (intake ratification 3, OC-1). *(Binary-observable: the CLI path reaches reports through the same registry/executor/renderer symbols as the MCP path ‚Äî [construction + review audit].)*

### R-0056 ‚Äî Uniform structured output

*Elaborates Frame D6 (SC3); carried worked-reference R88/R89. Anchors: SC3; P-TrustworthySignal (truncation and NULL are honest); Simplicity (one renderer).*

- **R-0056-a (three output shapes):** Each report SHALL declare one of three output shapes ‚Äî `scalar`, `matrix`, `row-list`; user reports SHALL be `row-list` (columns = the statement's result columns); built-ins declare theirs. *(Binary-observable: a user report's declared shape is `row-list`; a built-in declares one of the three ‚Äî [acceptance test].)*
- **R-0056-b (canonical structured result; uniform rendering):** The canonical result on MCP SHALL be a structured, shape-tagged JSON object (columns, rows, row count, truncation flag, parameter echo); the admin CLI SHALL render `table`/`csv` views from that same structured result through the one engine-owned renderer. A built-in and a user report of the same shape SHALL render identically for the same format. *(Binary-observable: a built-in and a user report of the same shape produce identical rendering for the same format on a fixed dataset ‚Äî [acceptance test].)*
- **R-0056-c (empty result is success; scalar always carries one value):** Zero rows SHALL be a success with the format's empty rendering (empty `rows: []`, header-only table), never an error. The empty-rendering clause applies to the `row-list` and `matrix` shapes; a **`scalar`-shaped report SHALL always produce exactly one value** ‚Äî an aggregate over zero underlying rows yields that aggregate's defined value (`COUNT` ‚Üí `0`; `SUM` ‚Üí NULL, rendered per R-0056-d), never an empty result and never an error. *(Binary-observable: a report matching zero rows exits success with the empty rendering; a scalar built-in over an empty fixture returns exactly one value (its aggregate's zero-row value), not an empty result ‚Äî [acceptance test].)*
- **R-0056-d (NULL renders explicitly, never dropped; table token pinned):** A NULL cell SHALL render explicitly ‚Äî JSON `null`, empty CSV field, and the literal token **`NULL`** in the CLI `table` view (the engine-pinned table NULL token, consistent across the engine) ‚Äî and a NULL-bearing row SHALL never be dropped. The `table` view is a human convenience rendering: a data string spelling `NULL` is ambiguous there by acceptance; JSON/CSV are the machine-consumable forms. *(Binary-observable: a row with a NULL column renders `null` in JSON, an empty CSV field, and the literal `NULL` in the table view, and is present in every rendering ‚Äî [acceptance test].)*
- **R-0056-e (truncation is reported):** When the row cap (R-0048-b) truncates a result, the structured result SHALL carry the truncation flag and the capped count. *(Binary-observable: a truncated result carries `truncated: true` and the capped count ‚Äî [acceptance test].)*

### R-0057 ‚Äî Parameters: closed V0 type set, typed binds

*Elaborates Frame D2/D6; carried worked-reference R93. Anchors: D6; P-ShiftLeft D2 (validator before field ‚Äî the closed type set has a mechanical validator); Simplicity.*

- **R-0057-a (closed type set):** Each parameter SHALL declare a `type` from the closed V0 set `{string, date, int}`, an optional `required` boolean (default `true`), and an optional `default`. Richer types (`float`, `bool`, `enum`, ‚Ä¶) are out of scope for V0; adding one is a spec amendment. *(Binary-observable: a parameter typed `string`/`date`/`int` validates; a parameter typed `float` is rejected ‚Äî [acceptance test].)*
- **R-0057-b (per-type validation + binding):** `string` binds any UTF-8 text; `date` SHALL match ISO-8601 `YYYY-MM-DD` (a non-matching value is rejected naming the parameter and the format) and binds as a typed date; `int` SHALL parse as a signed 64-bit integer ‚Äî a non-integer value, **including a numeric string outside the i64 range** (overflow is a rejection, never a wrap or clamp), is rejected naming the parameter ‚Äî and binds as an integer. *(Binary-observable: `since=2026-01-01` on a `date` param binds; `since=2026-13-99` is rejected naming the param and format; `n=abc` and a 30-digit `n` on an `int` param are each rejected naming the param ‚Äî [acceptance test].)*
- **R-0057-c (required / default / NULL semantics ‚Äî all four cases):** A `required=true` parameter with no supplied value and no `default` SHALL cause a rejection naming the missing parameter. A `required=false` parameter with no value and no `default` SHALL bind SQL NULL (the SQL author handles NULL). **A parameter with a declared `default` and no value supplied at invocation SHALL bind the declared default** (regardless of `required` ‚Äî the declared default satisfies a required parameter). A `default` value SHALL itself pass the type's validation (R-0042-c). *(Binary-observable: a required param with no value and no default is rejected; an optional param with no value and no default binds NULL and the report runs; a `date` param with `default = "2026-01-01"` invoked with no value binds `2026-01-01` and the result equals an explicit `since=2026-01-01` invocation on the same fixture ‚Äî [acceptance test].)*
- **R-0057-d (unrecognized parameter keys are rejected):** A `report.invoke` whose `params` object carries a key the resolved report declares no parameter for SHALL be rejected with a structured error naming the unknown key ‚Äî the invocation-time mirror of R-0042-c's placeholder‚Üîparameter reconciliation; silently ignoring an extra key would mask a caller typo as a wrong-but-successful report. *(Binary-observable: invoking a report declaring only `since` with `params: { sinse: ‚Ä¶ }` is rejected naming `sinse`; no execution occurs ‚Äî [acceptance test].)*

### R-0058 ‚Äî Statistics: substrate-native percentiles; one-definition parity

*Elaborates Frame D6; SC3 uniformity. Anchors: D6; SC3; P-MinBlastRadius (one percentile definition, not two implementations that can silently disagree). Substrate reason: Postgres natively provides `percentile_cont`/`percentile_disc`; the worked-reference custom `median()`/`percentile()` registrations are dropped (SC4 map).*

- **R-0058-a (native percentiles serve user SQL):** User report SQL SHALL use Postgres-native `percentile_cont` (linear interpolation) and `percentile_disc`; this feature SHALL NOT register custom `median()`/`percentile()` aggregates. `median(x)` is expressible as `percentile_cont(0.5)`. *(Binary-observable: a user report calling `percentile_cont(0.5)` / `percentile_disc(0.95)` executes ‚Äî [acceptance test]; no custom `median`/`percentile` aggregate is registered on the report connection ‚Äî directly checkable live via a `pg_proc` name query against the report connection ‚Äî [operator SQL surface] + [construction + review audit].)*
- **R-0058-b (built-in percentile parity):** Any built-in report computing a percentile SHALL use the same linear-interpolation definition as `percentile_cont` (by calling the same SQL function or an implementation proven equal on fixed data), so a built-in p95 and a user `percentile_cont(0.95)` agree on identical input. *(Binary-observable: on a fixed dataset, a built-in p95 and a user `percentile_cont(0.95)` return equal values ‚Äî [acceptance test].)*

### R-0059 ‚Äî Instrumentation: per-execution telemetry emission

*Elaborates Frame D7 (OC-9). Anchors: P-InstrumentBefore (+ the `P-Defer ‚áÑ P-InstrumentBefore` edge, resolved to instrument ‚Äî this is a production surface); [P-0017](../src/adrs/P-0017-storage-cluster-model.md) D-CP (the query-instrumentation surface); [P-0010](../src/adrs/P-0010-storage-substrate-engine.md) D8/E1 (telemetry emitted, not stored in-app).*

- **R-0059-a (per-execution telemetry, emission-only, workspace-tagged):** The engine SHALL ship its telemetry emission in its first increment: per-execution events ‚Äî `{workspace_id, report_name, kind, caller_role, duration_ms, row_count, truncated, timeout_killed, prepare_failure_reason_class}` ‚Äî emitted through the observability baseline. **`workspace_id` is REQUIRED on every emitted event** (the baseline's own per-verb metric records carry it; a sink that mixes tenants leaks usage patterns cross-tenant, and neither tripwire is diagnosable per-workspace without it). Emission, **not in-app storage** ‚Äî so intake Non-goal 2's no-new-schema posture holds on the execution path. *(Binary-observable: a report execution emits a telemetry event carrying all named fields including `workspace_id` through the observability baseline ‚Äî [acceptance test] over the emitted stream; no product-measurement table for report runs is added ‚Äî [construction + review audit].)*
- **R-0059-b (the tripwires are computable over the retained emitted stream ‚Äî the firing instruments made concrete):** The two deferral firing instruments ‚Äî the R-0048-d cost-ceiling tripwire (timeout-kill count) and intake Non-goal 3's materialization tripwire (per-report latency p95 crossing the pinned budget, ¬ßNumeric calibrations) ‚Äî SHALL each be computable as a **windowed aggregate over the emitted event stream**: a rolling-7-day computation over the per-execution events, performed at the operator-chosen telemetry sink or directly over retained stdout structured logs. This is the observability baseline's own derived-aggregate pattern (per-verb p50/p95/p99 "are derivable from the emitted `duration_ms` values, computed at the operator-chosen sink ‚Äî there is no in-app metrics table"); the reporting engine adds no storage the baseline does not (routine decision 11 unchanged). **Operator prerequisite, named:** the deployment SHALL retain at least the tripwire window (7 days) of emitted report-execution events ‚Äî stdout log retention or a sink; the V0 dogfood default is the host's stdout structured-log capture. Absent that retention the tripwires are not computable, and that absence is an operator-deployment gap, not a silent spec gap. The prerequisite is routed into the shipped operator artifact via ¬ßOperator deployment requirements ‚Äî never spec-prose alone. *(Binary-observable: given 7 days of retained emitted events, each tripwire value is computable by a documented aggregation over the retained stream, and a fixture stream containing ‚â•5 timeout-kill events in-window fires the cost-ceiling tripwire computation ‚Äî [operator telemetry query].)*

### R-0060 ‚Äî Built-ins in the report context; grant-set coverage of built-in reads (carry-forward 3)

*Elaborates Frame D3 + Stage-3 carry-forward 3 + intake Non-goal 7. Anchors: D3 (uniform execution posture); intake Non-goal 1 ("of any kind") + Non-goal 7 (built-ins keep their computation); P-TrustworthySignal TS2 (no silent grant extension).*

- **R-0060-a (built-ins execute in the report context):** Built-in report SQL SHALL execute under the report role / read-only transaction / RLS (R-0043‚ÄìR-0045), carrying additionally its host-threaded P-0006 `WorkspaceCtx` predicate as a third layer. Registering a built-in in the engine SHALL change only its dispatch path (registry lookup) and rendering (engine-owned renderer), never what it computes (Non-goal 7). The "executes under the report role" half is verified **structurally, not by live session inspection**: the built-in dispatch path routes through the same single executor entry point as user reports (one choke point acquiring the report pool and setting the session context), and the R-0059 telemetry event records `kind` per run; a mid-execution `pg_stat_activity` sample is NOT the acceptance instrument (timing-coupled, inherently flaky). *(Binary-observable: a built-in invoked through the engine produces the same result as its standalone computation on identical data ‚Äî [acceptance test]; the built-in dispatch path routes through the same executor entry point as user reports ‚Äî [construction + review audit].)*
- **R-0060-b (built-in reads fall inside the grant set; no silent grant extension):** Every canonical built-in report's table reads SHALL fall inside the report-role grant set (R-0044-a). A built-in needing a default-denied table SHALL route to the gate as a decision (a P-0020 grant-set amendment), never as a silent grant extension. *(Binary-observable: a static check reconciles each built-in's table reads against the granted set; a built-in read of a non-granted table fails the check ‚Äî [construction + review audit] + [operator SQL surface].)*

### R-0061 ‚Äî Live schema is the contract; prepare-failure is self-announcing; no view layer

*Elaborates Frame D2; carried worked-reference R98/R99. Anchors: D2; P-Defer + Simplicity (no curated view layer at V0); P-TrustworthySignal (the failure states its reason).*

- **R-0061-a (live schema is the contract):** The user-report data contract SHALL be the live substrate schema (the report-role grant set, read-only). No curated `report_*` view layer ships at V0. A definition referencing a table or column that does not exist ‚Äî or that a later migration renamed ‚Äî SHALL fail at statement preparation with a structured error naming the unknown identifier; no row is read. *(Binary-observable: a report referencing a granted table runs; a report referencing a nonexistent table fails at preparation naming the identifier, with no partial output ‚Äî [acceptance test].)*
- **R-0061-b (the deferral is self-announcing):** The dropped curated-view-layer deferral SHALL be self-announcing (DF1 outcome b): a broken definition surfaces its actual reason to its next invoker, and prepare-failure telemetry (R-0059) gives the operator the aggregate signal. *(Binary-observable, per half: a broken definition returns a prepare-failure error to its invoker ‚Äî [acceptance test]; and emits a prepare-failure telemetry event ‚Äî [acceptance test] over the emitted stream; the operator's aggregate signal is a windowed computation over the retained stream ‚Äî [operator telemetry query], R-0059-b.)*
- **R-0061-c (execution adds no schema):** Report **execution** SHALL add no table, column, migration, or writer, and SHALL NOT modify the substrate schema. The only *schema* this feature adds is the one `reference`-cluster definition table (R-0042-a) written through the existing content machinery; the feature's migration set additionally carries the **non-schema security DDL** (role, grants/revocations, RLS policies ‚Äî R-0044-e), which creates no table or column. *(Binary-observable: running any report leaves the substrate schema and row contents unchanged; the feature's migration set adds exactly one table (the definition table) and zero other tables or columns ‚Äî [operator SQL surface] + [construction + review audit].)*

## Scenarios

Given/When/Then for the behaviors this spec locks; each renders an intake success criterion or a Frame trust-boundary resolution.

### Scenario: author-then-invoke, no restart ‚Äî the runtime-extensibility trace (R-0041/R-0042/R-0056/R-0057; SC1)

**Given** a running mnemra-core host, an `admin` token, and no built-in covering "dispatch cost grouped by agent since a date"
**When** the operator authors a report definition with a typed `since` date parameter and a single-`SELECT` body over `dispatch_metrics`, then invokes it with `--param since=2026-01-01`
**Then** the write path validates and registers the definition (R-0042-c); the definition is resolvable on the next lookup with no process restart (R-0041-d); the invocation binds `since` as a typed date (R-0057-b), executes under the read-only report context (R-0043), and returns a shape-tagged `row-list` structured result (R-0056); exit success. [SC1]

### Scenario: destructive user SQL is blocked twice over ‚Äî read-only execution holds adversarially (R-0043; SC2)

**Given** a definition whose body would be `DELETE FROM dispatch_metrics` (rejected at write, R-0042-c) ‚Äî and, for the mechanism-isolation tests, a hand-constructed write attempt exercised directly against the report context via the test-gated direct-executor seam (T2 names it; the write-time gate would otherwise reject the fixture before the execution-layer blocks are ever exercised)
**When** each write-block is tested in isolation (role privileges alone; read-only transaction alone) and together
**Then** each mechanism alone rejects the write before any row is touched (R-0043-c); the database is unchanged; the rejection is structured. [SC2]

### Scenario: workspace isolation on a caller-authored SELECT (R-0045/R-0050; SC2)

**Given** two workspaces each holding rows in every granted table, and a user report whose body SELECTs across content-artifact and measurement tables
**When** the report is invoked under workspace A's `WorkspaceCtx`
**Then** the result contains zero workspace-B rows on every channel ‚Äî enforced by the report-role RLS keyed on the host-set session settings (R-0045-a), with content-artifact tables carrying both workspace-RLS and policy-envelope predicates and measurement tables carrying workspace-RLS only (R-0050); the OC-10 reconciliation confirms every granted table has RLS enabled with the host path bypassing and the report role not (R-0044-d/R-0045-b). [SC2]

### Scenario: session-key re-key attempt fails (R-0046; SC2)

**Given** a definition whose body embeds a `set_config('mnemra.workspace_id', '<other-workspace>', false)` call (in an expression and, in a second fixture, inside a `MATERIALIZED` CTE)

### Scenario: policy-envelope predicates hold on the report path (R-0049/R-0053; SC2)

[SC2]

### Scenario: uniform output ‚Äî empty is success, NULL is explicit, truncation is reported (R-0056; SC3)

**Given** (a) a user report matching zero rows, (b) a report whose result contains a NULL cell, (c) a report matching more rows than the row cap, and (d) a built-in and a user report of the same shape

## API Contract

Payload field names below are binding at contract level; exact JSON serialization is implementation-tier. Every `report.invoke` response carries the shape-tagged structured result with a truncation flag and a parameter echo.

### MCP verbs (host-served, stdio; R-0055)

#### `report.list()` ‚Äî enumerate reports (read verb; both roles)
- **Response:** an array of `{ name, description, kind: "built-in" | "user", output_shape: "scalar" | "matrix" | "row-list" }` ‚Äî filtered to the caller's `visibility`-entitled set plus all built-ins (R-0053-a).
- **Errors:** auth failure (pre-dispatch).

#### `report.get(name)` ‚Äî fetch a report's metadata (read verb; both roles)
- **Response:** `{ name, description, kind, output_shape, parameters: [{ name, type, required, default? }] }`.
- **Dispositions:** a `visibility`-withheld definition ‚Üí **not-found**, indistinguishable from a nonexistent name (R-0053-b); a `dont_use`-killed definition ‚Üí a **metadata-only stub** carrying the policy reason (R-0053-b).

#### `report.invoke(name, params?)` ‚Äî run a report (read verb; both roles)
- **Request:** `{ name, params?: { <param-name>: <bound-value> } }`.
- **Response (shape-tagged structured result):** `{ shape, columns, rows, row_count, truncated: bool, param_echo }` ‚Äî `rows` is `[]` on an empty result (success, R-0056-c); NULL cells are explicit `null` (R-0056-d); `truncated: true` with the capped `row_count` when the row cap fires (R-0048-b/R-0056-e). Report rows carry **no** `{trust, freshness, citation}` envelope (R-0051-a).
- **Errors:** structured not-found (unknown or `visibility`-withheld name); parameter validation error naming the parameter (R-0057-b/-c); unknown-parameter-key error naming the key (R-0057-d); prepare-failure naming the unknown identifier (R-0061-a); timeout error (R-0048-a); **pool-saturated** error naming the condition (R-0048-c ‚Äî returned within the acquire-timeout, retry-safe); all before any partial output.

#### Definition authoring ‚Äî host-served MCP **write** verbs (admin-gated; `read_observer` denied pre-dispatch, R-0052-b)
- Create / update / delete a definition, and policy-field flips, ride the existing content-write internals (R-0042-a). Request carries the definition frontmatter + body; the write path runs the R-0042-c validation before the row lands. Every write and policy flip produces an attributable audit row (R-0054-a).

### Admin-CLI `report` subcommand (R-0055-c)
A convenience frontend over the identical registry/executor/renderer ‚Äî `list` / `get` / `invoke` with `--param k=v` and `--format table|csv` rendered from the same structured result. Never a second execution or enforcement path.

## Threat-test suite

The folded review conditions (D3 H1, M1‚ÄìM3, L1) and the Frame's Stage-3 carry-forwards make these REQUIRED spec content. Each is a binary-observable negative or reconciliation test a reviewer can validate without the author.

**Wiring note (plan-tier task, named here per P-TrustworthySignal TS2's silent-gap discipline):** the embedded-Postgres-touching tests below land in PG-touching test binaries that MUST be enumerated in the justfile's PG-test enumeration (`PG_TEST_FLAGS`, and thereby the `verify-test`/`verify-coverage` recipes) ‚Äî an unwired test binary is a silent-failure class this repo has hit repeatedly (see the justfile's own `PG_TEST_FLAGS` comments); the implementation plan carries this wiring as an explicit task, never an assumed step.

- **T1 ‚Äî seeded re-key attempt fails (R-0046; SC2 negative-acceptance, D3 verbatim obligation).** The suite SHALL seed a `set_config`/materialized-CTE re-key attempt inside a report body that demonstrably fails to alter the effective workspace/role RLS key ‚Äî one fixture with `set_config(...)` in a SELECT expression, one inside a `MATERIALIZED` CTE. The report returns only the invoking workspace's entitled rows. *[acceptance test]*
- **T2 ‚Äî the R-0043 write-block pair proven independently (R-0043-c).** Two negative tests, each isolating one mechanism, with the constructions named: (a) role privileges alone block a write ‚Äî the report role runs the write attempt inside an explicitly read-write transaction; (b) the `READ ONLY` transaction alone blocks a write ‚Äî a write-capable test role runs the attempt inside an explicit `READ ONLY` transaction. Each demonstrably blocks the write before any row is touched. **Named seam:** because the write-time gate (R-0042-c) rejects a write-shaped body before it ever registers, the isolation halves require a **test-only direct-executor seam** ‚Äî invoking the execution context with an arbitrary SQL string, bypassing definition authoring ‚Äî which SHALL be test-gated (compiled only under the test harness), never an always-compiled production hook. *[acceptance test]*
- **T3 ‚Äî OC-10 reconciliation with seeded violations (R-0044-d/R-0045-b).** Per granted table, the reconciliation asserts the full matrix: RLS enabled AND the R-0050 per-family predicates present (content-artifact: workspace-RLS AND policy-envelope; measurement: workspace-RLS) AND the host primary path provably bypasses via ownership-without-`FORCE` AND the report role provably does not AND relkind is base-table-or-`security_invoker`-view. On the function axis: no revocation-class function, no non-`pg_catalog` function beyond the (empty) inventory, no `SECURITY DEFINER` function is executable; the executable surface is GUC-setter-free. The check passes against the true configuration and **fails demonstrably** on each seeded violation: an extra table grant; a granted table missing its workspace-RLS policy; **a content-artifact table carrying workspace-RLS but missing its policy-envelope predicate** (the R-0050 silent-gap case); **a non-`security_invoker` view seeded into the grant set**; a host path that does not bypass; a report-role-executable function outside the permitted structure. **Standing-trigger seeding (the R-0044-e fail-closed consequence, tested end-to-end):** at least one violation is seeded through the standing-trigger path itself ‚Äî a fixture migration adding a granted-family table without its RLS policy ‚Äî asserting the migration-apply call *fails* with the structured reconciliation error, not merely that a reconciliation call site exists in the apply path. *[operator SQL surface] + [acceptance test]*
- **T4 ‚Äî extended differential conformance over the report-role RLS rendering (R-0049-e; carry-forward 1).** The differential run compares the fragment module's RLS-`USING`-rendered predicates against `decide()` over the generated dimension √ó pole √ó undecidable resource set, and **fails demonstrably** on a seeded fragment‚Üî`decide()` divergence on the report-role rendering (not only the serving-`WHERE` form). *[acceptance test]*
- **T5 ‚Äî write-time rejection of data-modifying CTEs (R-0042-c; L1).** A definition body of the form `WITH t AS (DELETE ‚Ä¶ RETURNING *) SELECT ‚Ä¶` is rejected at write time naming the data-modifying CTE; it never registers. *[acceptance test]*
- **T6 ‚Äî PE-4 disposition tests (R-0053-b).** (a) a `visibility`-withheld definition fetched/invoked by an unentitled caller returns not-found, byte-identical to a nonexistent name; (b) a `dont_use`-killed definition returns the metadata-only stub with the policy reason; (c) row-level `visibility` withholds are caller-silent with the `policy_decision_audit` row (R-0049-d); (d) **the both-flags fixture** ‚Äî a definition that is `admin-only` AND `dont_use`-killed returns not-found to a `read_observer` (visibility evaluated first; no stub, no reason leak) and the stub to an `admin`. *[acceptance test] + [operator SQL surface]*
- **T7 ‚Äî enumeration-discipline tests (R-0053-a/-c).** `report.list` omits `visibility`-withheld definitions with no marker or count; reserved-name-collision and same-name-conflict authoring errors disclose nothing across principals. *[acceptance test]*
- **T8 ‚Äî default-denied-table reads are blocked (R-0044-b).** A report selecting from `admin_tokens`, `policy_write_audit`, `policy_decision_audit`, or **`egress_events`** (the retrieval-cluster sensitive-content exemplar) fails with a permission error. *[acceptance test]*
- **T9 ‚Äî function-EXECUTE lock holds, outside-set AND inside-set (R-0044-c/R-0046).** Outside-set: `set_config(...)`, a `pg_stat_*` read, `pg_read_file(...)`, and an in-schema `SECURITY DEFINER` fixture function each fail as the report role; a named permitted function (`percentile_cont`) executes. **Inside-set (the D-KEY "indirectly" clause made testable):** a static sweep over the report role's *executable* function surface asserts it is GUC-setter-free ‚Äî every revocation-class function non-executable, `pg_proc.prosecdef` empty over the executable set, zero non-`pg_catalog` functions beyond the (empty) inventory ‚Äî so no permitted function can set a GUC directly or indirectly. *[acceptance test] + [operator SQL surface]*
- **T10 ‚Äî injection value is bound, not interpolated (R-0047-b).** A `string` parameter bound `'; DROP TABLE ‚Ä¶ ; --` searches for the literal value and leaves the database unchanged. *[acceptance test]*
- **T11 ‚Äî built-in grant-set coverage (R-0060-b).** A static reconciliation of each built-in report's table reads against the report-role grant set passes; a seeded built-in read of a non-granted table fails the check. *[construction + review audit] + [operator SQL surface]*

## pg_catalog hardening checklist

The mitigation artifact the [P-0020](../src/adrs/P-0020-report-execution-context.md) accepted-Low pg_catalog residual is conditioned on ("the spec carries a hardening-checklist item" ‚Äî P-0020 Consequences; the Frame's TB1 residual). Each entry is auditable; P-0020 owns the residual's disposition.

1. **Sampled-value statistics surfaces ‚Äî `pg_stats`, `pg_stats_ext`, `pg_statistic` ‚Äî closed, closure recorded and contingent.** These carry **row-derived sampled column values** (`most_common_vals`, `histogram_bounds`), outside the "schema shape, not row data" boundary the pg_catalog acceptance drew ‚Äî so their closure is recorded here, not left incidental. `pg_statistic` (the base catalog) is superuser-read-only by default: unreadable to the report role. The `pg_stats`/`pg_stats_ext` views carry the built-in `(NOT relrowsecurity OR NOT row_security_active(oid))` guard: for every RLS-enabled granted table (R-0045-a), the view returns **zero rows** to the report role ‚Äî and the report role cannot disable row security (`SET` is a utility statement structurally absent under R-0047-a; `set_config` is revoked, R-0044-c). **The closure is contingent on R-0045 RLS-enablement per granted table, which the OC-10 reconciliation asserts at its standing trigger (R-0044-d/-e)** ‚Äî a table that lost its RLS fails reconciliation before its sampled stats could leak. **Version assumption, pinned:** the `pg_stats` guard exists in PostgreSQL ‚â• 9.5; the embedded engine is pinned `=16.4.0` (`EMBEDDED_PG_VERSION`, the P-0010 engine) ‚Äî re-verify this entry on any engine-version change. *(Auditable: `SELECT count(*) FROM pg_stats WHERE tablename = '<granted table>'` as the report role returns 0 ‚Äî [operator SQL surface].)*
2. **Server-info / statistics / file-access functions ‚Äî revoked.** The `pg_stat_*` view+function family, `pg_ls_*`, `pg_read_file`/`pg_read_binary_file`/`pg_stat_file`, `pg_current_logfile` are revocation-class (b) under R-0044-c; asserted by T9 and the reconciliation. *(Auditable: T9.)*
3. **What can be revoked without breaking query planning:** the function-EXECUTE revocations above (classes (a)/(b) and the `SECURITY DEFINER` bar) are safe ‚Äî the planner and executor do not invoke them on the report role's behalf. **What cannot:** relation-metadata catalog read (`pg_class`, `pg_attribute`, `pg_namespace`, `information_schema`) ‚Äî catalog readability is assumed by client drivers, tooling, and backend operation; revoking it is unsupported hardening with breakage risk. Disposition: this remainder IS the accepted-Low residual ‚Äî disclosure bounded to the schema *shape* (table/column names) of the shared-schema instance, no row data ‚Äî and it stands, per P-0020.
4. **GUC read surface ‚Äî no revocation needed.** `current_setting(...)`/`pg_settings` reads expose the caller's *own* session keys (its own workspace id and role) ‚Äî not cross-tenant disclosure. A `pg_settings` UPDATE is closed by the **write-time gate** (R-0042-c admits only `SELECT`/`WITH‚Ä¶SELECT` bodies and rejects data-modifying CTEs, so the statement never registers) plus the **`set_config` EXECUTE revocation** (R-0044-c) ‚Äî `pg_settings` is an updatable view whose rule rewrites the UPDATE into `SELECT set_config(‚Ä¶)`, so the R-0043 write-blocks are *not* what closes this path (the rewritten form carries no DML node and needs `EXECUTE`, not table-write privilege). Reading is left permitted.
