## Purpose / context

The host has a verified-load mechanism and an MCP server handler, but it has **never actually run**: the production entry point `run()` is an empty `Ok(())` stub (`libs/mnemra-host/mnemra_host.rs:36`), the embedded signing root is a non-production placeholder (`libs/mnemra-host/signing/root_material.rs:35-40`), and the on-disk plugin manifest is placeholder-signed (`plugins/mnemra-echo/manifest.toml:77-96` — `public_key = "ROOT"`, `sig_bytes = "PLACEHOLDER_SIG"`, no content-hash field). This spec designs the change that makes the host **start as a live stdio MCP server backed by real plugin signing**, so the V0 reference workload runs end-to-end with no fallback to prior tooling.

This is the closed world the Frame locked. This spec adds two new requirement slots — **R-0021** (the `{{R-ContentHashBinding}}` slot: the supply-chain binding behavior) and **R-0022** (the `{{R-RunAssembly}}` slot: the `run()` assembly behavior) — and **amends / instantiates / composes** a set of existing parent-spec requirements by supersession-by-reference (it does not re-derive the signing primitive, custody model, transport, or startup ordering — all locked). The manifest **schema** change (the new content-hash field) is governed by an in-place amendment to [P-0003](../src/adrs/P-0003-plugin-manifest.md) (the `{{P-ManifestContentHash}}` slot), authored with this change; **R-0021** governs the runtime **behavior**, P-0003 governs the **schema** — the Frame's slot split.

**Code surfaces this spec's requirements land in** (load-bearing anchors): `libs/mnemra-host/signing/root_material.rs` (embedded root); `libs/mnemra-host/signing/verify.rs:337-344` (the signed-body slice the content-hash field must fall within); `libs/mnemra-host/startup/pool_population.rs:102-110,126,141-143` (the verified-load gate, the path-read load, and the intentional-decoupling comment this spec retires); `libs/mnemra-host/mnemra_host.rs:36` (`run()`); `libs/mnemra-host/mcp/server.rs` (the rmcp `ServerHandler`); `libs/mnemra-host/storage/postgres.rs:82` (`start_embedded()`); a new `/health` module; the `verify-build` / `verify-smoke` Justfile recipes; `plugins/mnemra-echo/manifest.toml` (real signing + the content-hash field — an implementation-tier artifact, not in this spec's touch scope).

## Requirements

RFC-2119 keywords used throughout. `SHALL`/`MUST` = mandatory. `SHALL NOT`/`MUST NOT` = prohibited. `SHOULD` = preferred. `MAY` = optional.

### R-0021 — Component content-hash binding (the `{{R-ContentHashBinding}}` slot)

*Governs the supply-chain-integrity **behavior**: the manifest binds the component bytes, and the verified-load gate enforces the binding fail-closed. The manifest **schema** field this behavior reads is governed by the [P-0003 amendment 2026-06-30](../src/adrs/P-0003-plugin-manifest.md) (`{{P-ManifestContentHash}}`). Keystone anchor: `P-SecurityLayered` — manifest-signature verification (provenance of the declaration) and component-byte binding (integrity of the artifact) are **distinct, independently load-bearing layers**; real signing instantiates the first and leaves the second absent (`architecture-values.md:31` "each layer independently load-bearing"; `constraint-edges.md:42`). Co-anchors: R-0005-a/-b (synchronous fail-closed, structured error), R-0005-h (core-by-provenance forces the field into the signed body). [Frame D2/D3, QA-1]*

- **R-0021-a (content-hash field in the signed canonical body; named strong algorithm):** The signed canonical manifest body SHALL carry a `[component]` section declaring `hash_alg` and `hash`, where `hash` is the digest of the **component (`.wasm`) bytes** under `hash_alg`. The `[component]` section SHALL appear **before** the `\n[signature]` marker so it falls within the byte-exact payload the signature covers (the slice computed at `libs/mnemra-host/signing/verify.rs:337-344`) — a hash in the unsigned `[signature]`-adjacent region would let an attacker swap both the component and its declared hash (R-0005-h core-by-provenance). `hash_alg` SHALL be one of the strong set `{"blake3", "sha256", "sha384", "sha512"}`; the V0 locked value is **`blake3`** (reusing the in-tree BLAKE3 primitive already mandated for admin-token hashing, R-0008-b — Simplicity / `P-StackDiscipline`, no new dependency). `MD5` and `SHA-1` SHALL be **rejected** structurally — a manifest declaring `hash_alg = "md5"` or `"sha1"` (or any value outside the strong set) SHALL fail to load. The strong set is the schema's **forward-allowance**; at V0 the only value any manifest carries (and the only one a load-path test exercises on the accept side) is `blake3` — `sha256`/`sha384`/`sha512` are accepted by the schema but not exercised by a V0 manifest, so the V0 acceptance test asserts **`blake3` accepts** and the **rejection boundary** (`md5`/`sha1`/any out-of-set value rejects). *(Binary-observable: the verifier's signed-body slice for a valid manifest contains the `[component]` table; a `blake3` manifest loads and a manifest with `hash_alg = "md5"`, `"sha1"`, or any non-strong-set value is rejected at load; the same hash-algorithm constraint is locked in the [P-0003 amendment](../src/adrs/P-0003-plugin-manifest.md).)* [Frame D2; R-0005-h]

- **R-0021-b (producer computes-and-embeds over the exact load artifact, before signing):** The signing producer (the V0 ceremony, Frame D4) SHALL compute `hash` over the **exact component artifact that the runtime will load** (the signed component artifact, per the [§ Constraints](#constraints) integrity-gated-load-path clause — *not* a fresh `target/` rebuild), embed it in the `[component]` section of the signed body, and **then** sign the canonical body. The artifact the producer hashes SHALL be **byte-identical** to the artifact the runtime loads — the producer must not hash a build-intermediate, a differently-optimized, or a re-serialized copy. *(The byte-identity discipline is **ceremony/code-review-enforced** — a black-box test cannot observe which copy the producer hashed; the externally-observable property is the round-trip below.)* *(Binary-observable: for the real-signed `mnemra-echo`, recomputing `hash_alg` over the on-disk loaded `.wasm` equals the `[component].hash` in the signed manifest body; an artifact whose bytes differ from the hashed one fails R-0021-e at load.)* [Frame D2/D4]

- **R-0021-c (field-presence fail-closed — mandatory, not skip-on-absent; parsed only from the signed slice):** The verified-load gate SHALL require the `[component].hash` field to be **present** in the signed body; **absence SHALL be rejected fail-closed** (a missing content-hash is a load rejection, not a "no hash declared → skip the check" path). The gate SHALL parse `[component].hash_alg`/`hash` **only from the byte slice the signature covers** — the slice `extract_signed_payload` returns (`libs/mnemra-host/signing/verify.rs:337-344`, the bytes before `\n[signature]`) — and a `[component]` table appearing **outside** that slice SHALL NOT satisfy the field-presence requirement and SHALL cause rejection (complete-mediation: the enforced value is the signed value, never an unsigned-region copy). *(Today the `toml` crate errors on a duplicate `[component]` table, so a signed-body table plus an unsigned-region table already fails closed; this clause pins that the field is read from the signed slice as an explicit invariant rather than relying on the parser's duplicate-table behavior.)* Because the manifest stays at `schema_version: 1` (no version bump — see [P-0003 amendment 2026-06-30](../src/adrs/P-0003-plugin-manifest.md)), the `schema_version` number **cannot** signal the field's presence; the parser SHALL therefore enforce presence directly. This presence rule and the `schema_version: 1` disposition are **one decision**, recorded together in the P-0003 amendment. *(Binary-observable: a signed manifest whose body omits `[component].hash` is rejected at load with the R-0021-f structured error — not loaded with the hash check skipped; a manifest carrying `[component].hash` only in an unsigned position after `\n[signature]` is likewise rejected, not accepted.)* [Frame D2; required-lock #2 + #6; P-0003 amendment]

- **R-0021-d (single-read component load — TOCTOU; code-review-enforced):** On the verified-load path the runtime SHALL perform the steps in this exact order, so the requirement is self-contained: **(1) read** the component bytes **once** into an in-memory buffer (`std::fs::read` of the component path); **(2) hash** — compute `hash_alg` over **that buffer**; **(3) compare** the result to the signed `[component].hash` (R-0021-e); **(4) instantiate** the component via `wasmtime::Component::from_binary(engine, &bytes)` **from the same buffer**. A path-read-twice pattern — computing the hash from one read and then instantiating via `wasmtime::Component::from_file(...)` (which re-opens and re-reads the path) — SHALL NOT be used on the verified-load path, because the second read reopens the swap window between the hash check and the load. The current load site `Component::from_file(pool.engine(), &component_path)` (`libs/mnemra-host/startup/pool_population.rs:141-143`) SHALL be replaced by the read-once-then-`from_binary` pattern. **Verification surface:** that the gate instantiates from the *same buffer it hashed* (i.e. calls `Component::from_binary` and not `Component::from_file`) is an **implementation property a black-box test cannot assert non-flakily** — it is **enforced at code review** (the security reviewer's domain), not by a black-box scenario. The externally-observable consequence — a `.wasm` swapped after signing is rejected — is covered black-box by the keystone swap scenario (R-0021-e), which is what a test can deterministically assert. *(Code-review-observable: the verified-load path reads once, hashes that buffer, compares, then calls `Component::from_binary` on the same `&[u8]`, and contains no `Component::from_file` call on the verified-load path.)* [Frame D3; required-lock #1]

- **R-0021-e (recompute-and-reject on mismatch, fail-closed before any pool/instance):** After manifest-signature verification returns `Ok` (R-0005-a), the gate SHALL recompute the loaded component buffer's hash (per R-0021-d) and compare it to the signed `[component].hash` **parsed from the signature-covered slice** (R-0021-c — never an unsigned-region copy). On **mismatch** the load SHALL be **rejected**, with **no `PluginPool` and no plugin instance constructed** — the same verify-first / build-nothing-on-failure discipline as the signature gate (R-0005-a) — and SHALL return a structured error naming the plugin (R-0005-b); no raw filesystem or wasmtime internals SHALL be echoed to the caller. The hash check SHALL run **after** the signature check (a component whose manifest signature does not verify never reaches the hash recompute). *(Binary-observable: with the correct component `populate_verified_pool` returns `Ok` (signature verifies **and** recomputed hash == signed hash); with a swapped component (different bytes, same valid manifest) it returns `Err` on the hash path; on the `Err` path no pool and no instance exist; the error names the plugin and leaks no FS/wasmtime internals.)* [Frame D3; R-0005-a/-b; QA-1]

- **R-0021-f (distinct hash-mismatch error variant — tamper-distinguishable):** A content-hash mismatch SHALL surface a **distinct** structured error variant — it SHALL NOT reuse `StartupError::ComponentLoad` (`libs/mnemra-host/startup/pool_population.rs:54`, today returned for a benign "component could not be loaded from the target dir" failure). A new variant (e.g. `StartupError::ComponentHashMismatch`; the enum is `#[non_exhaustive]`, so adding a variant is non-breaking) SHALL carry the plugin name and SHALL render a `Display` / structured-log form **distinguishable** from a benign component-load failure. Rationale: a hash mismatch is a **tamper signal**, and the runtime backstop layer must make a tamper event observable apart from an operational load error in logs and metrics (`P-SecurityLayered` runtime sub-layer — the backstop control must be legible, not folded into a generic failure). *(Binary-observable: a swapped-component load returns the distinct `ComponentHashMismatch`-class variant, not `ComponentLoad`; the two are matchable as separate arms and render distinct `Display` strings.)* [Frame D3 note; required-lock #7; `P-SecurityLayered`]

*Test-infrastructure note (R-0021 verification surface — not a new lock).* The adversarial R-0021 scenarios below (field-absence R-0021-c, keystone measure-5 unsigned-position hash, algorithm-rejection R-0021-a) require **validly-signed** adversarial manifests, but the V0 ceremony private key is not available to red-phase authors. The test-key-substitution seam **already exists** and SHALL be used: `populate_verified_pool(manifest_toml, root_material)` (`libs/mnemra-host/startup/pool_population.rs:126-129`) takes `root_material: &[u8]` as a parameter (its doc-comment already records that "callers (including tests) inject a per-run generated key"). Red-phase tests SHALL generate a **test root keypair**, sign adversarial manifests with the **test private key**, and pass the **test public key** as `root_material` — constructing validly-signed adversarial inputs without the real ceremony key. The real embedded root (`signing::root_material::ROOT`) is exercised only on the production path (R-0022-e) and the real end-to-end smoke (R-0022-c). *(This is a constraint on test infrastructure — the implementation tier supplies the keypair-generation/signing fixture; the seam is named here so the red-phase author is not blocked on the ceremony key.)*

### R-0022 — Production `run()` assembly (the `{{R-RunAssembly}}` slot)

*Governs the `run()` production-startup **behavior**: `run()` is an empty `Ok(())` stub today (`libs/mnemra-host/mnemra_host.rs:36`), so the composition behavior has no existing home — `run()` **composes** the already-locked ordering requirements; it does not amend them. Anchors: `P-LockContract` (the startup order is an intrinsic invariant that locked at the stage it was defined; `run()` is its instantiation), `P-InstrumentBefore` (the readiness instrument comes online before the host's first real run). Composes R-0004-g, R-0002-c, R-0016-a, R-0013-a, R-0010-a/-e (parent spec). [Frame D6/D7/D8/D9; intake SC5/SC6; QA-4/QA-5]*

- **R-0022-a (production startup in the locked order; each boundary independently assertable):** `run()` SHALL perform production startup composing the locked ordering, with each ordering boundary **independently assertable**. Boundaries are asserted by **deterministic readiness signals**, not by racing two concurrent events on the wall clock: an MCP request issued before the host reaches its accept-ready state is **deterministically refused** (the listener is not yet accepting), and `/health` answers while MCP is in that not-yet-accepting state — so the "before" relations below are observed against known pre-/post-readiness states, not by timing-window comparison.
  - **(5-pre)** the **startup file-mode invariant check** (R-0005-f, instantiated over the admin-token file per [§ Supersession-by-reference](#supersession-by-reference)) runs **first** — **before any listener (`/health` or MCP) binds** — and **fails the host closed** (non-zero exit, no listener bound) if the checked file is world-readable;
  - **(5a)** the `/health` loopback listener binds **before config load** and **before** the MCP server accepts requests (R-0004-g);
  - **(5b-i)** all **seven** builtins initialize **before any plugin is loaded** (R-0002-c); this ordering is made observable as a **startup-failure property**: a builtin-init failure causes `run()` to **fail before any plugin load is attempted** — the host exits non-zero, **no `PluginPool` is constructed**, and **no plugin-load log line is emitted** (rather than requiring a builtin-readiness field in the `/health` body);
  - **(5b-ii)** the verified plugin pool is populated (R-0016-a, 3–5 instances) **before** MCP accept;
  - **(5c)** a storage pool is sourced and schema initialization runs (R-0022-d) **before** the server is constructed and begins serving; if storage init **fails**, `run()` **fails closed before MCP accept** (the server is never constructed and no MCP request is served), and because `/health` is already bound (5a) it answers with its R-0004-g detail body reporting a **not-ready** state (the host does not proceed to serving on a storage-init failure);
  - **(accept-last)** **no** MCP request is accepted before steps 5-pre–5c complete.
  *(Binary-observable: with a world-readable admin-token file the host exits non-zero with no listener bound (5-pre); `/health` answers on the loopback listener while MCP is not yet accepting (5a); a builtin-init failure yields a non-zero exit with no `PluginPool` constructed and no plugin-load log line (5b-i); an MCP request issued before the accept-ready signal is refused (5b-ii / accept-last); on a storage-init failure the server is never constructed and `/health` reports not-ready (5c); each boundary is assertable in isolation.)* [Frame D6; intake SC5; QA-4]

- **R-0022-b (`/health` loopback listener stood up and bound first):** The `/health` loopback listener — fully specified by parent R-0004-g (loopback `127.0.0.1` only, `MNEMRA_HEALTH_PORT` default `8877`, `GET /health` only, the structured detail body, the non-loopback-bind tripwire) — SHALL be **built** (the previously-planned Task-25 surface) and **bound first** by `run()` (the 5a boundary). This requirement is **build + wire**, not a change to R-0004-g's text. *(Binary-observable: a `GET /health` on `127.0.0.1:8877` (or `MNEMRA_HEALTH_PORT`) returns the R-0004-g structured detail body, and answers before the MCP server accepts a request.)* [Frame D7; R-0004-g; `P-InstrumentBefore`]

- **R-0022-c (production stdio serve-loop; stdio only, no HTTP MCP transport; the smoke gate exercises the real signed-artifact path):** `run()` SHALL stand up a production **stdio** MCP serve-loop over the existing `MnemraMcpServer` `rmcp::ServerHandler` (`libs/mnemra-host/mcp/server.rs`), serving over stdio at V0 (R-0010-a, P-0012 rmcp v1.7.0). Streamable-HTTP MCP transport SHALL NOT be activated (R-0010-e); the only non-stdio listener `run()` stands up is the loopback `/health` server, which is **not** an MCP transport (R-0004-g). The async runtime this serve-loop needs is **already present** (`cmd/mnemra/main.rs:10-11` is `#[tokio::main] async fn main`); `run()` is sync today (`libs/mnemra-host/mnemra_host.rs:36`), and making it async (or driving it on the existing runtime) is an implementation-tier change, not a new design decision.

  **Smoke gate (no rebuild in the gate path).** The **real** `verify-smoke` recipe SHALL exercise the host starting and serving end-to-end **over the production integrity-gated path** — the real embedded root (R-0005-d), the real on-disk signed manifest (R-0022-e), and the **signed component artifact** (the exact `.wasm` whose hash is in the signed manifest, per the [§ Constraints](#constraints) integrity-gated-load-path clause). `verify-smoke` SHALL NOT **rebuild** the component in its gate path — it SHALL NOT carry the `plugin` recipe as a prerequisite and SHALL NOT invoke `cargo build … -p mnemra-echo --target wasm32-wasip2` — because a fresh `target/` rebuild is **not** the signed artifact (the signed hash binds specific bytes; a non-reproducible rebuild produces different bytes and would false-reject via R-0021-e). The currently-scaffolded `verify-smoke` (the `if true` placeholder in the `justfile`, the previously-planned smoke surface) is a **false green**: a stubbed `verify-smoke` SHALL NOT be treated as gating M2 — M2 SHALL NOT be considered shipped while `verify-smoke` is the `if true` stub; the real end-to-end smoke is a **merge-gate precondition for M2**.

  *(Binary-observable, serve-loop: the binary starts a live stdio MCP server that answers an MCP request end-to-end; no HTTP MCP transport is opened; the sole non-stdio listener is the loopback `/health`.)* *(Binary-observable, smoke gate: the `verify-smoke` recipe body is not the `if true` placeholder and invokes a real fallible end-to-end command; it has no `plugin` prerequisite and runs no `cargo build` of `mnemra-echo`; the host it starts loads the signed component artifact and recomputing `blake3` over the loaded `.wasm` equals the signed `[component].hash`; the run returns `Ok` end-to-end.)* [Frame D8; R-0010-a/-e; intake SC6 + Hard constraint; QA-5; `P-SecurityLayered`]

- **R-0022-d (storage pool from `start_embedded()`; schema init; NOT the LLM-key/config pathway):** The production storage pool SHALL be sourced from `PostgresStorage::start_embedded()` (`libs/mnemra-host/storage/postgres.rs:82` — the embedded-Postgres engine startup path, P-0010), and schema initialization SHALL run (R-0013-a: bootstrap on the embedded engine, enable bundled `pgvector`, create substrate tables + the `default` workspace), **before** the server is constructed (the 5c boundary). This SHALL NOT use the R-0014 LLM-key / embedding-config pathway (R-0014 governs that and does not apply here). **Verification surface:** the *call-ordering* (storage init + schema init complete before `MnemraMcpServer` is constructed) is an internal ordering property **enforced at code review**, not by a black-box test; the externally-observable property is that the served host is **storage-backed** — after startup, an MCP verb that reads/writes the substrate succeeds against the initialized schema, and the storage-init-failure path fails closed before serving (R-0022-a 5c). *(Binary-observable: after a successful start, an MCP request exercising a storage-backed verb succeeds against the `default` workspace and `mnemra init` semantics (R-0013-a) hold; the LLM-key/config pathway is not on the storage-pool path. Code-review-observable: `start_embedded()` + schema init precede `MnemraMcpServer` construction.)* [Frame D9; P-0010; R-0013-a]

- **R-0022-e (decoupling-reversal — `run()` feeds the real on-disk manifest to the verified-load gate):** `run()` SHALL feed the **real on-disk manifest** (`plugins/mnemra-echo/manifest.toml`) to the verified-load gate `populate_verified_pool` (`libs/mnemra-host/startup/pool_population.rs:126`), **retiring** the intentional V0 manifest/component **decoupling** documented at `libs/mnemra-host/startup/pool_population.rs:102-110` (the "verifies the parameter it is passed … does NOT read or verify the on-disk manifest … This decoupling is intentional V0 design; do not 'fix' it" comment). Stage 3 **reverses** that decoupling: the production gate SHALL read and verify the real on-disk manifest (real-signed per the R-0005-d supersession, below) against the real embedded root, then enforce the R-0021 content-hash binding on the component **bytes the manifest binds by hash** — the manifest binds the component's *bytes* (via `[component].hash`), **not** its path or name; the schema adds no path/name field (the P-0003 amendment adds only `hash_alg`/`hash`), and a path/name field is an out-of-scope multi-plugin concern. The component the gate loads SHALL be the **signed component artifact** (per the [§ Constraints](#constraints) integrity-gated-load-path clause) — the bytes whose hash the manifest binds — **not** a fresh `target/` rebuild; this is what makes R-0021-e a true integrity check (recompute over the loaded signed artifact == signed hash) rather than a build-reproducibility check. The "deferred production wiring — not yet called by `run()`" note at `pool_population.rs:112-116` SHALL be resolved by this wiring. *(Binary-observable: the production startup path reads `plugins/mnemra-echo/manifest.toml` from disk and passes its bytes to `populate_verified_pool`; the intentional-decoupling comment block is removed; with the real-signed on-disk manifest the gate returns `Ok` against the real embedded root loading the signed component artifact, and a swapped component fails per R-0021-e.)* [Frame D6/§5 keystone; required-lock #5]

## Scenarios

Given/When/Then for the behaviors this spec locks. The keystone scenario (the supply-chain swap) carries all five sub-measures from Frame QA-1.

### Scenario: Supply-chain component swap rejected — the keystone (R-0021)

**Given** the verified-load gate `populate_verified_pool` with the R-0021 content-hash binding active and the manifest signature verifying against the real embedded root
**When** an attacker swaps the on-disk component (`.wasm`) for a different one **after** the manifest is signed, leaving the validly signed manifest untouched, and the host attempts to load the plugin
**Then** all of the following hold:
1. With the **correct** component, `populate_verified_pool` returns `Ok` — the signature verifies **and** the recomputed hash == the signed `[component].hash`.
2. With the **swapped** component (different bytes, same valid manifest), it returns `Err` on the **hash-mismatch** path **not** the signature path (the signature still verifies).
3. On the mismatch path, **no `PluginPool` and no plugin instance are constructed** (fail-closed; the gate fires before construction — R-0021-e).
4. The mismatch returns a **structured error naming the plugin** (R-0005-b), with **no** raw filesystem / wasmtime internals echoed to the caller, via the **distinct** `ComponentHashMismatch`-class variant (R-0021-f), not `ComponentLoad`.
5. Moving the content-hash to an **unsigned** position (outside the `\n[signature]` slice) does not defeat the binding — such a manifest fails (the signature no longer covers the hash, or the field is absent from the signed body → R-0021-c fail-closed). [Frame QA-1]

### Scenario: Content-hash field absent from the signed body — fail-closed (R-0021-c)

**Given** a manifest whose signature verifies against the real embedded root but whose **signed body omits** the `[component].hash` field
**When** the host attempts to load the plugin
**Then** the load is **rejected** with the R-0021-f structured error (a missing field is a load rejection, **not** a "no hash declared → skip the check" path); because the manifest is `schema_version: 1`, the `schema_version` number cannot signal field presence, so the parser enforces presence directly.
**And given** instead a manifest carrying `[component].hash` **only in an unsigned position** (after the `\n[signature]` marker, outside the slice `extract_signed_payload` returns), **when** the host attempts to load it, **then** the load is **rejected** — the unsigned-region copy does **not** satisfy the field-presence requirement (the enforced value is read only from the signature-covered slice; complete-mediation). [R-0021-c; required-lock #2 + #6]

### Code-review property (NOT a black-box scenario): Single-read load closes the swap window — TOCTOU (R-0021-d)

This is enforced at **code review** (the security reviewer's domain), not by a black-box test — which copy the gate instantiates from is an implementation property a black-box test cannot assert non-flakily. The externally-observable consequence (a `.wasm` swapped after signing is rejected) is covered black-box by the keystone swap scenario above; this entry records the code-review check, not a test scenario.

**Code-review check:** On the verified-load path the gate reads the component bytes **once** into a buffer, hashes **that buffer**, compares to the signed `[component].hash`, and instantiates via `Component::from_binary(engine, &bytes)` from the **same buffer**; it does **not** re-open the path via `Component::from_file` after hashing — there is no second read between the hash check and instantiation in which a swap could land, and the verified-load path contains no `Component::from_file` call. [R-0021-d; required-lock #1]

### Scenario: Only the pinned real root can ship — build gate (R-0018-f amended)

**Given** the `verify-build` recipe with the **pass-iff-pinned-real-root** gate (positive/allowlist; mechanism per the R-0018-f supersession entry), invoked by CI (`just ci`)
**When** a build is attempted while the embedded root is the bootstrap placeholder — **or any key other than the pinned real root** (e.g. a dev key)
**Then** `verify-build` returns **non-zero** and emits `GATE build FAIL` (no binary carrying a non-pinned root is produced); and **only** when the embedded root **equals the pinned real root** does `verify-build` return **zero** and emit `GATE build PASS`. Because the check runs inside `just ci`, a non-pinned root cannot reach `main` — and because the assertion is positive, a third/dev key does not slip through the implicit-allow gap a blocklist would leave. [Frame QA-2; R-0018-f amended]

### Scenario: Real-signing round-trip (R-0005-d instantiated)

**Given** `verify_plugin` (`libs/mnemra-host/signing/verify.rs:172`) with the **real** root embedded and the `mnemra-echo` manifest real-signed by the V0 ceremony
**When** the host verifies the manifest
**Then** the genuine real-signed manifest verifies to `Ok(CoreStatus::Core)` against the **real** embedded root (not the bootstrap placeholder); a manifest with any byte mutated in the signed body fails verification (the byte-exact slice at `verify.rs:337-344` is what is signed); and the on-disk `plugins/mnemra-echo/manifest.toml` no longer carries `public_key = "ROOT"` / `sig_bytes = "PLACEHOLDER_SIG"`. [Frame QA-3]

### Scenario: Production startup ordering (R-0022-a)

**Given** `run()` performing production startup
**When** the host binary is started
**Then** the locked order holds, each boundary independently assertable, ordering observed against **deterministic readiness states** (not wall-clock races): **(5-pre)** the file-mode invariant check (R-0005-f, over the admin-token file) runs **first** — with a world-readable admin-token file the host exits non-zero **before any listener binds**; **(5a)** `/health` answers on `127.0.0.1:8877` (or `MNEMRA_HEALTH_PORT`) while MCP is **not yet accepting**, before config load (R-0004-g); **(5b-i)** all **seven** builtins initialize **before** any plugin load (R-0002-c) — observed as a startup-failure property: a builtin-init failure yields a **non-zero exit with no `PluginPool` constructed and no plugin-load log line**; **(5b-ii)** the verified plugin pool (3–5 instances) is populated **before** MCP accept — an MCP request issued before the accept-ready signal is deterministically refused (R-0016-a); **(5c)** a storage pool is sourced via `start_embedded()` and schema init runs before the server is constructed — and on a storage-init **failure** the server is never constructed, the host fails closed before MCP accept, and `/health` (bound at 5a) reports **not-ready**; and **no** MCP request is accepted before steps 5-pre–5c complete. [Frame QA-4]

### Scenario: Live serve end-to-end (R-0022-c, serve-loop + smoke gate)

**Given** the production stdio serve-loop with a populated verified pool, loading the **signed component artifact** (not a `target/` rebuild — § Constraints integrity-gated-load-path)
**When** the host binary is run and an MCP client issues a request over stdio
**Then** the **real** `verify-smoke` recipe passes (it is **not** the `if true` stub, and it does **not** rebuild the component — no `plugin` prerequisite, no `cargo build` of `mnemra-echo`): the binary starts a live stdio MCP server that answers the MCP request end-to-end (intake SC6); recomputing `blake3` over the loaded `.wasm` equals the signed `[component].hash` (the bytes served are the bytes signed); the transport is **stdio** (no HTTP MCP transport is opened — R-0010-a/-e); the only non-stdio listener is the loopback `/health` (not an MCP transport, R-0004-g). [Frame QA-5; intake SC6 + Hard constraint; R-0022-c smoke gate]

### Scenario: Key-custody invariant — degenerate same-machine case (R-0005-c/-f instantiated)

**Given** the host starts on a machine that is **also** the build host (single-developer dogfood), with the V0 ceremony's key placement and the R-0005-f startup file-mode check
**When** the host starts
**Then** the private signing key file is **mode 600** and resides **outside** every directory the runtime reads; the runtime performs **no** key-fetch at startup or load (only the embedded public key is used — R-0005-d); and the startup file-mode invariant check (R-0005-f) refuses to start if the admin-token file is world-readable. [Frame QA-6]

## Data Model

### Manifest schema delta — the `[component]` content-hash section

The V0 plugin manifest schema (governed by [P-0003](../src/adrs/P-0003-plugin-manifest.md), amended in place by this change) gains a `[component]` section in the **signed canonical body** (before the `\n[signature]` marker — within the slice the signature covers, `libs/mnemra-host/signing/verify.rs:337-344`). `schema_version` stays **`1`** (reasoning + the mandatory-field pairing are locked in the [P-0003 amendment 2026-06-30](../src/adrs/P-0003-plugin-manifest.md)).

```toml
# ... [plugin], [verbs], [content_types], [state_scopes], [host_fns] (unchanged) ...

[component]
# Content-hash binding of the component (.wasm) bytes (R-0021). In the SIGNED body
# (before [signature]) so the signature covers it (R-0005-h core-by-provenance).
hash_alg = "blake3"            # one of {blake3, sha256, sha384, sha512}; V0 = blake3 (R-0008-b in-tree). MD5/SHA-1 rejected.
hash     = "<lowercase-hex digest of the component .wasm bytes under hash_alg>"   # mandatory; absence is fail-closed (R-0021-c)

[signature]
# unchanged — populated at build time by the signing chain (P-0005); covers everything above this marker.
```

| Field | Type | Constraints | Notes |
|-------|------|-------------|-------|
| `[component].hash_alg` | `str` (TOML) | One of `{"blake3","sha256","sha384","sha512"}`; `"md5"`/`"sha1"`/other → reject (R-0021-a) | V0 locked value `"blake3"` |
| `[component].hash` | `str` (TOML) | **Mandatory** (absence → fail-closed, R-0021-c); lowercase-hex digest of the component `.wasm` bytes under `hash_alg` | In the signed body; producer computes over the byte-identical loaded artifact (R-0021-b) |

*The `[component]` section is the only schema change; all other manifest sections are unchanged. The concrete TOML serializer/parser changes and the re-signed `plugins/mnemra-echo/manifest.toml` are implementation-tier artifacts, not in this spec.*

## Constraints

Inherited from the parent spec and the locked Frame; restated where load-bearing for this increment.

- **Signing primitive:** Ed25519 via `ed25519-dalek` (P-0005, locked).
- **Custody:** build-host on-disk key, mode 600, off the runtime-read path; only the public key embedded; no runtime key-fetch (R-0005-c/-d). The minimal V0 ceremony honors these even when build host == run host. "Off the runtime-read path" is checkable against the **enumerated runtime-read directories** (R-0005-c supersession entry): the plugin manifest dir, the signed component artifact dir, and the admin-token file dir; the private key stays outside all three.
- **Content-hash location:** the content-hash MUST live in the **signed canonical body** (R-0005-h core-by-provenance), and SHALL be read **only from the signature-covered slice** (R-0021-c — `extract_signed_payload`, `libs/mnemra-host/signing/verify.rs:337-344`); an unsigned-region copy SHALL NOT satisfy the check.
- **Content-hash algorithm:** SHA-256 or stronger / BLAKE3; MD5 and SHA-1 are banned (R-0021-a). V0 value: BLAKE3 (R-0008-b in-tree; no new dependency). The strong set is forward-allowance; the only value a V0 manifest carries is BLAKE3.
- **Integrity-gated load path (the bytes run == the bytes signed):** The integrity-gated load path — production `run()` (R-0022-e) and the real `verify-smoke` gate (R-0022-c smoke gate) — SHALL load the **signed component artifact**: the exact `.wasm` whose BLAKE3 hash is embedded in the signed manifest (`[component].hash`), held at a stable, version-controlled location distinct from the rebuilt `target/wasm32-wasip2/release/mnemra_echo.wasm` output (the exact committed path is implementation-tier; today `echo_component_path()` at `libs/mnemra-host/startup/pool_population.rs:165-172` resolves the `target/` rebuild — that resolution moves to the signed artifact for the production path). The integrity-gated path SHALL NOT **rebuild** the component (the `verify-smoke` recipe SHALL NOT depend on `plugin` or invoke `cargo build … -p mnemra-echo`). A **deterministic / reproducible `wasm32-wasip2` build is NOT required** at V0; the binding's correctness rests on loading the signed artifact, not on a rebuild reproducing identical bytes. *(Anchors: `P-SecurityLayered` — the gate verifies the artifact actually run, the security-meaningful unit; Simplicity / `P-StackDiscipline` — no reproducible-build toolchain burden at V0; `P-MinBlastRadius` — the integrity gate's correctness does not depend on the whole build toolchain being bit-reproducible. See [§ Decisions of note](#decisions-of-note).)*
- **Transport:** rmcp v1.7.0, stdio, no HTTP feature (P-0012, R-0010-a/-e).
- **Startup ordering:** the **file-mode invariant check fires first**, before any listener binds (R-0005-f instantiation / R-0022-a 5-pre); `/health` before config load and before MCP accept (R-0004-g); seven builtins before any plugin load (R-0002-c); verified pool before MCP accept (R-0016-a); storage + schema init before the server is constructed (R-0022-d). Ordering is asserted by deterministic readiness signals, not wall-clock races (R-0022-a).
- **Storage source:** the production pool is sourced from the embedded-Postgres engine startup path `start_embedded()` (P-0010) — not the R-0014 LLM-key/config pathway. A storage-init failure fails the host closed before MCP accept; `/health` (bound first) reports not-ready (R-0022-a 5c).
- **Sequencing:** the content-hash binding (R-0021) lands **with or before** real signing (R-0005-d instantiation) within M1; M1 (the security foundation) merges before M2 (intake Hard constraint; `P-SecurityLayered` — do not ship a partial control). **M2 SHALL NOT be considered shipped while `verify-smoke` is the `if true` stub** — the real end-to-end smoke (R-0022-c smoke gate) is a merge-gate precondition for M2 (a stubbed `verify-smoke` is a false green).
- **Build/test discipline:** worktrees; `main` protected; `just ci` is the sole CI entry; no new dependencies without license-tier review (inherited parent constraints, G-0006).
- Architecture MUST NOT be schedule-pressured; marketing-tier dates are not architectural inputs.

## Decisions of note

Each locked decision and its canon anchor; tied trade-offs would surface to the Principal Architect, not resolve silently. **None surfaced** — the Frame's constraint-walk cleared the `conflicts-with` candidates (Security ⇄ Simplicity does not fire: the binding is simultaneously the Security default and a minimal mechanism).

- **Supersession-by-reference, not in-place edit of the parent spec.** *Anchors: `P-LockContract`, `P-MinBlastRadius`.* The parent spec is BOM-locked at blob `3118fbb`; editing it would drift its audit chain. The delta lives here, citing parent R-IDs. (See [§ Supersession mechanism](#supersession-mechanism).)
- **Two new R-IDs (R-0021, R-0022), matching the Frame's two new slots exactly.** The `verify-build` placeholder gate is homed as an **R-0018-f supersession entry** (R-0018-f already owns the verify-build contract; the gate is a new AC on it), not a third new R-ID — preserving Frame↔Spec slot traceability and avoiding double-encoding. *Anchor: Frame §5 slot allocation; the Frame puts R-0018-f in its amended-in-place table, not the new-slot table.*
- **BLAKE3 as the V0 content-hash algorithm; field constrained to a strong set.** *Anchors: Simplicity, `P-StackDiscipline`, R-0008-b.* Reuses the in-tree BLAKE3 primitive (already mandated for admin-token hashing); the `hash_alg` field rejects MD5/SHA-1 so a weak algorithm cannot be declared.
- **Integrity-gated load path loads the signed artifact; a reproducible `wasm32-wasip2` build is NOT required (round-1 decide-and-lock).** *Anchors: `P-SecurityLayered` (the gate verifies the artifact actually run — the security-meaningful unit), Simplicity / `P-StackDiscipline`, `P-MinBlastRadius`.* The question (Bolt HIGH): `just ci` runs `verify-test: plugin` (rebuilds the `.wasm`) before `verify-smoke`, so a real `verify-smoke` loading the rebuild would false-reject via R-0021-e unless the rebuild is byte-identical to the signed artifact. Two options were live: **(a)** require a deterministic/reproducible release build so a rebuild reproduces the signed bytes; **(b)** the integrity-gated path loads the *signed component artifact* and the smoke gate does not rebuild. **Locked: (b).** Canon pre-decides it — the content-hash binding's security property (R-0021 / QA-1) is *"the bytes the runtime loads equal the bytes that were signed,"* which (b) realizes directly by loading the signed artifact (production `run()` R-0022-e and the R-0022-c smoke gate load the signed `.wasm`, distinct from the rebuilt `target/` output; the smoke gate carries no `plugin` prerequisite and runs no `cargo build`). **(a) is rejected** as unsubstantiated and brittle against the *real* build setup: the repo has no toolchain pin (`rust-toolchain.toml`), no `--remap-path-prefix`, and nothing establishing `wasm32-wasip2` build reproducibility, so asserting determinism would assert a constraint that does not hold; worse, (a) realizes a *different, weaker* property — build bit-reproducibility — coupled to the exact toolchain, so a benign `rustc`/`wasm-tools`/`wit-bindgen` version bump would flip the hash and false-trip the gate, training operators to ignore the very tamper signal R-0021-f exists to surface. No escalation needed — canon resolves the tie; this is an application, not a novel call. *(Implementation consequence: the production load resolution at `echo_component_path()` (`libs/mnemra-host/startup/pool_population.rs:165-172`) moves from the `target/` rebuild to the signed committed artifact; the exact committed path is implementation-tier.)*
- **`verify-build` gate is a positive (allowlist) real-key pin; mechanism is a class, not a const layout.** *Anchors: `P-SecurityLayered`, QA-2 measure 2.* The R-0018-f gate passes **iff** the embedded root equals the pinned real root (placeholder or any non-pinned key fails) — closing the implicit-allow gap a blocklist ("fail iff `== placeholder`") would leave for a third/dev key. The spec locks the property + the mechanism *class* (compile-time `const` assertion preferred; build-script / `verify-build`-invoked check as the fallback when the pin is a fingerprint that requires non-`const` hashing) — not the concrete Rust const layout, which is implementation-tier. The recipe's emission convention is `GATE build`, not `GATE verify-build`.
- **`[component]` section in the signed body as the field home.** *Anchors: R-0005-h (core-by-provenance forces the signed-body location), `P-LockContract` (the manifest schema is the locked contract — a schema field locks into the schema's governing record, P-0003).*
- **`schema_version` stays `1`, paired with the mandatory-field rule (one decision).** Recorded in the [P-0003 amendment 2026-06-30](../src/adrs/P-0003-plugin-manifest.md) with the rebuttal of P-0003's "increment on structure change" text — so the ADR is not silently self-contradicted (required-lock #6). R-0021-c enforces the pairing: at version 1 the version number cannot signal presence, so the parser requires the field.
- **Distinct hash-mismatch error variant.** *Anchor: `P-SecurityLayered` runtime sub-layer — a tamper signal must be observable apart from a benign load failure.* R-0021-f.
- **Verification-surface split: some properties are code-review-enforced, not black-box.** The single-read-then-`from_binary` TOCTOU property (R-0021-d) and the storage-before-server-construct call-ordering (R-0022-d) are **implementation properties a black-box test cannot assert non-flakily**; they are enforced at **code review** (the security/correctness reviewer's domain), with the externally-observable consequences (a swapped `.wasm` is rejected; the served host is storage-backed) covered black-box. Recording the split keeps each acceptance criterion binary-observable from a single reader (the validatability lens) rather than smuggling a code-inspection criterion into a black-box clause. *(Anchor: the binary-observable-or-it-isn't-a-requirement lens; surfaced by the Stage-3 testability review.)*
