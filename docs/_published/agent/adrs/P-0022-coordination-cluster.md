# P-0022: Coordination Cluster

**Project:** mnemra-core

## Status

`accepted` (spec-exit gate, 2026-07-07)

Authored at the coordination-wedge Stage-3 spec; flipped to `accepted` at the spec-exit gate (the sibling-cluster precedent — the three retrieval ADRs P-0014/P-0015/P-0016 and the two reporting ADRs P-0020/P-0021 were each authored `proposed` at their cluster's Stage 3 and moved to `accepted` at the cluster's spec-exit gate as part of the reviewed package). Resolves the Frame's `{{P-CoordinationCluster}}` open ADR slot. Binding requirement text: [coordination-wedge spec](../../specs/2026-07-06-coordination-wedge.md) R-0063–R-0077.

## Context and Problem Statement

Multiple concurrent orchestrator instances operate in one workspace and cannot coordinate safely. Today's substrate is file conventions: resource claims are advisory files with no atomic acquisition and no liveness binding; handoffs carry no delivery or consumption state; cross-session continuity rides hand-rolled carrier files. The intake records four failure instances live — a handoff consumed with no trace, an addressed item no instance could safely archive, a live instance's claim file adopted by another session after a context reset, and a documented archive convention that decayed silently despite exact commands. These are one failure class: a guarantee carried by convention breaks silently under agent authorship (**P-GuaranteeByMechanism**, the canon principle the wedge exists to apply to the workspace's own coordination substrate).

The coordination cluster replaces that file substrate with substrate-enforced coordination inside the mnemra-core host: durable actor identity that survives session death, leases with exactly-one-holder atomicity and a mechanical recovery rule, and addressed messages whose delivery and consumption are recorded facts. It is the system's **first live workload by design** — a coordination plane over **zero migrated content** — so "the workspace starts using mnemra" acquires a concrete, observable meaning: the file conventions stop being the medium (migration step 1 of the strangler program).

This ADR is the durable, adopter-facing record of the cluster's load-bearing contract decisions. It records at ADR precision what the [locked Frame](../../intent/coordination-wedge-frame.md) (blob `4d1a8c2`, directions C1–C13, trust-boundary resolutions TB-1..TB-5) and the [spec](../../specs/2026-07-06-coordination-wedge.md) (R-0063–R-0077) already lock; **it decides nothing new** — where the spec text is the authority on an acceptance criterion, this ADR cites the R-ID as binding text rather than re-deriving it.

**One slot, not a split.** The Frame (§9) considered splitting lease semantics into a separate ADR and rejected it: leases and messages share the actor spine, the write-authority model, the isolation default, and the availability contract, so a split would state the shared authority rules in two places (a PD1 duplication liability), and lease semantics have no independent reversal story — reversing them reshapes the same spine the messages ride. This contrasts the retrieval cluster's three slots, where each contract (retrieval topology, provenance envelope, edge schema) had separable consumers; nothing consumes lease semantics separately from this cluster. **Explicitly outside this slot** and owned elsewhere: the operational-log topology (decided in Frame C10 — it is subscriber composition behind the [P-0011](P-0011-logging-facade.md) facade, not a new contract, so it needs no slot); the audit-store detail (owned by the forthcoming event-bus ADR that [P-0018](P-0018-core-entity-manifest.md) D-SURFACE names); and the auth-bundle policy composition (Frame TB-5 name-and-defer).

## Decision Drivers

- **Guarantee by mechanism, not by convention (P-GuaranteeByMechanism → Quality, Observability).** Every coordination invariant the cluster locks must be substrate-enforced — exactly-one-holder by a schema constraint, disposition-trace by schema, liveness by TTL — never by an instruction a future session must recall. This is the cluster's reason to exist, stated as canon.
- **The core/plugin line is drawn at foundational-substrate dependency (P-0002).** Actors ARE the [P-0018](P-0018-core-entity-manifest.md) core entity; safe multi-instance operation is a precondition for plugin-era work; coordination state must be usable the moment a session binds, before any plugin loads. The verb-on-content criterion places this host-side.
- **Least authority on every write (P-LeastAuthority → Security).** Takeover is a distinct privileged path, never a side effect of acquire; release/renew are holder-only; ack/disposition are addressee-only; every coordination write runs under a host-resolved actor context.
- **Lock the intrinsic, preserve the separable (P-LockContract ⇄ P-PreserveDecisionSpace).** Intrinsic to the cluster's identity and locked now: exactly-one-holder, the duration line, disposition-trace with post-send immutability, role-instance actor grain, the two-tool surface, fail-closed writes. Separable options stay behind named tripwires: push transport, the one-tool fallback, force-break, the runtime type registry, default-deny reads.
- **A losing contender receives structured evidence, never silence (P-TrustworthySignal → Observability).** The exactly-one-holder observable needs the loser's mechanical refusal visible and logged; unavailability renders as a structured state, not an empty result.
- **The blast-radius unit is the flip unit (P-MinBlastRadius → Maintainability).** Leases and messages are separate tables and separate tools with no cross-dependency, so the two cutover flip passes are independently schedulable.
- **Validator before field (P-ShiftLeft D2).** A resource-identifier family, a message type, or a disposition member does not register without a mechanical validator — closed enums and versioned schemas get validators; freeform prose does not.
- **Confidentiality is secondary at single-team scope, with one edge (attribute priority; Security | Simplicity edge; usable-security clause).** Integrity and availability are load-bearing; the visible-reads default is a deliberate, ratified, tripwired concession on the Security | Simplicity edge, not a silent one — a control whose friction drives bypass has failed as security.

## Considered Options

The Frame ratified the cluster's shape at elicitation (maintainer, 2026-07-06); this ADR records the ratified direction against its genuine alternatives.

1. **CS1 — A core host subsystem: actors/leases/messages as first-class coordination-owned rows, two agent-visible MCP tools (chosen).** Host code above the plugin boundary, beside the retrieval subsystem; substrate-enforced coordination invariants; `claim` + `message` as host-served verbs.
2. **CS2 — A coordination plugin.** The verbs export through the plugin ABI as domain verbs.
3. **CS3 — Retain the file-convention substrate.** Advisory claim files + a file inbox — the status quo, hardened by better conventions.

## Decision Outcome

**CS1 — a core host subsystem.** Seven decisions lock; each records a Frame direction and the spec R-slot that carries its binding acceptance criteria.

### D-RESIDENCE — Coordination is core host code, not a plugin

The coordination cluster — session plane (actor resolution + attachment), lease subsystem, message subsystem, message-type registry, observability emission — is **host code inside the mnemra-core host process**, beside (not inside) the retrieval subsystem. It exports nothing through the plugin ABI: no WIT-surface change, no plugin-exported coordination verb, no [P-0013](P-0013-plugin-invocation-model.md) domain-verb dispatch machinery, and no new signed plugin artifact.

The [P-0002](P-0002-core-plugin-partition.md) walk, recorded: **verb-on-content?** No — acquire/release/takeover/send/ack/disposition are state transitions on operational substrate, not CRUD on content-artifact rows (coordination rows carry no document identity — no frontmatter/body, no provenance envelope). **Foundational-substrate dependency?** Yes — actors ARE the [P-0018](P-0018-core-entity-manifest.md) core entity, and safe multi-instance operation must exist before and independent of plugin work, the same property that put workspaces/users/agents/sessions in the builtin tier. **Bootstrap ordering:** coordination state must be usable the moment a session binds, before any plugin loads; plugin residence would invert the ordering. Two consequences fall out and are stated rather than left implicit: the [P-0013](P-0013-plugin-invocation-model.md) deferred-domain-verb tripwire **does not fire** (its firing condition is the first domain verb that must dispatch at the plugin *export* surface; host-served MCP verbs are not plugin exports — the retrieval cluster's host-placement precedent), and [P-0005](P-0005-v0-signing-chain.md)'s plugin-load verification surface is **unchanged** (no new signed artifact is minted; intake Hard constraint 6's signing clause is satisfied vacuously-by-residence). *(Anchors: [P-0002](P-0002-core-plugin-partition.md) discriminator; [P-0018](P-0018-core-entity-manifest.md) — actors are the core entity; [P-0013](P-0013-plugin-invocation-model.md) tripwire scoping; [P-0005](P-0005-v0-signing-chain.md); precedent — retrieval as a host subsystem, report definitions as host-core-owned (P-0021). Binding text: spec R-0063-a.)*

### D-VERBS — Exactly two agent-visible tools, actions as closed enums; the verb budget quantified

Exactly two agent-visible MCP tools ship: **`claim`** (actions `acquire`, `renew`, `release`, `takeover`, `list`) and **`message`** (actions `send`, `poll`, `list`, `ack`, `disposition`), each action set a closed enum in the tool schema (an out-of-enum action is refused with a structured error), registered on the existing single stdio MCP server as host-served verbs following the [P-0012](P-0012-plugin-runtime-and-mcp-sdk.md) `rmcp` handler contract. Two nouns with small action sets were chosen over one `coordinate` mega-tool (kept as the named budget fallback) and over one-tool-per-verb (rejected under the verb budget); the claim/message split also matches the two cutover flip passes. **Registration is deliberately not a third tool** — resolve-attach is a session-plane primitive below the tool surface (D-ACTORS), realized at the `message` tool's `poll` action.

The verb-budget accounting the Frame quantified (residual 3): the designed agent-visible MCP tool surface across all locked clusters stands at **8** against the single-digit ceiling read at its tightest (**≤ 9**) — retrieval 3 (`get_context_for`, `search`, `get_artifact_by_citation`), reporting 3 (`report.list/get/invoke`), coordination 2 (`claim`, `message`). **One slot of headroom remains, and it is contended, not free:** it must absorb every cluster's authoring path (the forthcoming MCP-verb→CRUD content-write amendment and the reporting cluster's definition-authoring write surface), which has the recorded consequence that the ≤9 ceiling forecloses a dedicated report-authoring verb (R-0055-b's dedicated-verb branch), and this cluster SHALL NOT consume the reserved slot. The pressure valve, in order: collapse `claim` + `message` into one `coordinate` tool (−1, the ratified fallback, preserved not built); only past that, a budget expansion justified against the recorded accuracy finding per the constraint's own rule. *(Anchors: the verb-budget constraint (maintainer-ratified 2026-07-06, falsifiable form — content-type/action as parameter, never a tool name); [P-0012](P-0012-plugin-runtime-and-mcp-sdk.md) handler contract; [P-0013](P-0013-plugin-invocation-model.md) front-door discipline; P-ShiftLeft D2 — closed enums get validators; [P-0009](P-0009-rls-admin-token.md) verb-category gating per action. Binding text: spec R-0063-b/-c/-d, §API Contract.)*

### D-ACTORS — Role-instance is one durable actor row; sessions are ephemeral attachments; the principal is host-derived

Each role-instance (operator, design lane, merger, research lane, …) is exactly **one durable `actors` row** with `actor_type = agent`, at the [P-0018](P-0018-core-entity-manifest.md) D-ACTOR grain — **no actor↔session intermediate entity**. A session presents its role-instance identifier and **resolves-or-creates** that row through the identity-builtin path (`P-builtin-agents`, [P-0002](P-0002-core-plugin-partition.md)), never minting a duplicate for an existing identifier; the resolution key is the actor's per-workspace unique name. The acting principal on every coordination write is derived from **host-held attachment state** — never a caller-supplied per-write parameter (the [P-0019](P-0019-plugin-contract.md) host-derived-context rule, generalized from `workspace-ctx` to the acting principal), so a write cannot launder itself as a different actor than its session attached as.

Four attachment invariants are the locked contract (the realization is not — see below): **(1) at most one live attachment per actor row** (a second concurrent session declaring the same role-instance is refused while the first is live); **(2) agent-type-only** — attach validates the resolved row's `actor_type` and refuses `human`/`system` rows, so a session cannot launder its writes as host-initiated or human action; **(3) stale-recoverable under a TTL** — attachment liveness rides the same stale/TTL discipline as leases, renewed implicitly by every successful coordination operation and by `poll`; **(4) succession over a stale attachment is an audited takeover at TB-2 parity** — automatic at the successor's bind, recorded with the prior attachment holder (session), the successor session's identifier (equal identifiers being the mechanical discriminator between an idle-gap resume and a genuine succession), and the staleness/expiry evidence, because an actor-identity succession has *higher* integrity stakes than a resource-lease takeover (it governs the attribution of every subsequent write). A live attachment is not succeedable. The **attachment TTL is security-load-bearing**, not merely an availability knob: it sets the TB-3 impersonation-after-death window on one side and successor-lockout on the other, and its spec-tier default is calibrated under exactly that security tension. The canonical (and only) bind call is the `message` tool's `poll` action carrying the role-instance declaration — one call that resolves, attaches, audits, and delivers, at parity with today's session-start injection; every other action is refused `not_attached` when the session holds no live attachment. The **preferred realization** holds the attachment under the lease mechanism itself (an attachment is held intent on the role-instance — one liveness mechanism, two uses; the reserved `actor` resource family); a dedicated linkage is admissible provided the four invariants hold identically — the invariants are locked, the realization is not (P-LockContract applied by the Frame itself). *(Anchors: [P-0018](P-0018-core-entity-manifest.md) D-ACTOR — one `actors` table, closed `actor_type`; [P-0019](P-0019-plugin-contract.md) context-derivation precedent; P-GuaranteeByMechanism — the type restriction is enforced, not implied; P-LockContract — invariants lock, realization varies. Binding text: spec R-0064-a..-f.)*

### D-LEASES — Leases are the mutual-exclusion primitive on the duration line; acquisition is substrate-atomic; takeover is expired-only and audited

The claims split follows **duration, not residence**: an **atomic write** to in-system state is a storage transaction — the engine's semantics absorb it, no claim entity involved ([P-0010](P-0010-storage-substrate-engine.md) `Storage` trait); **held intent** — a resource held across time, inside or outside the system (an hour-long design-doc edit, a repo's main-merge lane, a config surface) — is a **lease**: a first-class entity with holder (`actor_id` FK), a structured resource identifier, acquisition time, TTL/expiry, and explicit state. Acquisition is **substrate-enforced atomic** — a uniqueness-constrained insert or compare-and-set inside a single transaction, never application-level check-then-act, never advisory-file semantics — and a schema-level constraint (a partial unique index or equivalent) makes at most one live lease per `(workspace_id, resource)` able to exist, enforced in the acquisition transaction rather than by serving-path filtering. The losing contender receives a structured, logged refusal (`resource_held`, carrying the holder and expiry), never an error string, never empty-success.

`renew`/`release` are holder-only; `acquire` declares a requested duration bounded by a policy maximum. Stale semantics have **no manual-wipe limbo**: an expired lease is takeover-eligible via the explicit, privileged, **always-audited `takeover` action** — expired leases only (a live lease is not takeover-able through the tool surface; there is no force-break verb at V0 — a wedged live lease waits out its TTL, and an emergency break is a maintainer store operation, audited), any workspace actor may take over an expired lease, and the audit record carries prior holder, new holder, expiry timestamp, and takeover timestamp. Takeover is never a flag or side effect on `acquire`. A deposed holder discovers the deposition at its next coordination write, which fails closed — two actors cannot both proceed as holder under any interleaving. The **resource identifier** is a registered structured string `<family>:<qualifier>` over a closed V0 family set (`repo-lane`, `file`, `surface`, and the reserved `actor` for the D-ACTORS attachment realization — reserved **and barred from the entire `claim` surface**: every `claim` action refuses it with the dedicated `reserved_family` code and no listing surfaces `actor:` rows, so attachment lifecycle flows only through the session plane's audited bind/succession path, spec R-0067-c); each family ships its own mechanical validator, and adding a family is a spec amendment, never a config knob. The merger lane's `repo-lane:<repo>/main-merge` is the first designed consumer. Renew is **uncapped at V0** — a live-and-renewing holder can hold indefinitely; that absence of a total-hold bound is a deliberate, named V0 non-bound (nuisance-class at trusted single-team scope, visible in the op log), re-examining with the force-break deferral. *(Anchors: P-GuaranteeByMechanism — the invariant lives in the substrate, not a convention; [P-0010](P-0010-storage-substrate-engine.md) — transactions absorb atomic writes; P-LeastAuthority — takeover authority scoped to the recovery job, never standing break authority; P-ShiftLeft D2 — a resource family without a validator does not register. Binding text: spec R-0065-a..-e, R-0066-a..-c, R-0067-a..-c.)*

### D-MESSAGES — Messages are rows: post-send immutability, a monotonic closed lifecycle, and a closed disposition vocabulary; types are code-registered

A message is a **row in a coordination-owned table**: sender `actor_id` FK, addressee `actor_id` FK, message-type name + schema-version, schema-validated JSON payload, lifecycle state, and timestamps. Addressing is role-instance (the actor row), so a queue survives session death exactly as leases do. Two invariants make the intake's traceless-consumption and unarchivable-item failure modes **structurally impossible** rather than procedurally discouraged: **post-send immutability** — payload, sender, addressee, type, and schema-version are immutable after send, with no edit/recall/retraction path at any layer (not merely no verb — no host function or substrate helper mutates these fields post-insert); and a **monotonic lifecycle with append-only consumption records** — the state enum `sent → delivered → acknowledged → dispositioned` admits no reverse or repeated transition, and the consumption timestamps (delivered/acknowledged/dispositioned) are written exactly once at their transition and never cleared. Deletion is **not a lifecycle operation** at V0 (deleting a pending message is retraction by another name; deleting a dispositioned one erases the trail); any future retention policy is a spec amendment preserving consumption records under the same append-only shape.

`ack` and `disposition` are **distinct addressee-only steps** (receipt vs terminal outcome) — collapsing them would erase the delivered-but-unresolved state the intake's stale-addressed-item evidence needs; a non-addressee attempt is refused `not_addressee` and logged. The terminal **disposition vocabulary is closed at `{completed, declined, obsolete}`** — `obsolete` is the archive path the intake's unarchivable-stale-item evidence demands — and extension is a spec amendment. Message **types are host-owned, code-registered, and closed at V0**: a type is a named, versioned schema shipped with the host (no message-type registry table exists — that would be a second source of truth for a code-owned set), and `send` validates the payload against the named type + schema-version and **fails closed** on violation (`schema_violation`, in the same refusal grammar as lease contention). Schema evolution is additive-only within a version; a breaking change mints a new version on the same type name. Two founding types ship: **`merge-request` v1**, carrying the merger-role input contract's locked particulars each as a named typed field (none freeform prose), and a deliberately minimal **`handoff` v1** (`subject` + `body` + `artifact_refs`) so unstructured instance-to-instance handoffs have a schema-valid home without this cluster designing the stash-successor surface. Messages are **not content**: no provenance envelope, not served by `get_context_for` at V0 (coordination is not context — a named tripwire), and no reference into content tables. *(Anchors: [P-0017](P-0017-storage-cluster-model.md) D-SoT — coordination rows are their own source of truth; P-GuaranteeByMechanism — "structurally impossible" is structural, not procedural; the append-only integrity shape [P-0018](P-0018-core-entity-manifest.md) D-SURFACE applies to audit, applied to the message store; P-TrustworthySignal — delivery is a recorded fact, not an implication of send; P-ShiftLeft D2 — a type without a schema validator does not register; Simplicity — governance rides the existing designed-tier pipeline. Binding text: spec R-0068-a..-e, R-0069-a..-c, R-0070-a..-c, R-0071-a/-b.)*

### D-ISOLATION — Reads default-visible, writes strictly authority-scoped; the write-authority matrix; the multi-team tripwire

Within a workspace, **reads are default-visible** — any resolved (attached) actor may `claim list` live leases and `message list` any actor's messages, payloads included — and **writes are strictly authority-scoped** per the write-authority matrix, enforced as host-side data-level checks under `WorkspaceCtx` ([P-0006](P-0006-v0-tenant-enforcement.md)/[P-0009](P-0009-rls-admin-token.md) era). Visibility grants no write authority anywhere; the matrix is unconditional on visibility. The visible-reads default is a **deliberate, maintainer-ratified, tripwired concession** on the Security | Simplicity edge: the confidentiality cost is coordination metadata and message-payload content readable by any resolved actor (recorded at its accurate extent, not understated as metadata-only), which is still strictly less exposure than the current file convention where every instance reads every claim file and the whole inbox; the counterfactual (default-deny with per-actor grants) would add a grant model the auth bundle owns later, break the live "do not mutate a claimed resource" workflow that *requires* reading others' claims, and drive bypass — a control whose friction drives bypass has failed as security (the usable-security clause).

Write authority (V0): resolve/attach onto an `actor_type = agent` row with no live attachment (succession over a stale attachment is an audited takeover); lease acquire by any resolved actor, renew/release holder-only, takeover any resolved actor on expired leases only; message send by any resolved actor (open in-workspace addressing — every message is attributed to its host-resolved sender, so anonymous or spoofed-sender messages are structurally impossible), ack/disposition addressee-only, poll self-scoped. [P-0009](P-0009-rls-admin-token.md) verb-category gating applies *before* actor-level authority, with a stated consequence: because every coordination action executes under an attachment and attachment is itself a write, **a `read_observer` token cannot participate in the coordination surface at V0** — monitoring reads for read-only consumers are a future need that arrives as its own intake. No rate/volume/queue-depth limiting exists on `send`/`acquire` at V0 — a stated non-control, not an omission (nuisance-class at workspace scale, visible in the op log). **Default-deny reads re-open on a named tripwire:** a deployment tier crossing single-team trust (hosted multi-team), **and** — the gate-confirmed sharpening — the first *designed* introduction of a trust-differentiated actor within the fleet (an untrusted plugin-supplied actor, a deliberately lower-trust lane), since the default rests on a mutually-trusted-fleet assumption of which "single team" is only the current instantiation. Fail-closed is the **integrity backstop** on this surface: a deposed lease holder or a session whose attachment lapsed discovers it at its next coordination write, which fails closed, so takeover races cannot yield two proceeding holders even across an outage. *(Anchors: the Security | Simplicity conflict edge, walked with the concession ratified and tripwired; intake attribute priority — confidentiality secondary; P-LeastAuthority on every write row; [P-0009](P-0009-rls-admin-token.md) verb-category gating; P-TrustworthySignal / P-SecurityLayered fail-closed. Binding text: spec R-0073-a..-d, R-0074-d.)*

### D-STORAGE — Own storage placement outside the content taxonomy; hard-FK to core entities only; standalone over zero content

Coordination tables (`leases`, `messages`, and attachment state if realized as its own linkage) are **host-owned operational tables** under the engine-agnostic [P-0010](P-0010-storage-substrate-engine.md) `Storage` trait, workspace-scoped `NOT NULL` under `WorkspaceCtx` ([P-0006](P-0006-v0-tenant-enforcement.md)), with hard FKs **only** to core entities (`actors`, and `projects` where a row is project-scoped — [P-0018](P-0018-core-entity-manifest.md) D-ENT / [P-0017](P-0017-storage-cluster-model.md) D-SoT); every other reference is a soft ref. They sit **outside the [P-0017](P-0017-storage-cluster-model.md) four-shape *content* taxonomy** — that taxonomy classifies content types, and coordination rows are not content (the retrieval cluster's operational tables set the precedent) — while adopting state-bearing-equivalent policy by construction: typed columns and closed enums (no frontmatter/body document layout), indexes on the hot predicates (live-lease-per-resource, undispositioned-per-addressee), audit emission on privileged mutation. The four-shape set stays closed; **no P-0017 amendment**. The **standalone boundary** is a property, not a wish (intake Hard constraint 1 / Success criterion 6): coordination tables carry no reference of any kind — FK, soft ref, join, or read path — into content tables; the two tools touch no retrieval, provenance-envelope, edge, or index machinery; message payloads are schema-validated JSON owned by the coordination store, not content artifacts; and a fresh deployment holding only core-entity and coordination rows supports full fleet coordination (bind → acquire → send → poll → ack → disposition → release) with zero content tables populated. Migrations are additive, as everywhere. *(Anchors: [P-0017](P-0017-storage-cluster-model.md) D-CM (closed four-shape set) + D-SoT (hard FKs reserved for core entities); [P-0010](P-0010-storage-substrate-engine.md) D5 (`Storage` trait); [P-0006](P-0006-v0-tenant-enforcement.md) (`workspace_id` under `WorkspaceCtx`); [P-0018](P-0018-core-entity-manifest.md) D-ENT; Simplicity — no new taxonomy shape for a non-content family. Binding text: spec R-0076-a..-e.)*

### Consequences

**Good:**

- Every invariant the file conventions demonstrably could not hold is now substrate-enforced: exactly-one-holder by a schema constraint, disposition-trace by an append-only monotonic lifecycle, liveness by TTL — the P-GuaranteeByMechanism failure class is closed on the workspace's own coordination substrate.
- The cluster is the system's first live workload over zero content, so day-one forward capture *is* the coordination baseline and "the workspace starts using mnemra" becomes observable — the file conventions stop being the medium.
- Leases and messages are independently flippable (separate tables, separate tools, no cross-dependency), so the two cutover passes are separately schedulable with drain-then-flip and no dual-authority window — the flip unit, the policy unit, and the blast-radius unit coincide.
- The [P-0018](P-0018-core-entity-manifest.md) core set stays closed at four: coordination introduces no new hard-FK-target core entity — actors is consumed, not forked; leases/messages hard-FK *to* core entities and are FK'd to by nothing.
- The write-authority rules are shaped as host-enforced data-level checks expressible as policy rows later, so the auth bundle adopts the matrix as policy content without a coordination redesign (actors are already the shared principal vocabulary).
- **The merger stays instance-shaped (Frame C9), recorded here as an in-scope consequence:** messaging changes the merger's *input medium* — `merge-request` messages replace inbox files (the founding type, D-MESSAGES) — but not its *wake mechanism*; under poll transport there is no standing process to wake, so a merger session still starts per batch, exactly as the instance shape works. The instance-vs-agent tripwire re-arms on push-transport landing (a designed event), not on a date.

**Bad / Trade-offs:**

- Actor identity is **declared, not proven** inside the workspace trust boundary until per-actor credentials land: any session authenticated at the workspace boundary may declare a role-instance and attach. Mitigated structurally by the host-derived principal (D-ACTORS), the one-live-attachment invariant, and audited attach/succession — the residual (a co-tenant attaching to an idle role-instance) is accepted at single-team scope and closes when the auth bundle delivers per-actor credentials.
- Coordination metadata **and message payloads** are visible to all workspace actors by design (D-ISOLATION) — a deliberate confidentiality concession, tripwired for the multi-team / trust-differentiated-actor re-open.
- TTL-expiry takeover can depose a slow-but-alive holder — bounded by renew discipline plus the fail-closed backstop (a deposed holder's next write fails closed, so two holders never both proceed), and every deposition is audited with the expiry evidence (TB-2).
- Open in-workspace addressing admits queue noise, unbounded by rate limits at V0 (TB-4) — the stated non-control (spec R-0073-c), nuisance-class at workspace scale and visible in the op log, re-examining at the isolation tripwire.
- The exactly-one-holder guarantee is on the lease *row*; for resources outside the system (git repos, config, filesystem) the lease is advisory, and a mid-action TTL expiry during a service outage does not gate an in-flight external side effect already underway — the narrow expiry-during-outage-during-external-action window, bounded by lease-duration policy sized to the largest external unit of work, is partly inherent to advisory leases over external resources.

These **five** residuals are the Frame §5 set, verbatim in its order, each accepted at single-team dogfood scope with a named re-open condition; none is silent. Distinct from the §5 residual set and recorded in its own register: **renew is uncapped at V0**, so a live-and-renewing holder can hold a resource indefinitely — the named V0 non-bound (TB-2; spec R-0065-d), nuisance-class at trusted single-team scope, visible in the op log, re-examining with the force-break deferral.

## Pros and Cons of the Options

### CS1 — Core host subsystem (chosen)

- Pro: satisfies the [P-0002](P-0002-core-plugin-partition.md) foundational-substrate criterion (bootstrap ordering — coordination usable before any plugin loads; actors are the core entity); the standalone constraint (Hard constraint 1) is met trivially — no plugin machinery, no content dependency.
- Pro: mints no new signed plugin artifact and does not fire the [P-0013](P-0013-plugin-invocation-model.md) domain-verb tripwire — the plugin export surface and signing-verification surface are unchanged.
- Con: coordination host code grows the host binary's own surface; the trade is accepted because the alternative inverts bootstrap ordering.

### CS2 — A coordination plugin

- Con: inverts bootstrap ordering — coordination state must be usable the moment a session binds, before any plugin loads; a plugin cannot provide it.
- Con: would fire the [P-0013](P-0013-plugin-invocation-model.md) deferred-domain-verb tripwire (coordination verbs are neither content CRUD nor host-served) and mint a new signed plugin artifact ([P-0005](P-0005-v0-signing-chain.md)) for a subsystem that has no content dependency to justify plugin residence.
- Con: fails the verb-on-content criterion — coordination verbs are state transitions on operational substrate, not CRUD on content.

### CS3 — Retain the file-convention substrate

- Con: the status quo the wedge exists to replace — advisory files have no atomic acquisition, no liveness binding, and no consumption state; the intake's four evidence items are exactly the convention-decay P-GuaranteeByMechanism predicts.
- Con: no mechanism makes exactly-one-holder, disposition-trace, or liveness hold — every guarantee rides an instruction a future session must recall.
